## 8. 我對「首波要不要先部署併發軟體」的建議

**建議：先不要，但要把「排隊不退純語音」這一小塊單獨挑出來提前做。**

理由：
1. 首波用戶量小（個位數到幾十），單卡單人在低併發下大機率夠用，此刻硬上整套 Gateway + GLOWS 自動擴縮，等於在還沒真人流量驗證前引入新的複雜度與新的故障點（例如 §1-9 提到的「軟體全綠但零真卡驗證」）。
2. 但「有臉」是產品的核心賣點（「聊聊＝Voice+Avatar，無靜態圖聊天」），現在的「第二人直接退純語音（沒臉）」在首波萬一同時有 2-3 人聊天，會直接讓部分用戶看到打折的產品——這個體驗風險比「多花 1-2 天工把排隊接上」更值得花時間補。
3. 折衷做法：**只接「一卡 N 槽 + 排隊等待畫面」這一小塊**（不需要 GLOWS 自動開卡、不需要 token），讓第二人不會直接掉臉，而是看到「有人在排隊，稍等」——這部分程式（`deploy/gateway/`）已經寫好且單元測試綠燈，只差「接上現役那張 tw-06 卡＋App 端排隊 UI」，卡西法／Codex 那份「多人併發」工作紀錄裡明確列著這是下一步。監控告警（§2-3）也建議搭配著一起做，因為排隊本身就需要「多久算合理」的可見度。
4. 真正的多卡自動擴縮（GLOWS token、自動開新卡）可以放心留到「真的看到用量逼近上限」再做——這部分本來就設計成「等 SDK token 修好」的階段 2，不擋首波。
