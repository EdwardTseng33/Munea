# 沐寧通知系統 P0 開發計畫

> 範圍：正式上線前統一完成本機通知、APNs 遠端推播、App 內通知收件匣、權限狀態、精準導頁、隱私與事件追蹤。家庭傳話只是第一個事件來源；邀請、用藥、看診與健康異常都必須走同一條通知管線。

## 0. 環境與前置

- 正式資料庫只使用東京 Supabase `fespbkdwafueyonppzwq`；雪梨只保留回復，不寫入 notification／device／outbox 資料。
- PR #41 提供雲端用藥提醒與 dose events；PR #42 提供指定收件人的家庭傳話。兩者先整合，通知平台才接事件。
- Backend secret、APNs `.p8`、Key ID、Team ID 只存在 Secret Manager／Backend，不得進 Web 或 App 包。
- RunPod／GLOWS 不在通知工程範圍。

## 1. 現況更正與必修缺口

### 1.1 用藥排程不是「每週星期」

目前 UI 的 `7 天／14 天／30 天／90 天／長期` 是**療程長度**，並沒有每週星期選擇。現有 `notify.js` 卻把所有用藥排成無限每日重複，會在有限療程結束後繼續提醒。

P0 必須把用藥 schedule 正規化為：

- `startDate`
- `endDate`（長期為 null）
- `timeSlots`
- `times`
- `timezone`（首發預設 `Asia/Taipei`，仍要明確存放）
- `weekdays`（目前預設每天；未來有 UI 再開放指定星期）

### 1.2 健康異常目前不是背景監測

目前 `HealthPlugin` 只有使用者開 App 後呼叫 `getSummary／getHistory`，沒有 `HKObserverQuery` 或 background delivery。現行危險值判斷也在 Web App 取得數值後才執行。

因此上線前只能二選一：

1. 完成 HealthKit 背景更新 PoC、真機驗證及後端事件上傳，再宣稱「App 關閉時仍可在同步到新數據後通知家人」；或
2. 收斂為「當你打開沐寧並同步到異常數據時，通知選定家人」，不得寫「第一時間監測」或「即時通知」。

即使完成背景 delivery，也不得把它描述成醫療級即時監測或保證送達。

## 2. 一套通知平台、六類事件

| 事件 | 本人裝置 | 家人裝置 | P0 行為 |
|---|---|---|---|
| 家庭傳話 | 遠端推播＋App 待聽卡 | 僅指定收件人 | 泛化通知；進 App 聽 AI 說或看原文 |
| 家庭邀請／申請 | App 內狀態 | owner／申請人遠端推播 | 申請送出通知 owner；通過／拒絕通知申請人 |
| 用藥到點 | 本機通知為主 | 不通知 | 點擊進用藥確認；依療程開始／結束 |
| 用藥漏服 | App 內＋本人可再提醒 | 選定照護者遠端推播 | 超過 grace period 且仍未確認才通知，不把沒按等同真的沒吃 |
| 看診／家庭活動 | 本機通知為主 | 有明確分享／受邀才通知 | 點擊進對應看診或活動 |
| 健康異常 | 本人 App 內提醒 | 選定安全聯絡人遠端推播 | 僅對已同步、通過去重與安全規則的事件發送 |

### 邀請的限制

尚未註冊、沒有裝置 token 的外部對象不能靠 APNs 收到第一封邀請。第一步仍由系統分享面板透過 LINE／訊息／Email 傳邀請連結或代碼；對方登入並送出申請後，owner 才能收到 APNs。通過或拒絕後，再推播給已登入的申請人。

## 3. 共用資料契約

### 3.1 `push_devices`

每次登入、token 變更、App 升版及登出同步：

- `person_id`
- token（僅 service role 可讀；log 只留 hash／末四碼）
- `environment`：sandbox／production
- `platform`、`app_version`、`locale`、`timezone`
- `notifications_enabled`
- `last_seen_at`、`invalidated_at`

換帳號、退出家庭圈及刪除帳號時，舊 token 關聯必須立即失效。

### 3.2 `notification_events`

事件先永久寫入，再排推播：

- `event_type`
- `recipient_person_id`
- `actor_person_id`
- `resource_type／resource_id`
- App 內顯示用的安全 title／body
- `sensitivity`
- `deep_link`
- `dedupe_key`
- `created_at／expires_at／read_at／archived_at`

App 內收件匣讀這張表，不依賴 APNs payload 還原訊息。

### 3.3 `notification_deliveries`／outbox

- event 與 outbox 在同一交易建立。
- 記錄 queued／APNs accepted／failed／invalid token／opened／actioned。
- 指數退避重試；永久無效 token 自動停用。
- APNs accepted 只能稱「已交給 Apple」，不可顯示成「使用者已收到」。
- 同一 `dedupe_key + device` 不得重送。

## 4. 本機通知修整

1. 不再 `removeAllPendingNotificationRequests()` 後才嘗試重建；先驗證完整排程，再只移除／更新沐寧自己管理且已過期的 identifier。
2. 長期用藥可按時段使用 repeating trigger；有限療程使用 one-time triggers，明確停在 `endDate`。
3. iOS pending notification 有容量限制，有限療程採 rolling window 並預留看診／活動額度；App 啟動與允許的背景時機補排。
4. 裝置回報 `localScheduledThrough`；覆蓋期結束但尚未補排時，由後端遠端提醒作網路保底，避免同時出現本機與遠端兩則。
5. 本機通知預設不顯示藥名，只顯示「你的用藥提醒到了」；解鎖後才顯示細節。
6. 點擊後必須帶 `type + resourceId`，不能只打開首頁。

## 5. APNs 與導頁契約

### 5.1 通知 payload

預設只帶：

- `eventId`
- `eventType`
- `resourceId`
- 泛化 title／body
- collapse／thread identifier

家人原文、藥名、病情、血壓與其他健康值不放在預設鎖定畫面 payload。

### 5.2 精準導頁

| 通知類型 | 點擊後 |
|---|---|
| medication_due／missed | 用藥確認底頁 |
| clinic_upcoming | 看診提醒詳情 |
| family_relay | 指定 relay 的待聽卡；可進聊聊或看文字 |
| invitation_applied | 家庭圈申請審核 |
| invitation_decided | 家庭圈結果頁 |
| health_alert | 該位家人的狀態與事件說明 |

App 冷啟動、背景及前景收到通知都要走相同 router，找不到或已過期時回通知收件匣並給明確說明。

## 6. 設定頁與 App 內通知

設定頁新增「通知與提醒」：

- 系統通知：已開啟／未開啟／被拒絕。
- 被拒絕時提供「前往 iPhone 設定」。
- 類別：家人傳話與邀請、用藥與看診、家庭活動、健康與安全。
- 鎖定畫面內容預覽：預設關閉。
- 測試通知：立即排一則安全的本機測試通知。
- 最近一次排程與同步狀態；失敗時不得安靜略過。

App 內不新增第六個底部分頁。沿用首頁「寧寧幫你留意」卡，提供「全部通知」底頁，至少包含未讀、已讀、時間、來源、精準導頁及移除／檢舉。

## 7. 各事件的安全規則

### 7.1 用藥漏服

- `沒按已吃` 只能表示「尚未確認」，不能寫成「確定沒吃」。
- 到點後先給本人一次提醒與 snooze；超過可設定 grace period 才建立 missed event。
- 僅通知使用者事先選定且仍在家庭圈內的照護者。
- 同一 dose 只通知一次；本人補登已吃後，家人端狀態同步更新但不發責備型推播。

### 7.2 健康異常

- 只使用已同步到 Backend 的真數據；附資料時間，舊數據不得當新警報。
- 同類事件設 cooldown／dedupe，不因每次開 App 重複通知。
- 文案使用「數據超出你設定的留意範圍」，不下診斷。
- 保護事件、疑似家暴／剝削不得自動推給可能是加害人的家庭圈；沿用 Guardian 的保護事件旗標。
- 緊急服務與 Critical Alerts 不在一般通知內假裝完成；未取得 Apple entitlement 前不宣稱能突破靜音。

## 8. 文案同步

上線前掃描網站、App Store、App 內與送審資料：

- App 關閉時是系統通知聲，不是 AI 自動「親口喊」。
- 可寫「點開後由沐寧說給你聽」。
- 健康異常只能依實際背景同步驗收結果描述。
- 家人通知要寫「通知你選定的家人」，不是所有家庭圈成員。
- 不使用「一定到」「即時監測」「保證送達」「破靜音」等未成立承諾。

## 9. 開發批次與分支

避免多張零碎 PR，也避免延後已完成的 PR #41／#42：

1. **Gate A**：合併 #41、更新並合併 #42，套用東京 `014／015`、部署 Brain／Voice、驗證既有功能。
2. **Batch B－平台新檔優先**：`push_devices`、notification event／outbox、APNs service、retry job、單元與 RLS 測試。
3. **Batch C－iOS**：Push capability、token lifecycle、權限狀態、local schedule v2、notification router。
4. **Batch D－事件接線**：傳話、邀請、用藥 missed、看診／活動與健康異常接同一 event service。
5. **Batch E－App UI／文案**：設定頁、全部通知底頁、精準導頁、鎖定畫面預覽與所有商店文案。
6. **Batch F－真機與部署**：東京 canary、sandbox APNs、production APNs、TestFlight、正式候選包。

Gate A 後只使用一條 `codex/notification-platform-p0` branch／PR；`web/src/app.js` 等其他 session 合併後再接，不與進行中的邀請 UI 工作競改。

## 10. 上線驗收門檻

1. 通知允許、拒絕、稍後決定、系統設定中關閉四種狀態都有正確 UI。
2. App 前景、背景、被關閉、冷啟動均能正確處理與導頁。
3. 登出、換帳號、退出家庭圈、刪除帳號後舊 token 不再收通知。
4. 有限療程結束後不再通知；長期療程維持；修改／刪除後舊排程清除。
5. 傳話只有指定收件人收到；邀請申請與結果只通知正確角色。
6. 漏服通知不把未確認說成確定未服，且同一 dose 不重複通知家人。
7. 健康事件只用新同步資料、具 cooldown，且不通知不該收到的家庭成員。
8. 鎖定畫面預設不顯示原文、藥名或健康數值。
9. APNs 失敗仍可在 App 收件匣看到事件；重試不產生重複通知。
10. 後台可區分 queued／APNs accepted／opened／actioned，不把 accepted 當 delivered。
11. sandbox 與 production token 完全分離，正式包不使用 sandbox APNs。
12. 網站、App Store、App 內文案與實際能力一致。

未全數通過前，遠端推播、漏服通知家人及健康異常通知不得標為正式上線完成。
