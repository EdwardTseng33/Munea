## 0. 結論摘要（3 秒版）

- 今晚 3 路 p95=2284ms（爆預算 138%），但 GPU 只吃 50–70%、CPU 只吃 ~5%——**不是硬體不夠，是軟體排隊排錯了**。
- 病灶不是一條全域鎖，是**兩層疊加**：① 上游模型每塊生成內建 8 處、約 14 次 `torch.cuda.synchronize()` 全裝置硬煞車（純粹拿來印 log 用，不是正確性需要）② 三個槽的 Python thread 全部擠在同一條 CUDA default stream、沒有任何一行程式碼幫它們分流。
- 三案：**A 真合批**（改上游模型支援 batch 維度，工程量最大、天花板最高）、**B CUDA streams 加拔 sync 硬煞車**（工程量最小、零 compile 風險、天花板較低但半天就能量到有沒有用）、**C 混合**（B 打底全路數都吃到、A 只做「滿 3 路」這一條固定路徑，用 padding 躲開 torch.compile 動態 batch 的地雷）——建議先做 B 當作低成本驗證彈，數字回來再決定要不要往 A/C 深挖。
- 相容性鐵律照抄 7/12 N 槽改造前例：`MUNEA_FH_SLOTS=1` 與現行 `MUNEA_FH_SLOTS=3`（不開合批）都必須一字不差；新機制一律用新環境變數預設關。
- 要不要為了榨吞吐犧牲單路最低延遲，是商業判斷（霍爾/Edward 拍板），這份文件只給技術路徑，不代答。

## 1. 病灶定位

### 1.1 先講清楚：今晚量的數字跟正式服務走的不是同一條並發路徑

今晚的 1/2/3 路數字很可能是用 `deploy/glows/run-concurrent-bench.sh` 跑出來的——這支腳本對每一路各自 fork 一個獨立 OS process（`deploy/glows/run-concurrent-bench.sh:29-32`：`"$PY" "$BENCH" ... &` 迴圈 N 次），`bench_fh.py` 每個 process 各自 import torch、各自 `get_pipeline()`、各自跑迴圈（`deploy/glows/bench_fh.py:36-38`）。

但正式服務 `flashhead_server.py` 的 N 槽是同一個 process 內的 N 條 Python thread，不是 N 個 process：`Feeder.__init__` 在 `auto_start=True` 時直接 `threading.Thread(target=self._loop, daemon=True).start()`（`deploy/runpod-avatar/flashhead_engine_core.py:284-285`），三個 Slot 的 Feeder 各自這樣起一條 thread，全部活在同一個 Python 解譯器、同一個 CUDA context 裡。

這兩種並發模型的序列化成因不完全一樣：
- **Process 級**（今晚 bench 用的）：多個獨立 CUDA context 搶一張卡，若沒開 NVIDIA MPS，GPU driver 對不同 context 的 kernel 是粗粒度 time-slice、不是真並行——這能合理解釋「GPU 只吃 50–70%」（context 切換空隙）。
- **Thread 級**（正式服務真正在跑的架構）：下一節細講，成因更明確、更好修。

兩者「GPU 沒吃滿卻變慢」的方向一致，今晚數字的量級可以參考，但要説清楚：正式服務要修的是 thread 級病灶，不是 process 級。Phase 0（第 5 節）第一件事就是把 bench 方法換成真正打 `flashhead_server.py` 的 3-slot-in-1-process 路徑重量一次，不能只信 bench_fh.py 的 process 版數字。

### 1.2 正式服務（thread 級）序列化的確切位置——已讀碼確認

**① 三個槽是三份完全獨立的模型實例，不是共用一份權重**

`flashhead_server.py:201-203`：
```python
def _load_slot(self, slot):
    t0 = time.time()
    slot.pipeline = get_pipeline(world_size=1, ckpt_dir=CKPT_DIR,
                                 wav2vec_dir=WAV2VEC_DIR, model_type="lite")
```
`_load_slot` 對每個 slot 各自呼叫一次，`FlashHead.load()`（同檔 187-189）用迴圈跑三次。也就是說現行 N 槽架構是「N 份 DiT 權重 + N 份 wav2vec 權重 + N 份 VAE 權重 + N 份各自的 torch.compile() 編譯圖」同時常駐 VRAM——這從一開始就不是合批架構，是「三台獨立引擎塞進一張卡」，這解釋了 3 路 VRAM 峰值接近理論 3×6.3GB（`docs/多人併發容量架構-2026-07-12.md` §3.2 測項 1 的預估），也是 Option A 相對現況能省的一塊（真合批只需要 1 份權重）。

**② 沒有任何一行程式碼幫三個槽分流 CUDA stream**

對 vendored 上游 repo（SoulX-FlashHead commit `9bc03de06bb0de82cd6bc477804512ae06144bf2`）全文搜尋 `cuda.Stream` / `cuda.stream(` / `set_stream`：零結果。`flashhead_engine_core.py` 全文一樣零結果。三條 Feeder thread 呼叫 `run_pipeline()` 時全部用 PyTorch 該裝置的預設 stream，沒有任何一處用 `with torch.cuda.stream(...)` 把某個槽的 kernel 導去專屬 stream。

**③ 上游 generate() 每塊都有 8 處、約 14 次無條件 torch.cuda.synchronize()——這是最主要的病灶**

`flash_head/src/pipeline/flash_head_pipeline.py`（clone 自上游、commit 同上）`generate()` 函式：

| 行號 | 用途 | 是否正確性需要 |
|---|---|---|
| 244 | denoise 迴圈每步開始前 | 否，純計時 |
| 278 | denoise 迴圈每步結束後 | 否，純計時 |
| 285 | VAE decode 前 | 否，純計時 |
| 290 | VAE decode 後 | 否，純計時 |
| 295 | color correction 前 | 否，純計時 |
| 301 | color correction 後 | 否，純計時 |
| 306 | motion frame encode 前 | 否，純計時 |
| 309 | motion frame encode 後 | 否，純計時 |

其中 244/278 是包在 `for i in range(len(self.timesteps)-1):` 迴圈裡（`flash_head_pipeline.py:243`），lite 模型 `sample_steps=4`（`flash_head/inference.py:36`），所以光這兩處迴圈內的就是 4 次 × 2 = 8 次，加上另外 6 個非迴圈的 sync call site，單一塊（960ms 預算）的一次 `generate()` 呼叫內共約 14 次 `torch.cuda.synchronize()`。

每個呼叫緊接著只有 `time.time()` 跟 `if self.rank == 0: print(...)`（例：`flash_head_pipeline.py:279-281`）——純粹是給開發者印「這一步花多久」的除錯訊息，denoise 迴圈本身的資料相依鏈完全在 GPU 端由 stream 自然排序，不需要 host 端讀值，不需要這些 sync。

`torch.cuda.synchronize()`（對應 cudaDeviceSynchronize()）語意是「等這個 CUDA context 底下、目前所有 stream 上排隊的全部工作做完」——不是等某個特定 stream。三條 Feeder thread 共用同一個 CUDA context（因為是同一個 process），所以只要有任何一個槽在跑 generate()，另外兩個槽當下已經送進 GPU 隊列的 kernel 都會被算進「要等完才准返回」的範圍。三路一起跑時，每一次 sync 都在做一次三槽混合的全裝置屏障——這比單純「三倍工作量」更慢，因為屏障之間夾雜大量 Python 端的 time.time()/print()/tensor 索引運算，這些 CPU 端小work 在三條 thread 之間還要搶 GIL，複合出來的劣化比線性 3× 更差，跟今晚「3 路不是 3× 慢、是 4.27× 慢（535→2284ms）」的形狀吻合。

**④ Python GIL 疊加**

`_gen_chunk`（`flashhead_engine_core.py:350-404`）在拿到 `self.slot.char_lock`（每槽各自一把鎖，不是全域鎖——這把鎖只防同一槽內「換角色」跟「生成」互撞，早就設計對了，`flashhead_engine_core.py:359`）之後呼叫 `self.run_pipeline(self.slot.pipeline, emb)`（360 行）。GIL 保證同一時刻只有一條 Python thread 在跑 bytecode，generate() 迴圈裡大量 host 端運算（tensor 索引、time.time()、print）都要重新搶 GIL，三條 thread 在約 14 次 sync barrier 之間反覆切換執行權，這是③之上再疊加的一層開銷。

### 1.3 上游模型目前對「真合批」的三個既有假設（會擋路，Option A 一定要改）

**① LtxVAE wrapper 硬寫死 batch=1**（`flash_head/ltx_video/ltx_vae.py:16-29`）：
```python
def encode(self, video):
    latents = self.model.encode(video, return_dict=False)[0].sample()
    out = self.normalize_latents(latents)
    return out[0]                      # 硬丟掉 batch 維度

def decode(self, zs):
    latents = zs.unsqueeze(0)          # 硬補回 batch=1
    image = self.model.decode(..., target_shape=latents.shape)[0]
    return image
```
底層 CausalVideoAutoencoder（`flash_head/ltx_video/models/autoencoders/causal_video_autoencoder.py`）本身是泛型 batch 寫法（`batch_size = sample.shape[0]` 這種通用寫法，行 739/774/904 等處），真正卡住的只有這層薄 wrapper，改動量小（拿掉 `[0]`、拿掉 `unsqueeze(0)`，target_shape 改傳批次後的 shape）。

**② rope_apply 硬寫死只算 batch index 0**（`flash_head/src/modules/flash_head_model.py:104-140`）：
```python
def rope_apply(x, freqs, grid_sizes, use_usp=False, sp_size=1, sp_rank=0):
    ...
    x_i = torch.view_as_complex(x[0, :s].to(torch.float64).reshape(s, n, -1, 2))   # 只取 batch 0
    ...
    x_i = torch.cat([x_i, x[0, seq_len:]])                                        # 只取 batch 0
    return x_i.unsqueeze(0).to(x.dtype)                                           # 永遠回傳 batch=1
```
這是真正的架構級路障：SelfAttention.forward（同檔 174-200）拿 `rope_apply(q,...)`/`rope_apply(k,...)` 餵進 attention，若上游沒改，batch>1 時 q/k 會被錯誤地砍回 batch=1、只有 v 保留真批次——輕則 shape 不合直接報錯、重則靜默算出錯誤結果（只用第 0 個樣本的內容去跟其他樣本的 v 做 attention，等於「串音」，比合批前的任何 bug 都嚴重，因為是同一次 forward 內部發生的資料污染，不是連線層級能攔的）。修法：因為本場景全部 session 共用同一組 grid_sizes（同解析度、同 frame_num），freqs_i 可以共用；把 `x[0, ...]` 改成對整個 batch 維度做（去掉硬索引），輸出维持真實 batch 數，不做 unsqueeze(0)。

**③ generate() 本身建構 noise 與呼叫 model 時到處手動 unsqueeze(0)**（`flash_head_pipeline.py:234-241` 建 noise 無 batch 維、249-254 `x=noise.unsqueeze(0)` / `y=self.ref_img_latent.unsqueeze(0)`）——這些是 Option A 需要新寫一支 `generate_batch()`（不動舊 generate()）的地方，把 N 個 session 的 noise/ref_img_latent/audio_embedding 疊成真批次餵進去。

**尚未 100% 排除、需要 Phase 1 一開始先跑 batch=2 smoke test 確認的可疑點**：`Head.forward`（`flash_head_model.py:299-321`）裡 `F = t_mod.shape[0] // B` 這行的 framewise 假設、`AudioProjModel.forward` 的 bz reshape 鏈（485-548 行），這兩處目前讀碼判斷是泛型寫法（能跟著 batch 走），但沒有實測過 batch>1，標「待實驗卡驗證」，不可跳過。

### 1.4 為什麼「GPU 只吃 50–70%」——現在能解釋清楚了

不管 process 級還是 thread 級，量到的都是「大量小屏障之間夾雜 host 端運算」的鋸齒狀 GPU 忙閒交替，不是 GPU 算力真的不夠——1 路已有 44% 餘裕（535ms / 960ms 預算）本身就是證據：如果算力吃緊，1 路不會有這麼多餘裕。這代表卡上還有沒被榨乾的並行容量，這正是 Option A/B 有機會有效的物理基礎，但「有機會」不等於「量到多少」，見第 2 節的估算與其不確定性標註。

## 2. 方案比較

### 方案 A · 真合批（N 路塊合成一次 batch forward）

**做法**：新增 generate_batch()（不動舊 generate()），收集最多 N_MAX（建議先鎖 3）個 slot 當下準備好的 chunk，疊成一個 batch 張量一次過 DiT + VAE，輸出再依序拆回各槽的 FrameSink/AudioOutBuffer。

**微批對齊窗口**：三個 session 的音訊到達節奏彼此不同步（各自講話時機不同），不能「等到剛好三個都準備好」才生成（會拖垮先講完的那個人的延遲）。做法：借用現有 `Feeder._loop` 已經有的「idle chunk」機制（`flashhead_engine_core.py:439-454`，真靜音時餵零音訊 chunk 頂著）——永遠固定跑 batch=N_MAX，沒有真實 chunk 待生成的槽位用「零音訊 idle chunk」墊，這樣：
1. 不需要「等其他人」的排程邏輯，每次批次組裝都是「有真 chunk 就塞、沒有就塞 idle」，零等待
2. 每次 forward 的 batch shape 永遠是固定的 N_MAX，torch.compile 只需要編一種靜態圖，不會被動態 batch size 逼著在線上重新編譯（重編譯稅實測約 100 秒等級，見 `deploy/glows/README.md`「全新冷編譯約106秒」——這在一通正在進行的電話裡是不能接受的停頓）
3. 代價：即使只有 1 人在講話，也用 batch=3 的算力去跑（多算 2 份 idle），如果 batch=3 forward 時間遠高於 batch=1，低併發時反而變慢——這正是 Option A 最大的效益不確定性，見下方預期加速

**畫質相關的批次內細節**（避免合批引入零損以外的行為差異，見第 4 節）：
- 噪聲取樣（`flash_head_pipeline.py:234-241` 初始 noise、276 行 denoise 迴圈內的 torch.randn）必須維持每個 session 各自的 torch.Generator 各自呼叫（torch.randn 一次只能吃一個 generator），取完再 torch.stack 成 batch，才能保證跟現行「每槽各自的 pipeline 各自的 generator 各自跑」在 RNG 軌跡上完全一致——這步驟本身開銷極小（CPU/GPU 隨機數生成不是瓶頸），只有 DiT forward 跟 VAE 是真正合批、吃效能的部分
- 不同 session 可能掛不同角色（a05/a06/demo a05d/a06d，`flashhead_server.py:73-99`）：prepare_params()（`flash_head_pipeline.py:180-192`）本來就會把每個角色的 ref_img_latent 都預先算好存進 ref_img_latent_dict，合批只要照 session 對到的角色去查表疊批次，不用重算——但索引錯位＝臉互換，風險等級比現行 3.2 節「串線隔離」測項更高（現行是連線層級隔離，合批後是同一次 forward 內部的張量順序對應，任何 off-by-one 都是資料污染而非連線錯誤，見第 6 節風險清單）

**預期加速（今晚數字推估、待實驗卡驗證）**：1 路 44% 餘裕（535/960ms）證明卡上有閒置產能，理論上 batch=3 一次 forward 的 wall time 不會是 3×535=1605ms（那是零批次效益的下限），也不會是 535ms（那是無限批次效益的不可能上限）。抓一個有推論依據但沒量過的區間：樂觀估 700–950ms（batch 效率高，貼近甚至達到 768ms 目標）；保守估 950ms–1.3s（若 VAE conv3d 段落受限於記憶體頻寬而非算力，batch 增益打折）——這兩個數字都是假設區間，不是量測值，Phase 1 第一步就是把這個區間量出真值，這是決定要不要往下做的關鍵指標。

**工程量**：大（新增 generate_batch() + 修 rope_apply/LtxVAE 兩處上游 patch + 新 BatchCoordinator 模組 + Head.forward/AudioProjModel 的 batch=2 smoke test）。
**風險**：見第 6 節（索引錯位臉互換、compile 靜態 batch 卡死彈性路數、上游未來版更新要重打 patch）。
**torch.compile 相容性**：固定 batch=N_MAX 是安全解法，見上述設計。

### 方案 B · 並行 CUDA streams + 拔除 debug 硬煞車

**做法**：兩件事，都只動 vendored 上游檔案，不動 flashhead_engine_core.py/flashhead_server.py 的槽位邏輯：
1. 把 flash_head_pipeline.py 內 8 處 torch.cuda.synchronize()（244/278/285/290/295/301/306/309）改成用環境變數門控（例如 MUNEA_FH_PROFILE_SYNC=0 預設關掉這些純計時用的 sync，需要除錯時開回來）——這些呼叫的唯一用途是量測並印出 wall-clock（`flash_head_pipeline.py:279-281` 等），denoise 迴圈的資料相依鏈全部在 GPU 端由同一條 stream 自然排序，不靠這些 host sync 保正確性
2. 在 Feeder._gen_chunk（`flashhead_engine_core.py:350`）呼叫 run_pipeline 前，用 `with torch.cuda.stream(slot.cuda_stream):` 包起來，讓每個槽的 kernel 提交到專屬 stream（Slot 需要新增一個 cuda_stream 欄位，wake()/_wake_slot 階段建立）——讓硬體排程器有機會真正 overlap 三條獨立的 kernel 序列，而不是被 host 端屏障逼著假性序列化

**預期加速（待實驗卡驗證）**：因為完全不改變總運算量（還是三次獨立 batch=1 forward），天花板比 Option A 低——理論上限接近「三路真並行、互不等待」，但不會突破單路 forward 本身的算力/頻寬瓶頸。抓推論區間：3 路 p95 從今晚 2284ms 降到約 1300–1700ms（介於今晚 2 路的 1309ms 與今晚 3 路之間，因為拔掉的是最主要的「不必要屏障」部分），不保證能壓到 768ms 目標——這也是待驗證數字。

**工程量**：小（拔 8 個 sync call + 加 per-slot stream，都是局部改動，不碰資料結構、不碰上游模型的 batch 語意）。
**風險**：
- 拔 sync 後長跑穩定性要重測——這些 sync 順便讓 CUDA caching allocator 的統計更新更即時，拔掉後 VRAM 洩漏偵測時機可能改變，需要 30-60 分鐘長跑驗證沒有緩慢增長（比照 `docs/多人併發容量架構-2026-07-12.md` §3.2 測項 4）
- per-slot stream 的實際效果高度依賴 kernel 本身夠不夠「小」——如果 DiT 的單一 kernel 已經吃滿一整批 SM，多 stream 也擠不出並行空間，這點沒把握，必須量
**torch.compile 相容性**：零新風險——每槽仍各自對各自的 pipeline 做 batch=1 forward，torch.compile 看到的 shape 跟現在一模一樣，不會觸發任何新的重編譯行為。副作用：拔掉 host sync 有時反而讓 CUDA Graph capture 更容易成功（sync 是 CUDA Graph capture 的天敵之一），對 `docs/多人併發容量架構-2026-07-12.md` §4.2 節的 CUDA Graph 路線是額外助攻，非必要但值得記一筆。

---

### 方案 C · 混合（B 打底 + A 只做「滿 3 路」這一條固定路徑）

**做法**：Option B 的兩個改動（拔 sync、per-slot stream）任何路數都套用，因為零 compile 風險、工程量小、值得當底盤；在此之上，只有「3 個槽同時都活躍」這個特定情境才觸發 Option A 的合批路徑（MUNEA_FH_BATCHING=1 且 MUNEA_FH_SLOTS=3），1 路/2 路維持現行各自獨立生成（但吃到 Option B 的 stream 優化）。

**為什麼這樣分層**：
- torch.compile 只需要準備兩種靜態圖（batch=1 legacy 路徑 + batch=3 合批路徑），不用處理 batch=2 這個中間態，降低編譯矩陣複雜度
- 3 路是今晚實測最慘的情境（-138% 餘裕），優先資源投在刀口上
- 2 路（-36% 餘裕）雖然也超預算，但用 Option B 的 stream 優化 + 拔 sync 大概率就能拉回安全帶，不必為它額外背合批的索引錯位風險
- 出問題時的降級路徑乾淨：合批路徑（新代碼、風險較高）關掉，自動退回「B 打底的獨立三槽」，不會整條退回最原始的滿版序列化

**預期加速**：3 路吃 Option A 的估算區間（700ms–1.3s，待驗證）；1-2 路吃 Option B 的估算區間（比現行今晚數字略好，待驗證）。

**工程量**：A + B 兩者之和，但因為 batch 路徑只需覆蓋「3 路」這一種固定 shape，比 Option A 獨立做（要考慮 1/2/3 動態切換）範圍更窄。

**建議**：先做 Option B 當作低成本驗證彈（半天到一天內可以量到「拔 sync + per-slot stream」到底有沒有用、值多少），數字回來後再由 Edward/霍爾決定要不要往 Option C 的合批深挖——如果 Option B 就能把 3 路拉到接近 768ms 目標，Option A/C 的額外工程量跟風險可能不值得冒。

## 3. 相容性鐵律

照抄 `flashhead_server.py:9,17-21` 已經立過的先例（2026-07-12 N 槽改造時的規矩）：

1. **MUNEA_FH_SLOTS 未設或設 1 時，行為必須跟現行單例版一字不差**——這條鐵律已存在，本次施工不能破壞。
2. **MUNEA_FH_SLOTS=3 但未開新機制時，行為必須跟現行（今晚量測的）3 槽版本一字不差**——這是本次施工新增的第二條鐵律，因為 Option B/C 都會動到目前 3 槽已經在跑的程式碼路徑（vendored flash_head_pipeline.py 的 sync 呼叫是全域的，不分槽），必須確保「新環境變數不開＝完全沒差」。
3. **新機制一律用新環境變數開關、預設關**：
   - `MUNEA_FH_PROFILE_SYNC`（Option B）：預設 1（保留現行 sync 行為，零風險）；0 才拔掉除錯用 sync。注意方向跟大部分開關相反——這裡「開＝現行行為、關＝新行為」，因為現行行為本來就是「開 sync」，要說清楚避免上線時開關方向搞反。
   - `MUNEA_FH_SLOT_STREAM`（Option B）：預設 0（不建立 per-slot stream，跟現行一致）；1 才啟用。
   - `MUNEA_FH_BATCHING`（Option A/C）：預設 0（不合批，各槽各自跑，跟現行一致）；1 且 MUNEA_FH_SLOTS=3 才啟用合批路徑。三個環境變數各自獨立開關，方便施工時逐項隔離測試，不必一次全開。
4. **上游 vendored 檔案的 patch 要用可重放機制記錄**，不能手改忘記留痕——flash_head_pipeline.py/flash_head_model.py/ltx_video.py 是整包 git checkout 部署到每台 GPU 機器上的第三方副本（FH_REPO=/root/SoulX-FlashHead，`flashhead_server.py:48-50`），不是 pip 套件。所有 patch 存成 `deploy/flashhead-patches/*.patch`，機器重建 SOP（`deploy/glows/README.md` 重建步驟 3-4）要多一步「apply patch」，避免下次重建機器或上游更新版本時漏補這些改動——這是新增的維運債，第 5 節施工計畫要把這步寫進 SOP。
5. **deploy/glows/README.md 與 deploy/runpod-avatar 的部署提醒要同步更新**（比照現行「兩個檔案要一起傳」的提醒風格）——若 Option A/C 落地，開機需要多傳 patch 檔或多一支 flashhead_batch_core.py，比照現行「只傳一支會 ImportError」的警語補一句。

## 4. 畫質零損驗證計畫

**目標**：同一段音訊、同一個角色、同一個 seed，新舊兩條路徑（現行序列化 vs 合批/streams 優化後）輸出的每一幀必須逐幀比對通過，不接受「肉眼看起來差不多」。

**驗證方法（自動化腳本，建議放 tools/face-acceptance/，比照現有驗收腳本風格）**：

1. **固定輸入**：沿用 poc-mandarin.wav（既有測試素材，deploy/glows/README.md 全篇都用它）、固定角色（a05）、固定 base_seed=42（現行程式碼本來就寫死這個值，`flashhead_server.py:208`）。
2. **舊路徑基準**：跑現行 flashhead_server.py（MUNEA_FH_SLOTS=1，不開任何新開關），對每一塊輸出的 frame array 存成 .npy（video.cpu().numpy()，`flashhead_engine_core.py:369`）。
3. **新路徑輸出**：同一段音訊，開對應新開關（Option B 或 A/C）重跑，同樣存 .npy。
4. **逐幀比對**：
   - **位元級**：np.array_equal(old_frame, new_frame)——Option B（不改變運算內容、只改排程跟 sync 時機）理論上應該能過位元級比對，因為 GPU 浮點運算的執行順序若真的完全不變，結果應該 bit-exact；若過不了位元級但過得了 PSNR 級，代表 sync 拿掉後某些 kernel 的 non-determinism（例如某些 cuDNN 演算法在允許非確定性模式下換了不同 kernel 實作）浮現出來，需要另外查證是否要開 torch.use_deterministic_algorithms(True)
   - **PSNR 級**（Option A 合批預期無法位元級對齊，因為浮點加總順序改變是正常的數值誤差來源）：skimage.metrics.peak_signal_noise_ratio，訂一個門檻（建議 ≥40dB，人眼幾乎無法分辨的等級），低於門檻視為失敗
5. **RNG 軌跡對齊自檢**（Option A 專屬，見第 2 節「噪聲取樣」設計）：额外寫一個更嚴格的檢查——把批次路徑裡各 session 各自的 torch.Generator 狀態單獨抽出來，跟同一段輸入下「單獨跑非批次版」的 generator 狀態序列比對，確認合批沒有意外讓 RNG 抽取順序偏移（這是 Option A 最容易踩的隱形雷：如果批次組裝邏輯不小心讓某個 session 多抽或少抽一次隨機數，之後幾塊的畫面會慢慢跟基準漂移，但單看某一塊可能還在 PSNR 門檻內，只有長跑幾十塊之後才會肉眼看出「臉開始跑掉」）。
6. **長跑重驗**：不只測單塊，測 20+ 塊連續生成（比照 bench_fh.py 預設跑 40 塊的量級），確認誤差不隨時間累積放大。
7. **通過門檻**：Option B 目標「位元級全過」；Option A/C 目標「PSNR ≥40dB 全過 + RNG 軌跡自檢無偏移 + 20 塊長跑誤差不放大」。任一項不過，视为「非零損」，退回檢討，不得帶著已知畫質差異上線。

## 6. 風險清單

1. **塊到達時序不齊**：三個 session 講話節奏不同步，合批需要「固定 batch=N_MAX + idle chunk 墊位」策略（第 2 節已定），但這代表低併發時也要付出 batch=3 的算力成本，若 Phase 1 量出來的 batch=3 wall time 沒有明顯優於序列化，這個 trade-off 就不划算。
2. **一路斷線重連時 batch 怎麼縮**：現行 SlotPool.release()（`flashhead_engine_core.py:518-527`）跟故障隔離（_on_fault/force_release_slot）都是「單槽獨立處理」的邏輯，合批後某一槽斷線，BatchCoordinator 要能正確把該槽換成 idle 墊位，不能讓斷線槽的殘留狀態污染其他兩槽的合批結果——這是新的協調邏輯，現行程式碼完全沒有對應機制，要從零設計。
3. **torch.compile 重編譯稅**：第 2 節已用「固定 batch=N_MAX + padding」規避動態 batch size 問題，但若未來要支援 N_MAX 以外的路數（例如某天要測 4 路），會需要重新編一種新的靜態圖，每次都要付冷編譯稅（deploy/glows/README.md 記載約 106 秒等級）——這是這個設計的固有限制，不是 bug，但要讓知道的人知道「N_MAX 不是隨便改的數字」。
4. **臉互換風險（合批專屬、比現行串線隔離風險更高）**：rope_apply/角色 ref_img_latent 疊批次時任何 index 對不上，後果是「A 的臉長在 B 的通話上」——這是同一次 forward 內部的張量順序錯位，不是連線層級能攔的錯誤，必須在 Phase 1 一開始就用「3 種不同輸入餵 3 個 batch 位置、逐一比對輸出只反映自己的輸入」做正面回歸測試（比照 `docs/多人併發容量架構-2026-07-12.md` §3.2 測項 3 的既有方法論，但這次是測合批內部索引，不是測連線隔離）。
5. **上游未發現的第二層 batch=1 假設**：rope_apply/LtxVAE 是目前讀碼確認的兩處，但 Head.forward 的 `F = t_mod.shape[0] // B` 這行、AudioProjModel 的 reshape 鏈還沒實測過 batch>1——不排除還有第三、第四處沒被讀出來的隱藏假設，Phase 1 開頭的 batch=2 smoke test 就是為了儘早抓出來，避免深入整合後才發現。
6. **vendored 第三方 patch 的維運債**：所有改動都動到 SoulX-FlashHead 這個 vendored 副本，不是我方 repo 原生程式碼，上游若之後有新版本更新（例如換更好的模型權重），現有 patch 需要重新對齊，若沒有第 3 節第 4 條的可重放 patch 機制，很容易在下次機器重建或升級時漏補。
7. **拔 sync 後的長跑穩定性**：torch.cuda.synchronize() 順便讓 caching allocator 的統計更即時，拔掉後 VRAM 洩漏偵測時機可能改變，Phase 0 一定要做 30-60 分鐘長跑，不能只看單塊數字就判定過關。
8. **不同 GPU 型號的合批效益不通用**：今晚數字是 RTX 4090；docs/RTX6000Ada-真768容量驗收-2026-07-13.md 等文件顯示不同卡（RTX 6000 Ada 48GB、RTX 5090）的算力/顯存/瓶頸特性不同，Option A/B 的效益數字只對 4090 有效，若之後主力機換卡型，第 2 節的估算區間要重新量，不能沿用。
9. **商業判斷邊界（技術不代答）**：合批犧牲的是「單人情境下的最低延遲」去換「滿載吞吐」（尤其 Option A 的固定 batch=N_MAX 設計，1 人講話也要付 3 人份算力），值不值得為了省卡／衝路數而讓單人體驗多等一點，這是產品/商業判斷，卡西法只給技術路徑跟數字，拍板交給霍爾/Edward。
