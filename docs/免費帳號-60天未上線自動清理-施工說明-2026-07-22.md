# 免費帳號 60 天未上線自動清理 · 施工說明（2026-07-22）

> Edward 拍板：「免費會員如果 60 天未登入上線，系統自動刪除該會員資料。」
> 本文件＝機制長什麼樣、怎麼上線、怎麼驗。純後端、**不影響 App 包版**。

## 機制一句話

每天一次，系統把「免費、60 天沒上線、沒踩任何保護傘」的帳號連鎖刪除
（資料庫個人資料＋登入身分一起清），刪之前 7 天先蓋警示章留通知窗口，
每一筆刪除都留審計證據。

## 「上線」怎麼算（為什麼不只看登入事件）

雲端的「最後登入時間」只在**重新登入**時更新；App 用既有登入證可以
連續用好幾個月都不產生新登入事件——只看它會誤刪活躍用戶。
所以取六路訊號的最近值：

1. App 每次開機蓋的上線章（本次新增，`/account-bootstrap` 成功就蓋）
2. 雲端正式登入事件
3. 推播裝置登記的最後露面時間
4. 最後一通聊聊
5. 錢包最後一筆異動（含免費體驗扣點）
6. 帳號建立日（兜底：從沒上線過的從建立日起算）

## 五道保護傘（碰到任何一道就不刪、也不警示）

| # | 誰 | 為什麼 |
|---|----|--------|
| A | 測試帳號（欄位標記或 @munea.net） | 自己人的 QA 帳號 |
| B | 有活訂閱（試用/訂閱中/寬限期） | 付費會員不在政策範圍 |
| C | 購買點數餘額 > 0 | 收過人家錢、東西不能拿走 |
| D | 企業席次在身 | B2B 客戶的人 |
| E | 是別人家庭圈的成員 | 刪他會把他從付費圈主的家庭圈踢掉 |

## 兩段式節奏

- **第 53 天**（60−7）：蓋警示章。之後通知管道（推播/Email）好了，就在這一步發「再不回來就要說再見了」。用戶回來上線＝清章重算。
- **第 60 天**：警示章蓋滿 7 天才真刪。刪除走「本人按帳號刪除」同一條連鎖路＋清登入身分；審計紀錄先落地、不隨帳號消失。

## 護欄

- **預設乾跑**：接口沒明講 `dryRun: false` 一律只出報告、不動資料。
- **單輪上限**：一天最多刪 20 個（參數可調、硬上限 200）——出蟲也炸不了全庫。
- **天數下限 30**：防手滑把 60 打成 6。
- **告警**：真刪有動作就發維運告警；登入身分清不掉會明講、不假裝成功。

## 上線步驟（照順序）

1. **跑 024 SQL**：Supabase 東京後台 SQL Editor 貼 `024_inactive_free_account_retention.sql` 執行（正式測試共用同一份東京庫、跑一次就好）。
2. **部署 Brain**：先測試機 `munea-brain-staging`、再正式 `munea-brain`（照 prod-deploy.sh 規矩；**記得帶完整 env**，deploy 掉 env 的地雷見 2026-07-12 教訓）。不用動 Voice／Gateway／App。
3. **乾跑驗證**（不動資料、隨時可跑）：

   ```bash
   curl -s -X POST "https://<brain-url>/admin/retention/run" -H "Content-Type: application/json" -H "X-Munea-Admin-Token: <管理鑰匙>" -d "{}"
   ```

   看報告：掃到幾個免費帳號、誰會被警示（`warnedSample`）、誰已到刪除線。確認名單合理（測試帳號、企業、付費的都不該出現）。
4. **掛定時鬧鐘**（Cloud Scheduler、比照晨料鬧鐘）：

   ```bash
   gcloud scheduler jobs create http munea-retention-daily --location=asia-northeast1 --schedule="30 4 * * *" --time-zone="Asia/Taipei" --uri="https://<brain-url>/admin/retention/run" --http-method=POST --headers="Content-Type=application/json,X-Munea-Admin-Token=<管理鑰匙>" --message-body="{\"dryRun\": false}"
   ```

5. **首刪發生在鬧鐘上線 7 天後**（第一輪只蓋警示章），有緩衝期盯報告。
