## 0. 白話摘要（Edward 60 秒看完這段就好，細節在後）

沐寧要存「長輩最私密的健康 + 生活資料」，這塊我把它分成**四個上鎖的抽屜**，每個抽屜放不同東西、用不同的櫃子、配不同的鑰匙：

| 抽屜 | 放什麼 | 白話比喻 |
|---|---|---|
| ① 用戶檔案 | 名字、家人、喜好、健康底子（慢性病、用藥、禁忌話題）| 長輩的「身分證 + 生活說明書」 |
| ② 健康數字 | 步數、睡眠、血壓、吃藥紀錄（一條時間軸）| 長輩的「生命徵象日誌」 |
| ③ 回憶 | 聊天淬煉出的「意思」（可語意搜尋）| 寧寧腦中「我記得你說過…」的那本記事本 |
| ④ 對話紀錄 | 原始逐字對話 | 「錄音存底」（最敏感、最該短期就刪） |

**最關鍵的三件事，我先用白話講清楚，Edward 一定要知道：**

1. 🟢 **好消息：沐寧的健康數字「不算」法律上最嚴的『特種個資』——只要我們守住『陪伴非醫療』那條線。** 台灣個資法把「病歷/醫療/健康檢查」列為最高規格保護（原則禁止蒐集），但國發會函釋說得很清楚：**長輩自己量、直接給我們、沒有醫生看診介入** → 就不是特種個資。這跟霍爾的品牌定位「陪伴非醫療」是同一條命脈——**醫療紅線守得住，這條法律保護傘就在；守不住（變成「幫你管理血糖」），就可能掉進特種個資、整個合規難度跳級。** 所以禁用詞表不只是行銷問題，是法律問題。〔已查證國發會函釋 · 但「直接適用沐寧情境」建議律師最終確認〕

## 1. 四抽屜細部設計（每個存什麼欄位、用哪種資料庫、為什麼）

> 設計原則：**資料最小化**（個資法核心）——只存「為了陪伴 + 安全網真正需要」的，不貪心。每個抽屜標「資料分類」（決定加密與權限等級）。

### 抽屜 ① 用戶檔案 Profile（結構化資料）

**存什麼欄位：**

| 欄位群 | 具體欄位 | 資料分類 | 備註 |
|---|---|---|---|
| 身分基本 | elder_id（主鍵·系統產生）、暱稱/稱呼、性別、年齡區間、所在地（縣市級·不存精確地址）| 🟡 PII | 年齡存「區間」不存生日、地點存「縣市」不存門牌——最小化 |
| 家庭關係 | family_group_id、家人成員[]（關係/暱稱/可聯絡方式）| 🟡 PII | 子女的 email/手機是「緊急聯絡用」、加密存 |
| 喜好人格 | 愛聽的（老歌/戲曲）、興趣、講話偏好、信仰、禁忌話題[]（過世老伴、傷心事）| 🟡 敏感（非特種）| 「禁忌話題」是尊嚴關鍵、餵給 AI 當「別踩的雷」 |
| 健康底子 | 慢性病史[]、長期用藥[]、過敏、行動能力、有無 Apple Watch | 🟠 **準特種·最敏感** | **見下方法律分析**——存「長輩自述/子女填」的健康底子，不是醫生診斷 |
| 角色設定 | 選的角色（寧寧/小昀/…）、語音偏好、提醒時間設定 | 🟢 一般 | |

**用哪種資料庫**：**結構化關聯式資料庫**（PostgreSQL）。理由：欄位固定、要做 JOIN（長輩 ↔ 家庭群組）、要強一致性（帳本相關）、要 row-level 權限控制（elder_id 隔離）。
- 🟠 **健康底子（慢性病/用藥）建議獨立資料表 + 獨立加密金鑰**：即使整體 DB 被入侵，這張表也是分開鎖的（envelope encryption，見第 2 節）。

〔已確定設計方向 · 欄位細節待 codex 實作時定 schema〕

### 抽屜 ② 健康數字 HealthTimeseries（時間軸資料）

**存什麼欄位：**

| 指標 | 來源 | 頻率 | 資料分類 |
|---|---|---|---|
| 步數、活動量 | Apple 健康（HealthKit）| 每日彙總 | 🟠 準特種 |
| 心率、睡眠 | Apple Watch / 健康 | 每日彙總 | 🟠 準特種 |
| 血壓 | 長輩手動 / 藍牙血壓計 / 健康 | 量測時 | 🟠 準特種 |
| 用藥紀錄（吃了/沒吃/延後）| 長輩在 App 回應提醒 | 每次提醒 | 🟠 準特種 |
| 跌倒事件 | Apple Watch 偵測 | 事件觸發 | 🔴 危急·直連守護腦 |

**用哪種資料庫**：**時間序列資料庫**（TimescaleDB——PostgreSQL 的時序擴充，或獨立的 InfluxDB）。理由：
- 資料本質是「時間戳 + 數值」的連續流，時序 DB 對「最近 7 天血壓趨勢」這種查詢快上數量級。
- 自動 retention policy（過期自動降採樣/刪除）——直接滿足個資法「資料保存期限」要求（見第 3 節）。
- 與抽屜 ① 同一套 PostgreSQL 體系（TimescaleDB）可降運維複雜度——**我建議用 TimescaleDB 而非另開 InfluxDB**，少一個要分艙、要加密、要備份的系統 = 少一個出事點。

**⚠️ 設計關鍵**：每一筆都帶 elder_id（卡西法分艙鐵律）+ 標「來源」（Apple 健康/手動/裝置）。「來源」欄位是合規證據——證明這是「長輩自己的數據」不是「醫療機構的病歷」。

〔已確定設計方向 · TimescaleDB vs InfluxDB 最終由卡西法定〕

### 抽屜 ③ 回憶 Memory（語意/向量資料）

**存什麼**：不是存逐字對話，是存「淬煉後的意思」——
- 範例：長輩聊天提到「我膝蓋一到冬天就痛」「我孫子小寶下禮拜考試」「我最討厭別人提我老伴的事」
- 淬煉成記憶條目：`{type: 健康觀察, content: "膝蓋天冷會痛", source_date, confidence}`、`{type: 家人事件, content: "孫子小寶考試"}`、`{type: 禁忌, content: "勿提過世老伴"}`

**欄位：**

| 欄位 | 用途 | 分類 |
|---|---|---|
| memory_id, elder_id | 主鍵 + 分艙 | — |
| type | 健康觀察/喜好/家人事件/禁忌/生命故事/情緒趨勢 | 🟠 敏感 |
| content（文字）| 淬煉後的意思 | 🟠 敏感（含健康觀察時準特種）|
| embedding（向量）| 語意搜尋用 | 🟠 |
| source_date, last_used, confidence | 時間衰減 / 可信度 | 🟢 |

**用哪種資料庫**：**向量資料庫**（pgvector——又是 PostgreSQL 擴充，或獨立的 Qdrant/Weaviate）。理由：
- 寧寧要做「我記得你說過…」的調動 = 語意搜尋（「天冷」要能搜到「膝蓋冬天痛」），這是向量檢索的本職。
- **我同樣建議優先 pgvector**（長在既有 PostgreSQL 上），理由同上：少一個獨立系統 = 少一個分艙/加密/出事點。除非規模大到 pgvector 撐不住才獨立。

**🔴 隱私設計關鍵（這抽屜最容易出事）**：
- 「健康觀察」型記憶（膝蓋痛、最近睡不好）= **準特種個資**，跟②的數字一樣等級保護。
- **記憶 = AI 對長輩的「畫像」**，比原始對話更敏感（它是「結論」）。這抽屜的存取必須跟②同級加密 + 同級權限。
- **禁忌話題（type:禁忌）是雙面刃**：存它是為了「保護長輩不被踩雷」（善意），但它本身揭露長輩傷痛（過世老伴、家庭裂痕）→ 加密 + 絕不出現在子女儀表板（第 4 節）。

〔已確定設計方向 · pgvector vs 獨立向量庫由卡西法定 · 「記憶淬煉」邏輯是管家腦的活、卡西法 + 霍爾人格層〕

### 抽屜 ④ 對話紀錄 Transcript（原始紀錄）

**存什麼**：原始逐字對話（長輩說什麼、寧寧回什麼）。

**🔴 這是四抽屜裡最該「短命」的一個：**
- **存的理由**：① 淬煉記憶的原料（淬煉完其實可以丟）② 出事時的 audit（危機誤判/客訴回溯）③ 模型品質改善。
- **不存/短存的理由**：原始逐字 = 長輩最私密的話（包括病情、家庭祕密、情緒崩潰），存越久風險越大；個資法資料最小化原則直接指向「用完就刪」。

**我的設計建議（強烈）：**

| 選項 | 做法 | 我的評估 |
|---|---|---|
| A · 不存原始、即淬即棄 | 對話完成記憶淬煉後立刻刪逐字，只留③回憶 | 🟢 最乾淨、最合規、最尊嚴。**但失去 audit 能力**（危機誤判無法回溯） |
| B · 短期保留 + 自動刪 | 逐字存 7-30 天（夠 audit + 淬煉），到期自動硬刪 | 🟡 **我建議這個**——平衡 audit 與隱私，retention 寫死在系統 |
| C · 長期保留 | 永久存逐字 | 🔴 不建議，風險最高、最難跟長輩交代 |

**用哪種資料庫**：一般文件/物件儲存（PostgreSQL JSONB 或物件儲存 S3-like）+ **強制 TTL（time-to-live 自動過期刪除）**。

〔🔴 **保留天數待 Edward 拍板**：建議 B（7-30 天自動刪）· 這直接影響「出事能不能回溯」vs「隱私風險」的取捨〕

### 四抽屜總表（一眼看完）

| 抽屜 | 資料庫類型 | 最敏感等級 | 加密金鑰 | 保留策略 |
|---|---|---|---|---|
| ① 用戶檔案 | 結構化（PostgreSQL）| 🟠 健康底子準特種 | 健康底子獨立金鑰 | 帳號存續期間 |
| ② 健康數字 | 時序（TimescaleDB）| 🟠 準特種 | 獨立金鑰 | 滾動保留（建議明細 1 年 + 趨勢摘要長期）|
| ③ 回憶 | 向量（pgvector）| 🟠 準特種（健康觀察）| 獨立金鑰 | 帳號存續期間 + 時間衰減 |
| ④ 對話紀錄 | 文件/物件 + TTL | 🔴 最敏感 | 獨立金鑰 | 🔴 **建議 7-30 天自動刪（待拍板）** |

**沙利曼設計總原則**：四抽屜**優先全部長在 PostgreSQL 生態**（關聯式 + TimescaleDB + pgvector），除非規模逼迫才拆獨立系統。理由 = 每多一個獨立資料庫，就多一套要分艙、要加密、要備份、要稽核、會漏 elder_id 的地方。**對個人作坊規模的城堡，「少一個出事點」比「每個 component 最優」更重要。**〔已確定建議 · 卡西法可基於負載反駁〕

## 2. 加密 + 權限 + 同意（傳輸/儲存加密、誰能看、子女看長輩的同意機制）

### 2.1 加密（傳輸 + 儲存）

**傳輸中加密（in-transit）：**
- 所有 App ↔ 後端：**TLS 1.3**（HTTPS）。
- WebRTC 串流（avatar 影音）：**DTLS-SRTP**（WebRTC 內建端對端加密）。
- 後端 ↔ 雲端語音 API（OpenAI/Google）：TLS + 見 2.4 境外資料處理。
〔已確定 · 業界標準、無爭議〕

**靜止中加密（at-rest）：**
- 資料庫整體：雲端 DB 預設磁碟層加密（AES-256）。
- 🟠 **準特種資料（健康底子/健康數字/健康觀察記憶）額外做「欄位級加密」（envelope encryption）**：
  - 每個 elder_id 一把資料金鑰（DEK），DEK 用主金鑰（KEK）加密、KEK 放雲端 KMS（Key Management Service）。
  - 效果：就算整個資料庫被拖走，沒有 KMS 的 KEK 也解不開健康資料。
  - **這是「準特種個資」配得上的保護等級**——不是所有欄位都要，但健康相關的值得。
〔已確定設計方向 · KMS 選型（AWS KMS / GCP KMS / Modal 環境）待卡西法定〕

**金鑰與機密管理（我的本業·Gate 5 鐵律）：**
- gpt-realtime-2 / Google / RevenueCat / Apple 的 API key、signing key：**一律環境變數 / Secret Manager，絕不硬寫進 code，絕不進 git**。
- 上線前我跑 secret 掃描（ghp_/AKIA/sk_live_/長 token pattern），命中 = 立刻 redact + rotate。
〔已確定 · Gate 5 標準動作〕

### 2.2 權限（誰能看誰不能看）

**權限主體與可見範圍：**

| 主體 | 能看什麼 | 不能看什麼 |
|---|---|---|
| 長輩本人 | 自己的全部（但 UI 上「永不見點數計量」是商業設計、非權限）| 別的 elder 的任何資料 |
| 子女（家庭成員）| **長輩授權範圍內**的健康摘要/安心摺（見 2.3）| 🔴 ④原始對話逐字、③禁忌話題記憶、長輩未授權的詳細數字 |
| 沐寧系統/AI | 該 elder_id 分艙內全部（提供服務必需）| 跨 elder_id（卡西法 DAO 強制 elder_id、缺即拋錯）|
| 沐寧員工/營運 | 🔴 **原則上零接觸個人健康資料**；除錯需要時走「最小權限 + 留痕 + 去識別」| 直接讀任何長輩的健康/對話明文 |

**🔴 三條權限硬底線（我會在 Gate 5 逐項審）：**
1. **跨戶物理隔離**：DAO 層強制 elder_id 過濾、缺了拋錯（卡西法已定，我背書——這是「上新聞等級」的防線）。
2. **子女 ≠ 長輩**：子女能看的是「長輩授權的摘要」，不是「長輩帳號的全部」。預設窄、授權才寬。
3. **員工最小接觸**：營運/除錯不該能裸讀長輩對話。需要時走去識別 + 留痕（誰在什麼時候因什麼看了什麼）。

〔已確定設計方向〕

### 2.3 🔴 子女看長輩資料的「同意」機制（Edward 點名要的尊嚴設計）

**核心原則：長輩授權，不是子女裝了就能全看。**

這是沐寧最特殊的隱私難題——**操作者（子女）≠ 使用者（長輩），而且付錢的是子女。** 一般 App「用戶看自己資料」很單純；沐寧是「子女想看『另一個成年人』的私密健康資料」，這在法律與倫理上都需要**被看的那個人（長輩）的同意**。

**我設計的三層可見度（長輩控制、預設保護）：**

| 層級 | 子女看得到 | 預設 | 長輩如何控制 |
|---|---|---|---|
| L1 安心燈號 | 「媽媽今天好/要注意」綠燈紅燈、有沒有完成今天的事 | ✅ **預設開** | 這層是「安全網最低限」、裝機時長輩同意一次 |
| L2 健康摘要 | 用藥順從度、活動趨勢（綠笑臉/白話、非精確數字）| 🟡 預設開（可關）| 長輩可在設定關掉某項 |
| L3 詳細數字 | 精確血壓值、睡眠時數、詳細時間軸 | 🔴 **預設關** | 長輩主動點頭才開、可隨時收回 |
| L4 永不給 | 原始對話逐字、禁忌話題、情緒崩潰紀錄 | 🔴 **永遠關·無開關** | 系統層禁止、連長輩想開都不提供（保護長輩免於被情緒勒索）|

**同意流程（裝機時 + 持續）：**
1. **裝機時**：子女裝好 → 長輩端第一次啟用，寧寧用語音引導長輩做「家人可見度設定」——「美華（女兒）想關心你，要不要讓她看到你每天有沒有好好的？」長輩說好 → 開 L1。詳細的問清楚再開。
2. **這是「告知後同意」（informed consent）**：不是埋在條款裡的勾選，是寧寧用長輩聽得懂的話講清楚、長輩口頭/點頭同意，**並留下同意紀錄**（誰、何時、同意了哪幾層）。
3. **可撤回**：長輩隨時可跟寧寧說「我不想讓他們看那個了」→ 收回授權。個資法的「當事人權利」要求可撤回。

**⚠️ 這裡有一個 Edward 必須拍板的張力：**
> 子女是付錢的人，會想「我花錢就是要能看清楚爸媽狀況」。但長輩有尊嚴與隱私權。**這兩者衝突時，預設偏哪邊？**
> - 我的建議：**預設偏保護長輩**（L3 詳細數字預設關），子女要更多 → 走「徵詢長輩同意」。理由：① 法律上長輩是資料當事人、② 品牌上「尊重長輩」是沐寧人設、③ 出事時（長輩抱怨被監控）站得住。
> - 但若 Edward 從商業（子女付費滿意度）判斷要更開放，那是你的決定——我把風險講清楚：**過度開放 = 長輩「被監控」感 + 潛在個資爭議**。

〔🔴 **待 Edward 拍板**：預設偏長輩保護（建議）vs 偏子女知情權 · 這是產品價值觀層級的決定〕
〔待律師確認：成年子女查看另一成年長輩健康資料，台灣個資法下「同意」的形式要件（是否需書面/可口頭+紀錄）〕

### 2.4 同意的法律形式（避免「同意」變成不算數）

個資法第六條的例外是「**書面同意**」，但施行細則說**書面同意得以電子文件為之**。對沐寧：
- 健康相關資料蒐集的同意 → **電子同意（App 內，可含寧寧語音引導 + 長輩確認 + 系統留紀錄）** 應可滿足。
- **但**：長輩是高齡、可能認知退化、可能由子女代操作裝機——「同意是不是長輩真的理解後做的」是真風險。
- 我的設計：**雙重確認**——子女裝機填基本資料，但「健康資料蒐集 + 家人可見度」的同意由**長輩端親自確認一次**（寧寧引導、留語音/點擊紀錄），不讓子女代長輩同意敏感項。
〔🔴 待律師確認：高齡 + 可能代操作情境下，電子同意的有效性與最佳實務〕

## 3. 台灣個資法 + 醫療資料合規（碰健康數據的法遵紅線）

### 3.1 🟢 核心好消息 + 命脈：沐寧的健康數字「不算」特種個資（前提：守住非醫療）

**法律事實（已查證）：**
- 台灣個資法第六條：**病歷、醫療、基因、性生活、健康檢查**列為「特種個資」——**原則禁止蒐集**，例外之一是「當事人書面同意」。([個資法§6](https://law.moj.gov.tw/LawClass/LawSingle.aspx?pcode=I0050021&flno=6))
- **國發會函釋（關鍵分野）**：民眾「自行操作器材所得之生理數據，直接提供予業者蒐集，**若未涉及醫事人員診察、治療**」→ **非屬第六條特種個資**。([國發會函釋](https://theme.ndc.gov.tw/lawout/NewsContent.aspx?id=408))

**對沐寧的意義（命脈級）：**
- 沐寧 = 陪伴非醫療、**沒有醫師看診、沒有診斷治療**、健康數字是「長輩自己量 / Apple 健康同步」→ **踩在「非特種個資」這一側。**
- **這跟霍爾的醫療紅線、品牌定位『陪伴非醫療』是同一條命脈：**
  - 守住 = 健康數字是「一般個資」（一樣要保護，但不必走特種個資的書面同意嚴規、不必證明六款例外）。
  - **守不住 = 一旦文案/功能變成「監測/管理/診斷/評估病情」，就可能被認定涉醫療 → 掉進特種個資 → 合規難度跳級 + 可能踩《醫療法》《醫師法》非法行醫紅線。**
- **所以禁用詞表（霍爾那份）不是行銷潔癖，是法律地基。** 我在 Gate 5 會把它當「合規掃描清單」逐條掃。

**沙利曼的合規建議（即使非特種，也比照高標）：**
> 雖然法律上是「一般個資」，但健康資料社會觀感極敏感。我建議**比照特種個資的保護等級做**（獨立加密、嚴格權限、明確同意）——**法律給的是地板，我們站高一點，出事時站得住、對外（Apple 審核、客戶、媒體）也是加分。**

〔🟢 已查證國發會函釋方向 · 🟡 但「直接適用沐寧」建議律師最終確認一次，因為「陪伴 App 存血壓」是新型態、函釋是個案見解非法條〕

### 3.2 同意（合法蒐集的基礎）

- 一般個資蒐集：個資法要求**告知 + 同意**（告知蒐集目的、項目、利用範圍、當事人權利）。
- 健康相關（即使非特種）：我建議比照書面/電子同意（見 2.3、2.4）。
- **告知義務**：裝機時要有清楚的「隱私權政策」說明「我們蒐集什麼、為什麼、給誰看、存多久、你有什麼權利」——這份我可以擬底線版（非正式法律意見）。
〔待建：隱私權政策 + 服務條款（我擬底線版 → 律師定稿）〕

### 3.3 資料保存 + 刪除權（當事人權利）

個資法給當事人這些權利，沐寧要能履行：

| 權利 | 沐寧要能做到 | 設計對應 |
|---|---|---|
| 查詢/閱覽 | 長輩/子女能調自己的資料 | 已有儀表板 |
| 複製 | 能匯出 | 待建匯出功能 |
| 補充/更正 | 能改錯誤資料 | 設定頁可改 |
| 🔴 **停止蒐集/刪除** | 退訂/要求刪除時，**28 天內**可硬刪全部 | 🔴 **待建·重要** |

**🔴 「被遺忘」的設計（建議）：**
- 帳號刪除 = 四抽屜全清（含向量記憶、含對話、含時序數字）+ 通知雲端 API 端（OpenAI/Google）刪除（若其保留）。
- 退訂 ≠ 立刻刪（可能想回來），但要有「明確退訂 + 寬限期 + 自動硬刪」的流程。
- ④對話紀錄的 TTL（7-30 天自動刪）本身就是「資料最小化 + 縮短曝險窗」的好設計。
〔🔴 待建：刪除/匯出/退訂資料清理流程 · 待 Edward 定「退訂後保留多久才硬刪」〕

### 3.4 🔴 跨境資料落地（Edward 初衷矛盾的真相 · 必讀）

**事實：**
- V1 語音腦 = gpt-realtime-2（OpenAI）/ Gemini（Google）= **雲端 API、伺服器在境外**。
- **長輩講的每一句話（含病情、家庭祕密）都會傳到境外伺服器處理。**
- 台灣現行：跨境傳輸採「**原則許可、例外限制**」——一般不需事前核准，**但主管機關可隨時祭出限制**，且未來可能轉向「原則禁止、例外許可」。([理律·跨境傳輸](https://www.leeandli.com/TW/NewslettersDetail/7198.htm)、[椽智·合規指南](https://wisebeamlaw.com/data-compliance-taiwan/))
- **產業特別限制**：衛福部已限制「電子病歷雲端服務商不得為陸資」、健康資料對中國傳輸有特別限制。沐寧用 OpenAI（美）/Google（美），**不是陸資、不傳中國 → 避開最硬的紅線**。([椽智](https://wisebeamlaw.com/data-compliance-taiwan/))

**🔴 這直接戳破「全自架不付每分鐘費」的初衷——但要分兩件事看：**
1. **成本面**：自架指的是 avatar（Ditto 已解）；**語音腦 V1 確實是付費雲端 API**（STATUS 已記此矛盾）。
2. **隱私面（我補的、更重要）**：語音腦雲端 = 長輩私密對話經境外。這是「自架」初衷真正沒做到的部分，**而且隱私比成本更難跟用戶交代**。

**我的三道防線（在「V1 必須用雲端語音」的現實下）：**

| 防線 | 做法 | 狀態 |
|---|---|---|
| ① zero-retention 合約 | 跟 OpenAI/Google 簽「不保留、不拿去訓練」的 API 條款（企業版/zero-retention endpoint）| 🟠 上線前必確認、卡西法已標 |
| ② 告知同意 | 隱私政策明說「對話會經 OpenAI/Google 處理」、長輩/子女同意 | 🟠 待建 |
| ③ 落地最小化 | 只傳「對話必需」、健康數字/記憶/檔案**留在我們自己的境內/可控資料庫**，不丟給語音 API | 🟢 架構上做得到 |

**根本解（誠實講給 Edward）：**
> 要「長輩的話完全不經境外」= 得自架語音模型。但 V1 做不到——**台語/中文即時對話模型自架技術上還沒成熟**（霍爾盤點：台語全市場無即時對話模型）。所以 V1 接受「雲端語音 + 三道防線」，**這是務實取捨、不是疏漏，但要對 Edward 和用戶誠實揭露**。台語自架（V2 護城河）若成，順帶把這塊隱私收回境內。

〔🔴 待 Edward 知情拍板：接受「V1 雲端語音 + 三道防線」· 🟠 待卡西法落實 zero-retention · 🟡 跨境法規 2025-2026 在修、建議律師關注修法走向〕

### 3.5 🔴 點數 = 預付/禮券的法規（商業模式檔提的台灣消保）

**事實（已查證）：**
- 台灣對「**商品（服務）禮券 / 預付型交易**」有明確規範：發行「記載一定金額的預付憑證」，**必須提供至少一年的履約保證**——錢存銀行信託專戶 / 金融機構足額保證 / 價金保管 / 同業連帶擔保，四選一。([經濟部·禮券定型化契約](https://law.moea.gov.tw/LawContent.aspx?id=GL000996)、[消保會·預付型交易](https://cpc.ey.gov.tw/Page/8A04F6D87D628151))
- 「**線上遊戲點數**」另有專門的「定型化契約應記載及不得記載事項」、也要求履約保證。([線上遊戲點數契約](https://www.rootlaw.com.tw/LawArticle.aspx?LawID=A040100051017000-1010611))

**對沐寧點數制的風險：**
- 沐寧的點數 = 預付（子女先儲值）+ 綁定服務（換 avatar 分鐘數）。**很可能落入「禮券」或「預付型交易」範疇 → 要做履約保證（信託）。**
- **這是真金白銀的法律責任**：一旦賣點數，預收的錢可能依法要存信託專戶、不能直接當營收花、且要揭露退費機制。
- 「永不過期」的購買點數（商業模式定的）→ 也牽涉「禮券不得記載使用期限」的規定，方向上**對沐寧有利**（你本來就設永不過期），但細節要對齊定型化契約。

**🔴 這條我明確不敢替你拍——建議找律師：**
> 「綁定服務的虛擬點數是否等同禮券、要不要做信託履約保證」**是專業法律判斷**，金額一上規模就是法律與財務責任。**強烈建議在開始賣點數前，找消保/金融法律師判一次**，並請他幫忙起草「點數/儲值定型化契約」（退費、過期、履約保證的揭露）。
> （這正是我的職責邊界——我給底線提醒，$ 規模的合約找外部律師。）

〔🔴 待外部律師：點數是否屬禮券/預付型交易 + 是否需信託履約保證 + 起草定型化契約〕

## 4. 危機告警資料流（危險訊號 → 通知家人 · 隱私界線 · 保長輩尊嚴）

> 核心張力：**安全網要夠靈敏（救命）vs 不能變成監控（尊嚴）。** 這節設計怎麼在兩者間走鋼索。

### 4.1 危機資料怎麼走（資料流）

```
偵測來源（三路）
  ① 守護腦監聽對話流 → 危機關鍵詞（想不開/嚴重不適）、情緒異常
  ② 健康數字異常 → 血壓極端值、長期沒吃藥、作息劇變
  ③ Apple Watch → 跌倒偵測、心率危急
        ↓
   危機分級（不是所有警訊都驚動家人）
        ↓
  ┌─ 🟡 關懷級（情緒低落、漏吃一次藥）→ 寧寧主動多關心 + 記錄，不告警家人
  ├─ 🟠 注意級（連續數天異常）→ 通知子女「媽媽這幾天…要不要關心一下」
  └─ 🔴 危急級（跌倒/想不開/危急生命徵象）→ 立刻通知家人 + 提示 1925/119
        ↓
   通知走「推播層」（非 avatar）→ 子女手機
        ↓
   留痕（誰被通知、何時、因什麼訊號）= audit + 避免誤報無法追溯
```

### 4.2 🔴 隱私界線（不過度監控、保尊嚴）—— 我的設計守則

**守則 1 · 分級告警，不是全都報。**
- 不是「長輩一有風吹草動就 call 子女」——那是監控、會讓長輩覺得「我的一舉一動都被盯」。
- 只有**真正危急**才驚動家人；關懷級由寧寧自己接住。**這個分級門檻**要謹慎設計（太鬆=監控+狼來了、太緊=漏救）。〔待 Edward + 卡西法 + 醫療顧問定門檻〕

**守則 2 · 危機告警「告知狀態」，不「轉播隱私」。**
- 子女收到的是「媽媽跌倒了，快聯絡」「媽媽這幾天情緒不太好」——**是『狀態 + 行動呼籲』，不是『把長輩崩潰的原話/逐字轉播給子女』。**
- 🔴 **長輩跟寧寧說的私密內容（為什麼難過、家庭祕密）不進告警**——告警只說「需要關心」，不說「因為她說了 X」。**這條是尊嚴底線**：危機介入是為了救人，不是把長輩的脆弱攤給全家看。

**守則 3 · 想不開/自傷類 → 轉介優先、家人 + 專線並行。**
- 角色不自己當諮商師（醫療紅線）→ 提示 1925（心理諮詢）、119（緊急）、通知家人。
- 🔴 **這類最敏感**：要通知誰、通知到什麼程度，可能涉及長輩自己「不想讓家人知道」的意願衝突。**生命安全 vs 隱私意願的取捨**，原則上**生命優先**，但「怎麼通知才既救人又不傷害長輩信任」需要極謹慎設計。〔🔴 待 Edward 拍板原則 + 建議諮詢心理/法律專業：自傷風險下「違反當事人意願通知家人」的界線〕

**守則 4 · 告警留痕、可回溯、可申訴。**
- 每次危機告警寫紀錄（訊號、分級、通知對象、時間）。
- 用途：① 誤報時能檢討校準 ② 長輩/家人事後能理解「為什麼那天通知了」③ 出事時的責任釐清。

**守則 5 · 長輩知道有這個安全網（透明，不偷偷監控）。**
- 裝機同意時就告訴長輩：「如果你發生危險，沐寧會幫你找家人/打電話求救」——**這是『有人罩著你』的安心，不是『被偷偷監視』。** 透明揭露讓安全網變成信任，而非監控。
- 🔴 例外：自傷偵測是否要「事先完全揭露」存在兩難（揭露可能讓真有意圖者規避）→ 這點需專業意見。

〔🔴 危機分級門檻 + 自傷通知界線：待 Edward 拍板原則、建議諮詢心理/法律專業 · 🟢 告警資料流架構已可定〕

### 4.3 危機告警的資料分類
