# 即時語音 · 逐節點診斷與測試設計

> 目的：語音體驗出問題時（卡頓、延遲、亂回），**能指出是「哪一段」壞的**，而不是用猜的。
> 立此因：2026-07-01 首次試講品質不好；需要可追蹤的問題節點。

## 語音這條路的節點
```
① 麥克風  → ② 上傳  → ③ Live 處理  → ④ 回傳語音  → ⑤ 播放
（手機收音）  （送到橋）  （AI 想＋生成）  （聲音傳回）   （手機發聲）
```

| 節點 | 壞掉的症狀 | 怎麼量／看 |
|---|---|---|
| ① 麥克風 | 她聽不到你 | 面板「上行 KB」有沒有在增加 |
| ② 上傳 | 斷線、送不出 | 面板「連線」狀態、上行 KB |
| ③ Live 處理 | 反應慢、亂回 | **「首次回聲 ms」**（送出到第一聲）＋字幕內容 |
| ④ 回傳語音 | 沒聲音／很少 | 面板「下行 KB」、伺服器 `node.turn_done out_bytes` |
| ⑤ 播放 | **卡頓、斷斷續續** | 面板「播放斷點」次數（越多越卡） |

## 三種追蹤工具（都已做好）
1. **伺服器逐節點日誌**：`engine/live_voice_server.py` 每通電話印 `[diag]` 標記——
   `connected` / `node.mic_uplink` / `node.first_audio latency_ms` / `node.turn_done out_bytes audio_ms` / `node.interrupted` / `node.error` / `closed in/out bytes`。
2. **自動診斷測試**：`python engine/voice_node_test.py "一句話"`（要先開語音服務）——
   送一句話、量出①連線②送出③首次回聲④下行語音⑤字幕⑥整段完成，並判讀反應快慢。用來抓退步。
3. **瀏覽器即時面板**：測試頁網址加 `?debug=1`（例 `http://127.0.0.1:8201/?debug=1`）——
   左上角即時顯示：連線 / 首次回聲 ms / 上行 KB / 下行 KB / **播放斷點**。手機測時一眼看哪節點紅。

## 判讀範例（2026-07-01 首測）
連線 81ms、首次回聲 **560ms（伺服器量）/ 1068ms（含瀏覽器）= 快**、下行 319KB≈6.7s、字幕短且自然。
→ 結論：**網路與 AI 都健康，卡頓落在節點⑤「瀏覽器播放」**。下一步優化聚焦播放平順（加緩衝、順排程），不是動 AI。

## 門檻（判好壞）
- 首次回聲：好 < 2000ms、慢 > 4000ms
- 播放斷點：一通電話 0～2 可接受；頻繁 = 播放要加緩衝
