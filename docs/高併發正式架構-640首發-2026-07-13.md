# Munea「聊聊」高併發正式架構：640 首發版

> 日期：2026-07-13
> Owner：Codex（CTO／技術經理）
> 狀態：**正式施工唯一基準**。取代先前混用 512、640、768 或未區分 GPU profile 的容量文件。
> 產品硬規則：聊聊服務＝Gemini Realtime Voice＋VocaFrame Avatar 綁定；不得在滿載、故障或點數不足時偷偷退成純語音或靜態 Avatar。

## 0. 已拍板決策

1. **首發品質固定 640×640**：集中優化畫質、嘴型、聲畫同步、尾音與穩定度。
2. **512 正式淘汰**：不得再作 App 即時動態服務的生產 profile，也不得拿 512 壓測數字宣稱首發容量。
3. **768 暫停投入**：保留研發成果，但不進首發容量、成本與 autoscale 計算。
4. **GLOWS Unit Qty 固定 1**：一個 instance 一張 GPU、一個可獨立開關與摘除的 Worker。Qty 2 會配置兩張 GPU 並疊加價格，但現有 VocaFrame 不會自動使用第二張卡。
5. **640 容量按卡型分開核准**：4090 暫定 2 sessions/card；RTX 6000 Ada 離線 compile 已達 3 路，但正式對外先鎖 2 路，真人 WebRTC 60 分鐘 soak 通過後才升 3。
6. **容量是 profile，不是常數**：必須按 `provider + region + gpu_model + model_commit + resolution + compile_mode` 載入核准 slots；任何一項變更都要重驗。

## 1. 給經營者的一頁結論

### 技術是否能服務 1–30 人？

**能，但不能一次宣稱 30 人已完成。** 現在已經有單卡多槽、Gateway、排隊與 GLOWS 開卡的測試骨架；尚缺正式持久狀態、Voice＋Avatar 原子配位、健康輪詢、點數預授權、App 接線、真 GLOWS 0→1→0 演練與分階段長壓測。

### 首發應該怎麼做？

- 第一階段只承諾 **1–2 人同時聊**：一張台灣 GLOWS RTX 6000 Ada 常駐、640、先開 2 slots。
- 第二階段擴到 **4–10 人**：多台單 GPU GLOWS Worker＋正式 autoscale／drain。
- 第三階段才擴到 **11–30 人**：GLOWS 主池＋一個已通過 WebRTC 長連線 PoC 的 burst provider。
- 30 人若全部使用 4090／640，最低 15 張卡；若要容忍一張故障，尖峰需準備 16 張可用容量。

### 最重要的產品規則

**先拿到 Voice 席位、Avatar 席位與點數授權，兩邊都真正 ready，才顯示撥通並開始扣點。** 任一邊缺席就排隊；準備失敗不扣點；通話中任一核心服務無法恢復就停止整通，不退成另一種服務。

## 3. 正式架構

```text
Munea App
  │ Supabase JWT + call idempotency key
  ▼
munea-call-control（Cloud Run CPU，無媒體流）
  ├─ 權限／方案／點數預授權
  ├─ 原子預留 Voice shard + Avatar slot
  ├─ Queue／Lease／Heartbeat／回收
  ├─ Worker health／drain／autoscale decision
  └─ 短效 session token／TURN credential
        │
        ├──────────────┐
        ▼              ▼
Gemini Voice Bridge   VocaFrame 640 GPU Worker
Cloud Run             GLOWS 4090 主池
        │              │
        └── App 直連媒體面 ──┘

Supabase Postgres（控制面唯一真相）
  capacity_profiles / voice_shards / gpu_workers
  call_queue / call_leases / call_usage_events
  provider_operations
```

### 為什麼控制面與媒體面分開

- Gateway 只做配位、狀態、計費與調度，不轉送音訊／影像，CPU 成本低且容易擴副本。
- App 拿到配位後直接連 Voice 與 Avatar，避免媒體多繞一層增加延遲。
- 所有服務都以同一個 `call_id` 與短效 token 對帳，不能只靠 App 端記憶體。

### 為什麼 1–30 人先用 Supabase Postgres，不先加 Redis

30 人不是資料庫吞吐量問題，真正難點是正確性。先用 Postgres transaction、row lock、唯一索引與 `FOR UPDATE SKIP LOCKED` 做原子配位，能沿用現有 Supabase、RLS 與稽核。等同時通話超過 100、或 queue 推播需要更低延遲時，再加 Redis 作快取／通知；Redis 不作最終帳本。

## 4. 控制面資料模型

### `capacity_profiles`

- `profile_id`
- `provider`、`region`、`gpu_model`
- `model_name`、`model_commit`、`resolution`
- `compile_mode`、`safe_slots`
- `p50_ms`、`p95_ms`、`headroom_pct`
- `approved`、`approved_at`、`evidence_url`

首發常駐 profile：`glows / Taiwan / RTX6000Ada / VocaFrame-640 / compile / safe_slots=2`；完成真人三路 soak 後才可把同一版 profile 核准為 3。後續 4090 scale-out 必須使用獨立 profile `safe_slots=2`，不可共用 6000 Ada 容量。512 profile 必須 `approved=false`；768 只保留 benchmark，不進 production selector。

### `gpu_workers`

- `worker_id`、`provider_instance_id`、`profile_id`
- `url`、`region`、`status`（starting／warming／ready／draining／unhealthy／terminated）
- `capacity`、`active_leases`
- `last_heartbeat_at`、`ready_at`、`idle_since`
- `hourly_cost`、`provider_operation_id`

### `voice_shards`

- 代表一個真正獨立配額邊界的 Google project／Live API 資源池。
- `limit` 必須來自實際專案 quota 與 soak 結果；**同專案增加 API key 不增加容量**。
- 記錄 `active_leases`、429、GoAway、connect p95 與健康狀態。

### `call_leases`

- `call_id`、`account_id`、`person_id`、`character_id`
- `state`：queued → reserved → connecting → active → ending → ended／failed
- `voice_shard_id`、`worker_id`、`slot_id`
- `lease_expires_at`、`last_heartbeat_at`
- `voice_ready_at`、`avatar_ready_at`、`active_at`、`ended_at`
- `billable_seconds`、`end_reason`、`idempotency_key`

### `provider_operations`

每次開卡、暖機、註冊、drain、刪卡都有唯一 operation id、預算、重試次數與最終狀態，避免 controller 重啟後重複開兩張卡。

## 5. 聯合准入與通話生命週期

1. App 送 `POST /v1/calls`：帶 Supabase JWT、角色與 idempotency key。
2. Call Control 在同一個 transaction 內檢查：帳號／方案、至少可支付下一計費單位、Voice shard 空位、Avatar 640 slot 空位。
3. 四項都成功才建立 `reserved` lease；缺任何一項就進 queue，不建立半套通話。
4. 回傳短效 `call_token`、Voice URL、Avatar URL、worker／slot 綁定資訊；不得回傳長效 provider key。
5. Voice Bridge 與 Avatar Worker 各自向 Call Control 回報 ready。
6. 兩個 ready 都收到後才將 lease 改為 `active`、開始計費，App 才切成「已接通」。
7. App／Voice／Avatar 每 15 秒 heartbeat；45 秒沒心跳自動回收 lease。
8. 結束時以 call id 原子釋放 Voice＋Avatar、結算使用秒數，再推進 queue。

### 禁止的 fallback

- Avatar 滿載 → **排隊**，不可純語音。
- Voice 失敗 → **整通不成立／結束**，不可只播 Avatar。
- 點數不足 → 不建立 GPU／Voice session，避免先燒成本再拒絕。
- Worker 失聯 → 暫停整體輸出並嘗試 10 秒內重連；失敗就收線、停止計費並提供重新撥打，不在通話中偷偷換成 2D。

## 6. 640 容量與 GLOWS 配置

| 同時通話 | 4090／640 最低卡數 | 建議可用卡數 | 放量條件 |
|---:|---:|---:|---|
| 1–2 | 1 | 1 | 兩角色同線 60 分鐘通過 |
| 3–4 | 2 | 2–3 | 真 GLOWS scale 0→1→0＋故障摘除 |
| 5–10 | 3–5 | 4–6 | 10 人 30 分鐘 soak、queue／billing 對帳 |
| 11–20 | 6–10 | 7–11 | burst provider 通過、Google quota 實證 |
| 21–30 | 11–15 | 12–16 | 30 人 60 分鐘 soak＋單卡故障演練 |

### Unit Qty 決策

- `Unit Qty=1`：一個 instance 一張 GPU；現階段唯一允許模式。
- `Unit Qty=2`：同一 instance 兩張 GPU、基本規格與價格相乘；必須另外啟動 `CUDA_VISIBLE_DEVICES=0/1` 兩個獨立 Worker 才會用到。
- Munea 優先多開單 GPU instances：故障隔離、逐台 drain、逐台省錢、Gateway 容量更清楚。

### 首發常駐卡標準設定

| GLOWS 欄位 | 首發值 | 原因 |
|---|---:|---|
| GPU | RTX 6000 Ada 48GB × 1 | 640 compile 實測可達 3 路，48GB 也保留後續模型優化空間 |
| Region | 台灣、優先已驗證同區 | 降低 RTT；跨區或換主機型號後仍須重跑容量基準 |
| Image | Ubuntu 24.04 Docker NV580 | 與 6000 Ada 實測環境一致，避免 CUDA／driver 漂移 |
| Unit Qty | `1` | 一個 instance 一張 GPU；不要付第二張未被程式使用的卡 |
| Extra vCPU | `0` | 基本 12 vCPU 已足夠目前推論服務 |
| Extra RAM | `0 GB` | 基本 80GB RAM 已足夠 |
| Extra Storage | `0 GB` | 基本 100GB 可容納目前 runtime、權重與 compile cache |
| Datadrive | 首發 runtime 不掛 | 正式服務採 private artifact＋snapshot 重建；不要把唯一模型副本留在 instance |
| Bind Public IP | 不綁 | 走 GLOWS HTTPS endpoint／Gateway；只有固定 IP allowlist 的明確需求才購買 |
| Service port | HTTP `8888`；SSH `22` 僅維運 | 與 `glowsctl.py` 及現役 FlashHead 啟動方式一致 |
| Frame size | `640×640` | 512 禁用，768 不進首發 selector |
| Production slots | 先 `2`，驗收後才升 `3` | 離線 3 路已過 20% 餘裕，但仍欠真人 WebRTC 60 分鐘 soak |
| Compute mode | `compile`，先 warm 完再 ready | 6000 Ada 三路實測必須用 compile；冷編譯約 106 秒、有 cache 約 30 秒 |

建好並完成安裝後要立即建立一份 production snapshot；instance 意外釋放時由 snapshot 重建，不能靠人工重裝。模型權重與設定另存私有物件儲存，snapshot 只是可快速啟動的部署產物，不是唯一備份。

常駐 production card 上線後只跑推論。模型訓練、權重轉換、容量 benchmark 與破壞性實驗必須使用另一張 staging／on-demand card；若首發階段只有一張卡，則只能在公告維護時段先 drain 全部通話再操作。

### RTX 6000 Ada 與 4090 的使用界線

- RTX 6000 Ada：0.720 Credit/hr，640 compile 離線實測 3 路 p95 735ms、約 23% 餘裕；滿載每人每分鐘成本與 4090 雙路幾乎相同，適合當首發常駐主卡，但第 3 路仍須通過真人 soak 才開放。
- RTX 4090：0.490 Credit/hr，640 暫定安全 2 路；每月 idle 成本較低，適合流量尚未超過 2 人時的成本優先方案，以及之後逐台 scale-out。
- 若只看最低固定成本，選 4090；若要一張常駐卡同時保留第 3 路容量與 48GB 優化餘裕，選 6000 Ada。本案依 Edward 2026-07-13 指示，首發常駐卡採 **RTX 6000 Ada × 1**，但未通過第 3 路驗收前 Gateway 只宣告 2 slots。

## 7. GPU 供應商策略

### 主池：GLOWS 台灣 6000 Ada／4090

- 台灣延遲最低；6000 Ada 作首發常駐卡，4090 作後續成本優先 scale-out 候選。
- 缺點是沒有真正 pause、刪除重建約 2–3 分鐘，因此它是**預測型暖池**，不是看到排隊才臨時救火。
- 若 Munea 對外承諾 24 小時可聊，至少一張卡應 24/7 warm。6000 Ada 按 0.720 Credit/hr、1 Credit=NT$32，約 NT$16,589/月；4090 按 0.490 Credit/hr，約 NT$11,290/月。常駐卡不可被 autoscaler 縮到 0。

### Burst PoC 優先序

1. **Cloud Run GPU RTX PRO 6000 Blackwell／新加坡**：與現有 GCP IAM、Cloud Monitoring、部署最一致；平台 GPU instance 約 5 秒可啟動，但模型載入／compile 仍可能需要 30–110 秒，必須實測 640 真正 ready 時間、slots、台灣 RTT 與 CUDA 13 相容性。
2. **RunPod Serverless Load Balancing endpoint**：可直接跑自訂 HTTP service並自動增減 Worker；只驗 load-balancing 型，不拿 queue job 型硬套 WebRTC。必測 WebSocket、WebRTC、session 固定、縮容不中斷與亞洲機房庫存。
3. **Modal Server**：官方支援 WebSocket、sticky `Modal-Session-ID`、target concurrency、buffer containers 與動態 autoscaler；現有整合經驗可沿用。但最近 compute routing 與 GPU 成本仍需 640 實測，未通過前只列候選。

只保留一個 burst 勝者，避免同時維護三套正式供應商。

## 8. 自動擴縮規則

每 15 秒由 `munea-capacity-controller` 計算：

```text
需求 slots = active + connecting + queue + forecast_p90_next_5m + failure_reserve(2)
目標 cards = ceil(需求 slots / approved_safe_slots_per_card)
```

### Scale up

- `queue_depth > 0`，或未來 5 分鐘預測需求大於 ready＋starting slots。
- GLOWS 開機慢，使用 App 進入聊聊頁的 warm intent、預約活動與時段 P90 提前 3–5 分鐘開卡。
- 每次 provider create 使用唯一 operation id；controller 重啟不重複開卡。
- 新卡必須完成模型載入、compile、640 輸出檢查與 `/health ready` 才能註冊。

### Scale down

- Worker `active_leases=0`、無 reserved／connecting lease、閒置至少 15 分鐘。
- 先標 `draining`，Gateway 不再派新通話；再次核對 worker 真實 session=0 才刪卡。
- 一次只退一張，退卡後 cooldown 5 分鐘；永遠保留服務承諾所需最低暖池。
- 日預算 70% 黃燈、90% 停止非必要 buffer、100% 禁止額外開卡並通知營運；不可刪除正在通話的卡。

### AI 可以做什麼

- 上線初期：規則＋時段 EWMA／P90，所有決策可解釋。
- 累積 2–4 週真流量後：AI 預測 5／15／30 分鐘需求，提供建議 desired cards。
- LLM 不直接刪卡、不能繞過 active lease、最低暖池與預算硬閘。

## 9. Voice 長通話與配額

- Live API audio-only session 若不開 context compression 有時間限制；連線本身約 10 分鐘會重置。
- 正式 Voice Bridge 必須加入 `context_window_compression`、`session_resumption`、GoAway 預接續、429 指數退避與 resume 成功率監控。
- Cloud Run WebSocket timeout 設 60 分鐘仍需 App 自動重連；session affinity 只是 best effort，狀態不可只放單一 instance 記憶體。
- 現有 `max_instances=5 × concurrency=40` 只是 Cloud Run 容器入口上限，不是 Gemini 200 人承諾。
- 30 人 60 分鐘 soak 前，必須取得實際 Google project quota，並以專案為 voice shard；同專案多 key 不可重複算容量。

## 10. 點數與計費

現有 `/avatar-session` 由 client 回報 duration，且可 fallback 2D，不可作正式 Voice＋Avatar 計費。

正式流程：

1. 配位前預授權至少一個計費單位，避免拿到 GPU 後才發現餘額不足。
2. Voice＋Avatar 都 ready、lease 進 `active` 才開始計時。
3. 每 30 秒由 server 寫入 usage heartbeat；client 只提供輔助事件，不能決定金額。
4. 結束以 server 的 `active_at`／`ended_at` 結算，grant／consume／release／refund 都有 idempotency key。
5. 準備失敗零扣點；平台故障造成中斷，停止計費並按產品政策回補未完整計費單位。

點數顯示可維持「一點對應一分鐘」的產品語言，底層仍保存精確 billable seconds，方便日後調整四捨五入政策而不重做帳本。

## 11. 安全與可靠性 P0

- App 使用 Supabase JWT；不得用寫在前端的共用 key 當正式身分驗證。
- Call Control 發 60 秒內有效、綁 `call_id + account_id + worker_id + slot_id` 的短效 token。
- TURN 改短效 credential；provider API key、Gateway admin key只放 Secret Manager。
- Worker admin／health callback 走服務身分或簽章，不對 App 開放。
- 每個 session 的音訊 queue、Avatar feeder、WebRTC peer與角色都必須物理隔離；跨使用者聲音／畫面容忍度為 0。

## 12. 監控與 SLO

| 面向 | 首發門檻 |
|---|---|
| 暖池撥通 | Voice＋Avatar ready p95 ≤ 8 秒 |
| 開場 | ready 後延遲 1–2 秒再招呼；首段完整、不變聲 |
| 聲畫同步 | A/V offset p95 ≤ 200ms |
| 語句完整 | 句尾截斷率 < 1%，下一句殘音率 < 0.5% |
| 容量 | compute p95 保留 ≥20% 餘裕；低於 20% 不算 safe slot |
| 穩定 | 30 分鐘斷線率 <1%；Voice resume 成功率 ≥99% |
| 排隊 | 2 秒內顯示位置／ETA；不得無提示等待 |
| 計費 | 準備失敗零扣點；重複 callback 不重複扣點 |
| 隔離 | 跨 session 聲音／影像／角色混線 = 0 |

告警進 Slack 與後台，不寄 Smoke 失敗信給 Edward。

## 13. 分階段施工與放量門

### P0-A：控制面地基（不碰 App 包版）

- 新增 capacity／worker／lease／queue／usage／provider operation migrations。
- Gateway v2 改為 Supabase 原子租約、TTL、idempotency、Voice＋Avatar 真正聯合扣位。
- Health watcher、lease reaper、drain 與 server-side meter。
- 建立 512 禁用、640 4090=2 的 capacity profile。

### P0-B：Voice 與 Worker 契約

- Voice 加 context compression、session resumption、GoAway／429 指標。
- Worker 驗短效 call token、回報 component ready／ended、health profile。
- 640 兩角色同線 60 分鐘 soak；未通過前 production slots 仍設 1。

### P0-C：App 接線（會影響包版，開工前另行公告）

- 移除硬編 GLOWS URL與所有純語音／2D fallback。
- 接 `/v1/calls`、queue poll、短效 token、聯合 ready 與整通重連／收線。
- 點數從 active 才開始；顯示排隊位置與 ETA。

### P1：4–10 人

- GLOWS SDK 真跑 create → warm → register → drain → delete。
- 兩張以上 Worker 故障演練；10 人 30 分鐘 soak。
- 上線規則 autoscale 與成本告警。

### P2：11–30 人

- Cloud Run GPU／RunPod LB／Modal 三選一 PoC，只留勝者作 burst。
- Google quota 實證與 30 人 Voice soak。
- 30 人 60 分鐘整鏈測試、單 provider／單 worker 故障演練、計費對帳。

## 14. 正式上線判定

目前可判定：**方向成立、測試骨架存在，但高併發尚未上線。**

第一個可對外承諾的里程碑不是「30 人」，而是：

> 一張 GLOWS 4090、VocaFrame 640、兩名真實使用者同時連聊 60 分鐘；Voice＋Avatar 全程綁定、無混線、無句尾截斷、聲畫同步過門檻、點數準確、任一人掛斷不影響另一人。

這項通過後才把 production ceiling 從 1 提到 2，之後依 4、10、30 的放量門逐級開放。
