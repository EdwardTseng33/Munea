# 沐寧 1-30 人即時語音＋Avatar 容量與自動調度決策

> 日期：2026-07-13
> 範圍：Gemini Live 語音必須與即時 Avatar 綁定，不允許滿載時退成純語音。
> 結論基準：以現役 FlashHead 640、GLOWS 台灣 RTX 4090 實測「每卡 2 人安全、3 人不安全」為準；舊文件的每卡 3-4 人理論值作廢，除非新版本重新實測通過。

## 一、執行結論

1. **1-30 人技術上走得通，但目前還不是可直接放量的完整服務。** Gemini 語音橋已做過 30 人連線成功測試；顯卡端已有雙槽 worker、Gateway／排隊骨架與 GLOWS／RunPod 控制腳本，但跨供應商自動開卡、健康註冊、故障轉移和 30 人長時間壓測尚未串成閉環。
2. **容量硬數字：640 畫質每張 4090 安全 2 路。** 30 人同時需要 15 張卡；若要容忍一張卡故障，尖峰配置需 16 張。切 768 前先重測，沒有數據前一律按每卡 1 路估算，不能拿 640 的雙槽結論硬套。
3. **GLOWS 適合低延遲主力暖池，不適合單獨承擔突發擴容。** 它沒有 pause，但公開 SDK 已能 create/query/delete，也能從 snapshot 建機；因此可以自動化，只是 2-3 分鐘重建延遲決定了它必須靠預測提前開，而不是排隊後才開。
4. **推薦架構：GLOWS 暖池＋Modal 或 RunPod Serverless 溢流＋RunPod Pod 災備。** GLOWS 保台灣低延遲；Modal／Serverless 提供平台級自動擴縮；既有美國 RunPod Pod 保留為故障備援，不把「停止後不一定有原卡可喚醒」當主要擴容機制。
5. **不要一開始用 LLM 自己決定開關卡。** 先用可稽核規則＋時間序列預測；AI 只做需求預測、異常摘要和建議。任何刪卡都要有「無活躍 session、閒置滿 15 分鐘、保留最低暖池、成本上限」四道硬閘。

## 三、Gemini 語音線判斷

### 3.1 為什麼 1-30 人可行

- Repo 已實測修掉 async 阻塞後，10／15／30 人 session 建立成功率均為 100%；30 人首聲 p50 約 4.2 秒，表示「容量能進」但體感仍需改善。
- Google 現行限額按 **project** 計算，不是多放 API key 就能乘開；公開文件不再保證每個模型固定表格，必須以 AI Studio 顯示的專案 active limits 為準。
- Tier 1 有 10 分鐘 US$10 的花費保護上限；超出會回 429。30 人若長聊，除了 concurrent session，還必須監控 spend-rate、audio token 與上下文重算成本。

### 3.2 上線前必補

1. 在 AI Studio 匯出目前 Live 模型的 concurrent session／RPM／TPM 限額，設定 70%／85%／95% 告警。
2. 開啟 context window compression；否則音訊 context 持續累積，長通話成本與限制都會惡化。
3. 實作 session resumption、GoAway 預接續、429 指數退避；Google 明確提醒 Live WebSocket 會週期性重置。
4. 做 30 人 × 60 分鐘 soak，不只測「接得上」，還要驗首聲、尾字、斷線、重接與每分鐘成本。

官方依據：
- Google Live best practices：https://ai.google.dev/gemini-api/docs/live-api/best-practices
- Google rate limits：https://ai.google.dev/gemini-api/docs/rate-limits
- Google pricing：https://ai.google.dev/gemini-api/docs/pricing

## 四、顯卡供應商決策

### 4.1 GLOWS：主力暖池

**優點**：台灣延遲最低、4090 實測效能最好、成本低；SDK 已提供建立、查詢、刪除與 snapshot 流程。

**限制**：沒有 pause；刪除後網址會變；實際重建約 2-3 分鐘；庫存不保證。因此不能等 queue 出現才開卡。

**正確做法**：
- 將已驗收 snapshot 當 immutable image。
- controller 以 `custom_meta_key` 冪等建立新機，輪詢到 Running → `/health` 暖機驗收 → 自動註冊 Gateway → 才接流量。
- 縮容先 Gateway drain，確認 active=0，再 snapshot（版本變更時才需要）與 delete。
- TW-03 無庫存自動試 TW-04，再切 overflow provider。

官方 SDK：https://sdkdoc.glows.ai/

### 4.2 RunPod：Pod 災備、Serverless 候選溢流

- Pod REST API 能建立與 start；停止後只付儲存，但既有實測有「原機沒空卡、喚醒失敗」風險，不能承諾秒醒。
- Serverless 可設 active workers、max workers、idle timeout、FlashBoot，並按 queue delay 或 request count 自動擴縮；平台能力比 Pod 更貼近溢流需求。
- 但 Munea 是 WebRTC／WebSocket 長連線，不是一般短任務。必須先做 2 人、10 分鐘的 Serverless 長連線 PoC，驗證 session affinity、連線上限、worker 收縮時不斷話，再決定能否正式使用。

官方依據：
- RunPod Serverless endpoint settings：https://docs.runpod.io/serverless/endpoints/endpoint-configurations
- RunPod Pod create API：https://docs.runpod.io/api-reference/pods/POST/pods
- RunPod Pod start API：https://docs.runpod.io/api-reference/pods/POST/pods/podId/start

### 4.3 Modal：最省自建控制器的溢流方案

- Modal Function／Server 可自動 scale-to-zero，設定 min／max／buffer containers、target concurrency 和 scaledown window，也能在執行中動態調整 warm pool。
- Repo 已跑過 Modal WebRTC，整合風險低於全新平台；但現役 L4 對 FlashHead 太慢，需以 L40S 或其他相容快卡重新 benchmark。
- 若新卡 2 路品質通過，Modal 是首選 overflow；若成本過高，只保留 0-2 張 burst buffer。

官方依據：
- Modal autoscaling：https://modal.com/docs/guide/scale
- Modal Server autoscaling：https://modal.com/docs/guide/servers
- Modal cold start：https://modal.com/docs/guide/cold-start

### 4.4 Google Cloud Run GPU：值得做一輪 PoC 的替代方案

- Cloud Run GPU 原生自動擴縮到 0，基礎 GPU instance 約 5 秒可啟動，現有 Cloud Run 監控／IAM／部署流程也最一致。
- 目前 GPU 地區以新加坡等地為主，不在台灣；而且 Cloud Run 依 CPU／request concurrency 擴縮，不看 GPU utilization，必須由我們設定正確 concurrency；預設專案 GPU quota 通常只有 3 張，30 人需提前申請至少 16 張。
- L4 已知不夠快；可評估新加坡 RTX PRO 6000，但 CUDA 13／模型相容性、延遲與價格都要實測。它是中期候選，不是本週直接替換 GLOWS。

官方依據：
- Cloud Run GPU：https://cloud.google.com/run/docs/configuring/services/gpu
- Cloud Run GPU best practices：https://cloud.google.com/run/docs/configuring/services/gpu-best-practices

### 4.5 Managed Avatar：只做災難備援

LiveAvatar FULL 約 2 credits／分鐘；Scale 方案額外 credit 約 US$0.10，等效約 US$0.20／分鐘，已高於 Munea NT$3／分鐘零售價。它可作全自架故障時的短期救援，不適合正常主線，且換臉／口型體驗不一致。

官方價格：https://docs.liveavatar.com/docs/faq/credits

## 五、推薦的 1-30 人架構

```text
App 開啟聊聊頁（預熱訊號）
        ↓
Call Control Plane / Gateway
  ├─ 原子預留 Gemini session + Avatar slot
  ├─ 有兩邊空位 → 才讓 App 撥通
  ├─ 缺任一邊 → 排隊，不退純語音
  └─ Provider adapters
       ├─ GLOWS warm pool（主力，2 slots/card）
       ├─ Modal / RunPod Serverless（burst）
       └─ RunPod warm Pod（災備）
```

### 容量表（640、每卡安全 2 路）

| 同時通話 | 最低卡數 | 建議卡數（含一張故障餘裕） |
|---:|---:|---:|
| 1-2 | 1 | 1-2 |
| 3-4 | 2 | 3 |
| 5-10 | 3-5 | 4-6 |
| 11-20 | 6-10 | 7-11 |
| 21-30 | 11-15 | 12-16 |

早期不應常駐 16 張。建議白天 GLOWS `min=2`（4 路），夜間 `min=1`（2 路），其餘由預測提前開或 overflow provider 接手。若 overflow 尚未通過 PoC，產品端同時在線上限應先硬鎖 4 人，而不是假裝可承接 30 人。

## 六、自動擴縮策略

### 6.1 決策公式

每 15 秒計算：

```text
需求槽位 = active_calls + queue_depth + forecast_next_3m + reserve_slots(2)
目標卡數 = ceil(需求槽位 / safe_slots_per_card)
```

- `safe_slots_per_card=2`（640）；768 未驗證前設 1。
- scale up：使用率連續 60 秒 >70%，或 queue_depth >0，或未來 3 分鐘預測超過現有容量。
- scale down：卡 active=0 且閒置 15 分鐘；一次只關一張；縮容後至少 5 分鐘 cooldown。
- 預熱：用戶打開聊聊頁就送 warm intent；預約／歷史時段預測可提前 3-5 分鐘開 GLOWS。
- 硬上限：每日 GPU 預算、每供應商 max cards、全域 16 cards；超過只能排隊並通知營運。

### 6.2 AI 的位置

1. **第一階段（上線）**：規則引擎＋每 15 分鐘時段 EWMA／P90，完全可解釋。
2. **第二階段（有 2-4 週真流量）**：quantile forecast 預測未來 3／10／30 分鐘需求，特徵含星期、時段、推播、預約、App warm intent、最近 queue slope。
3. **LLM 只做營運助手**：把異常、成本與建議整理成人話；不得直接刪有 session 的卡，也不得繞過預算上限。

## 七、監控與自動處置

| 類別 | 必收指標 | 黃燈 | 紅燈／自動動作 |
|---|---|---|---|
| 用戶 | 撥通成功率、queue p50/p95、等待放棄率 | p95 >20s | p95 >60s：加卡＋告警 |
| Gemini | active、connect success、首聲、429、1011、GoAway、resume success | 首聲 p95 >3s | 429 或成功率 <98%：停止新放行、排隊 |
| Avatar | slots、compute p95、headroom、underrun、freeze、WebRTC RTT/loss | headroom <20% | <10% 或 underrun：該卡 drain |
| GPU | utilization、power、VRAM、temperature、health age | health >20s | >45s：摘除 worker、補卡 |
| 成本 | 每通話分鐘 GPU／Gemini 成本、idle ratio、日預算、每點毛利 | 日預算 70% | 90%：禁非必要 buffer、通知營運 |
| 供應商 | 開卡時間、失敗率、庫存、區域延遲 | 開卡 >120s | 連續 2 次失敗：切下一 provider |

事件與 metrics 統一送 Cloud Monitoring／Prometheus；告警進 Slack／營運後台，不寄 Smoke 信給 Edward。每張卡必須有 provider、region、instance_id、worker_url、model_version、slots、active、draining、created_at、last_health、cost_per_hour。

## 八、執行順序與放量門檻

1. **P0：接完既有 Gateway 閉環**：健康輪詢、原子預留 voice＋avatar、App 排隊，不再退純語音。
2. **P0：GLOWS provider adapter**：snapshot create → readiness → register → drain → delete；做 0→1→0 演練。
3. **P0：2 路 60 分鐘 soak＋故障演練**：殺掉一張卡，既有通話明確結束／重接，新通話不再進故障卡。
4. **P1：Modal 快卡與 RunPod Serverless 二選一 PoC**：2 路 WebRTC 10 分鐘、冷啟、縮容不斷話、成本；只留勝者做 overflow。
5. **P1：分階段壓測 2→4→8→16→30**：每階段至少 15 分鐘，30 人最後跑 60 分鐘。
6. **P1：AI Studio quota＋30 人 Gemini soak**：拿到實際 quota 才設定正式 admission ceiling。
7. **P2：需求預測與成本最佳化**：累積真流量後才啟用 forecast，不用 LLM 猜卡數。

### 正式承諾門檻

- 30 人 60 分鐘：撥通成功率 ≥99%、首聲 p95 ≤3 秒、尾字截斷 <1%、視訊 freeze ratio <1%、queue p95 ≤20 秒。
- 任一 provider 掛掉：30 秒內停止派新流量；容量恢復或切備援後自動解除。
- 每卡 2 路只在 640 生效；任何模型／解析度／CUDA／GPU 變更都必須重跑容量驗收。

## 九、最終選擇

**短期上線**：GLOWS 1-2 張暖卡，先把可承諾容量鎖在 2-4 人；完成 overflow PoC 與 Gateway 閉環後再逐步放到 10、20、30。

**中期正式架構**：GLOWS 台灣低延遲主池＋Modal（若快卡 PoC 勝出）自動溢流＋RunPod Pod 災備。若 Cloud Run RTX PRO 6000 新加坡實測在效能、延遲、價格三項都優於 Modal，才把 Modal 替換成 Cloud Run GPU。

**不建議**：只靠 GLOWS 刪除重建做即時擴容、把 RunPod stopped Pod 當保證可喚醒、未驗證就宣稱 768 每卡 2 人、滿載退純語音、讓 LLM 無硬閘直接開關 GPU。
