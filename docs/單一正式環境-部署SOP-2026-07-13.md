# 沐寧 · 單一正式環境部署 SOP（2026-07-13）

> 背景：原本雲端有「正式」「測試」兩套服務，但沐寧是單人開發，App 早就只打「測試」那套（因為它才有最新的修復）。
> 2026-07-13 Edward 拍板收斂成一套：不再分正式/測試，只有一套「唯一正式」，靠 Cloud Run 內建的 canary（新版先不給流量）當安全閘。

## 一、人話版：以後怎麼安全發新版

1. **改完程式、想上線** → 跑 `canary-deploy.sh`。這一步只是「生出一個新版本」，**不會**讓任何人現在用的東西改變，新版先晾在旁邊。
2. 跑完會印一個**測試網址**（帶 `canary-日期-時間` 字樣）。拿這個網址自己測，或丟給 Edward / 主蘇菲測。
3. **測試 OK** → 跑 `promote.sh`，把「正式」流量切到新版，這時所有人才會看到新版。
4. **測試不 OK** → 什麼都不用做。沒 promote 就沒切流量，正式版完全沒被動到，改完再跑一次 canary 就好。
5. 真的切錯版、想退回上一版 → `promote.sh` 執行後會印回滾指令，照著貼就行（Cloud Run 每版都留著，秒切回去）。

## 二、現在的服務是誰（2026-07-13 定案）

| 服務名 | 角色 | App 打的網址 |
|---|---|---|
| `munea-brain-staging` | 唯一正式·管家腦（聊天/記憶/簡報/守護） | `web/src/app.js` 的 `BRAIN_URL_DEFAULT` |
| `munea-voice-staging` | 唯一正式·語音橋（即時通話） | `web/src/app.js` 的 `LIVE_VOICE_URL_DEFAULT` |

名字仍帶 `-staging` 字尾——那只是外觀沿用（改名要整個重建服務，非上線必要，之後有空再處理），**不代表它是「測試」身分**，它就是唯一正式。

原本無 `-staging` 字尾的舊「正式」（`munea-brain` / `munea-voice`）已於 2026-07-13 確認停更 5+ 天、近 14 天無真實流量、無排程/其他服務依賴後，**由 Edward 拍板刪除**。

**兩服務共用同一個 Supabase 資料庫**（`SUPABASE_URL=https://uhmpmystjjdqqxlpsthc.supabase.co`）——這是既有設計，不是分開的两套資料，删除/部署 Cloud Run 服務不會動到這個資料庫。

## 三、部署指令

```bash
# 1) canary：新版只生不吃流量，印測試網址
bash deploy/cloudrun/canary-deploy.sh brain   # 管家腦
bash deploy/cloudrun/canary-deploy.sh voice   # 語音橋

# 2) 測過 OK，才切 100% 正式流量
bash deploy/cloudrun/promote.sh brain
bash deploy/cloudrun/promote.sh voice
```

兩支腳本都在 `deploy/cloudrun/`，取代舊的 `更新正式環境.sh` / `更新測試環境.sh`（那兩支已改成退役提示、執行會直接擋下並指路過來這裡）。

## 四、⚠ env-drop 地雷（一定要知道）

2026-07-12 曾經因為部署指令漏帶完整 env/secrets，把 staging 服務的環境變數從 15 個洗到剩 3 個，登入壞、Supabase 斷線。原因：`--set-env-vars` / `--set-secrets` 是「先清空全部、再設你給的值」——只帶一兩個值，其餘全部消失。

新版 `canary-deploy.sh` 已改用 **`--update-env-vars` / `--update-secrets`**（合併語意：只更新你指定的 key，其他 key 原封不動保留），從架構上排除這個地雷，不必再靠人工記得帶全部清單。

如果之後要新增/修改 env 變數，**改 `canary-deploy.sh` 裡對應服務的那行 `--update-env-vars`**，不要手動在外面另外下 `gcloud run deploy` 指令（那樣繞過了這支腳本的安全設計）。

## 五、canary 部署細節

- 打包來源固定用 `git archive HEAD`（**committed 程式碼**），不是工作目錄現況（`--source .`）。
  原因：工作目錄裡常有別的 agent／人正在改、還沒 commit 的檔案（例：這次沐寧 App 前端在改的 `web/app.js`），
  直接打包會把半成品一起送上雲，混淆這次到底在測什麼。**要部署的東西一定要先 commit**。
- 新版用 `--no-traffic --tag canary-<日期-時間>` 部署，Cloud Run 會給一個帶 tag 的專屬網址（格式 `https://<tag>---<服務原網域>`），
  這個網址 100% 打得到新版，但**完全不影響**現在正式流量吃的還是舊版。
- `promote.sh` 執行 `gcloud run services update-traffic <svc> --to-latest`，把 100% 流量切到最新版；執行前會秀目前流量分布、要求輸入 `yes` 確認。
- `promote.sh` 切完流量後自動跑 `tools/door-sentinel.py`（跟 App 一樣的「不帶憑證匿名連線」戳門，抓「切完流量後大門被鎖」這類問題，不是只驗服務活著）。

這不是這次收斂範圍內能安全處理的事（牽動要不要開一個獨立的開發用 Supabase 專案/schema，是架構決策，不是部署腳本能單方面改的），
