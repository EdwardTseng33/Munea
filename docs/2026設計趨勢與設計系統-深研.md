# 2026 設計趨勢與設計系統深研 — Munea 沐寧

> 2026-07-03 · 深度網路調研 · 對象：全家共用 AI 健康照護 App（中高齡核心）
> 對應痛點：「不同的綠、不同的按鈕形式並存」「體驗要舒適完善、設計要有巧思、交互要有趣」
> 病因先講：`styles.css` 把 teal / teal-d / teal-dd 三個綠 + mint 兩階全部直接當按鈕、文字、連結色混用，字級出現 12.5 / 14 / 14.5px、間距出現 13 / 26px 這類自由值——**缺的不是品味，是 semantic token 那一層紀律**。

---

## 一、2026 趨勢權威整理：該追什麼、不追什麼

**Apple / Liquid Glass 之後的態度**：iOS 26 的 Liquid Glass 被 NN/g 直接判定「可用性受害」——半透明疊層讓文字貼在雜亂背景上、對比不足、控件縮小擁擠、「為動而動的動效是干擾加暈眩」。無障礙社群（Infinum 等）同樣點名它漏接對比與感官負荷標準。**結論：第三方 App 沒有義務跟進玻璃擬態；值得學的只有它背後的意圖——「內容優先、控件退位」與清楚的層級感。** Munea 的全平面、無漸層路線在 2026 反而站在批評者（NN/g）這一邊，是對的，守住。

**Material 3 Expressive**：Google 2025-2026 主推方向。可借的三件事：① 用「形狀與尺寸對比」而非顏色數量做強調——重要按鈕做大、做滿，其餘降階；② 一個版面只允許一顆高強調按鈕，其餘刻意壓低；③ 按下時的形狀回饋（微縮放）讓操作「有手感」。不可借：誇張的 shape-morph 彈跳與超大 hero 按鈕群——對中高齡是干擾不是趣味。

**跨媒體共識趨勢（Muzli、Envato、多家 2026 年度整理）**：Calm UI（安靜介面、去視覺雜技）、動效從裝飾轉為「結構」（告訴用戶剛發生什麼）、bottom sheet 成為次要內容標準容器、主行動從 FAB 移入底部欄、AI 個人化要透明可解釋、無障礙從合規項變成賣點。

**給 Munea 的取捨表**：

| 該追（契合暖照護） | 不該追 |
|---|---|
| Calm UI、大字報式排版、留白 | Liquid Glass / 玻璃擬態疊層 |
| 結構性動效（狀態變化才動） | 為動而動的炫技動畫、視差 |
| bottom sheet 承載次要操作 | 深層手勢依賴（長輩記不住） |
| 無障礙優先當差異化賣點 | 暗色優先（dark-first）、霓虹高彩 |
| AI 建議附「為什麼」的透明說明 | 介面隨使用行為自動重排（長輩要的是穩定不變） |

## 二、按鈕與色彩紀律：一線 App 怎麼管

**三階按鈕是業界標準答案**（Carbon、Material、Red Hat、Base 全部同構）：

- **Primary（主）**：實心填色，一個畫面**最多一顆**。Carbon 明文：「每頁只能有一顆 Primary，其餘一律降階」；Material：「單一高強調按鈕讓其他按鈕的次要性不言自明」。
- **Secondary（次）**：淡色底（tonal）或描邊，承接「也可以做」的動作。
- **Tertiary / Quiet（安靜）**：純文字或幽靈按鈕，放取消、查看更多這類低風險動作。工具列、清單裡的多動作場景一律用這階，不准擺一排實心鈕。

**常見 token 值**（HIG / M3 / Carbon 交集）：高度 44pt（HIG 底線）→ 48dp（M3、Carbon 標準）→ 56px（大按鈕 / 高齡取向）；圓角要嘛統一 8-16px 的方圓、要嘛統一全圓 pill，**禁止兩種圓角哲學並存**；字級 16-18px、字重 600-700；同階按鈕在全 App 任何畫面必須像素級相同。

**「同品牌多個綠」怎麼被規範掉**：一線做法是把色票拆成兩層——**primitive tokens**（`green-500`、`green-700`…純色值，設計師以外不可見）與 **semantic tokens**（`color-action`、`color-success`、`color-decorative`…按用途命名）。規則只有一條：**產品程式碼只准引用 semantic 層，且「可互動」的語意色只有一個**（Carbon 的 `interactive-01` 模式）。多個綠可以存在於 primitive 層（做深淺階），但「哪個綠代表可以按」全 App 只有一個答案。落地手段：Figma styles 鎖定 + code review 禁 raw hex + stylelint 掃描。

**Munea 現況實測（本次用 WCAG 公式實算）**：

| 組合 | 對比 | 判定 |
|---|---|---|
| 白字 on 亮綠 #3AA8A0 | 2.88:1 | ❌ 連大字 3:1 都不過 |
| 白字 on 中綠 #2E8A83 | 4.13:1 | ⚠️ 只過大字（≥19px 粗體） |
| 白字 on 深綠 #236C66 | 6.16:1 | ✅ 全尺寸過 AA |
| 亮綠 #3AA8A0 當文字 on 白 | 2.88:1 | ❌（現在的「更多」連結就是它） |
| 白字 on 磚橘 #E08B45 | 2.64:1 | ❌ |
| 白字 on 深磚橘 #C25716 | 4.50:1 | ✅ 恰好過 AA |
| muted #6B7B76 on 米白 | 4.16:1 | ⚠️ 小字不合格 |

這張表就是「不同的綠並存」的技術病因：亮綠好看但只能裝飾，能承載互動的其實只有深綠。

## 三、中高齡無障礙 2026 基準值

| 維度 | 基準 | 出處 |
|---|---|---|
| 觸控目標 | 法定底線 24×24 CSS px（WCAG 2.2 新增 2.5.8/AA）；**實務照 HIG 44pt / M3 48dp，高齡建議 48px 起跳**，相鄰目標留 ≥8px | WCAG 2.2、HIG、M3 |
| 文字對比 | 內文 4.5:1、大字（≥24px 或 ≥19px 粗體）3:1、圖示與元件邊界 3:1（1.4.11） | WCAG 2.2 AA |
| 字級 | 通用底線 16px；**高齡研究：14pt 已吃力，主內文 18-20px、關鍵數據 30pt 以上、次要文字 ≥15px**；必須支援 Dynamic Type / 系統字級放大不破版 | 2025 Springer 系統性回顧、Frontiers 高齡字級研究 |
| 動效 | 一律尊重 `prefers-reduced-motion`；單次過場 200-400ms；禁自動輪播、禁閃爍（三次/秒以上）；動效只用於回饋狀態變化 | WCAG 2.3、NN/g |
| 操作寬容 | 破壞性操作要二次確認、可撤銷；別依賴長按/雙指等隱藏手勢 | WCAG 2.2（Dragging 2.5.7）、高齡研究 |

2025 年高齡 App 系統性回顧另一結論：**與長輩共同設計（participatory design）比任何 guideline 都有效**——規範定完，要找真實長輩驗證。

## 四、「有巧思、有趣」的微互動：7 個被稱讚的案例

1. **Apple Watch 活動圓環**：合環瞬間的煙火動畫 <1.5 秒；真正的巧思是「與家人共享圓環」——看到媽媽快合環了可以送一句加油。慶祝短、社交暖，健康類微互動的祖師爺。
2. **Gentler Streak**（2024 Apple Design Award 社會影響獎）：把「連續紀錄」改成寬容版——今天狀態欄直接告訴你「該休息了」，休息日**不會斷鏈**。反 Duolingo 焦慮設計，跟照護定位完全同頻。
3. **Finch 自我照顧養鳥**：完成自我照顧任務→小鳥獲得能量出門冒險→回來講見聞。好友間只能送「good vibes」與小禮物，**沒有聊天、沒有比較、零社交壓力**——全家互動可以直接借這個模式。
4. **Waterllama**：喝水進度=可愛動物輪廓被水慢慢填滿，喝滿角色慶祝。把枯燥打卡變成「把小動物餵飽」的具象隱喻。
5. **Duolingo 的 Streak Freeze**：連續紀錄可以「凍結」補救——原諒機制讓用戶敢回來。對長輩：漏一天藥不該看到斷掉的鏈子，該看到「明天繼續就好」。
6. **Medisafe 的 Medfriend**：漏吃藥且多次提醒未回應→自動通知指定家人，家人一鍵回敬提醒。數據：有 Medfriend 的用戶 90 天留存高 59%。這是「家人互動=安全網」的最佳實證，Munea 的子女端就該這樣做。
7. **Headspace / Calm 的完成儀式**：冥想結束是一圈溫和擴散的呼吸光暈+一句「回來就好」；中斷多日的文案不是責備而是歡迎。**文案本身就是微互動。**

共通規律：慶祝要短（≤1.5s）且可跳過；寬容永遠優先於激勵；家人角色設計成「送暖」而非「監控」。

## 五、Munea 設計規範 v1 草案

### 5.1 色彩使用規則（唯一主綠制）

- **行動綠（唯一互動色）＝ `--teal-dd` #236C66**：所有主按鈕底、可點文字連結、tab 選中態、勾選完成態。白字對比 6.16:1、當文字對比 6.16:1，雙向全尺寸過 AA。
- **品牌亮綠 #3AA8A0 降級為「裝飾專用」**：logo、圖示描邊、進度條填色、插圖。**禁止**當文字色、禁止當白字按鈕底。
- **中綠 #2E8A83 只做按下態（pressed）與插圖中間調**，不再獨立出現在按鈕/文字。
- **mint 兩階（#E8F2EE / #D9EFE8）只做底色**：次要按鈕底、tag 底、卡片區塊底；上面的字只准 `--ink` 或 `--teal-dd`。
- **磚橘系＝第二強調色，每畫面最多出現 1 處**：kicker 小標、重要提醒、亮點數字。白字底只准用 #C25716（4.50:1），#E08B45 只做淡底（coral-soft）上的圖示與裝飾。
- 落地：styles.css 增加 semantic 別名（`--c-action` / `--c-action-pressed` / `--c-decor` / `--c-accent`），元件一律引用別名；**raw hex 出現在元件層＝違規**。

### 5.2 按鈕三階 token（全 App 只有這三種形式）

| 階 | 背景 | 文字 | 高度 | 圓角 | 字級/字重 |
|---|---|---|---|---|---|
| Primary | `--teal-dd` | #FFF | 56px | 18px | 18px / 700 |
| Secondary | `--mint` | `--teal-dd` | 48px | 18px | 17px / 700 |
| Quiet | 透明 | `--teal-dd` | 44px（觸控區） | — | 16px / 600 |

規則：一畫面至多 1 顆 Primary；並排時 Primary 在右（拇指側）；按下態統一 `scale(.98)` + 背景加深一階；危險動作用 `--danger` 只在確認彈層出現。**現有 .video-btn（58px/圓角 20）收斂進 Primary 規格，全站第四種按鈕形式一律刪除。**

### 5.3 間距階（8 的倍數）

主階 **8 / 16 / 24 / 32 / 48**，輔階 **4 / 12 / 20**（僅元件內部微調與頁面左右邊距 20px）。所有 margin/padding 必須取自這 8 個值——現有 13、14、26px 一律吸附到最近階（26→24、13→12、14→16）。卡片內距 16-20、卡片間距 12、區塊間距 24-32。

### 5.4 字級五階（沿用既有 v1 定案並鎖死）

**30（頁大標·serif·700）/ 20（區塊標·800）/ 17（內文與卡標·400-700）/ 15（說明·400-600）/ 13（註記·600）**。兩條補丁：① 13px 註記文字只准 `--ink-2` #4A5651（7.17:1），`--muted` 不得用於 15px 以下（4.16:1 不達標）；② 現存 12.5 / 14 / 14.5px 漂移值全部吸附到五階。長輩模式（放大檔）：整體 ×1.15，17→20、15→17，版面須預先驗證不破。

### 5.5 微互動三原則

1. **回饋即時、簡短、同一套語彙**：任何完成動作 150-400ms 內給視覺回饋（打勾彈跳、進度條推進）；全 App 只用一套緩動曲線與一顆勾選動畫元件；慶祝動畫 ≤1.5 秒、可點擊跳過。
2. **寬容優先於激勵**：連續紀錄有「休息日不斷鏈」機制（Gentler Streak 模式）；漏做的文案永遠是「明天繼續就好」，禁止紅色警告式歸零。
3. **家人互動＝送暖不監控**：預設互動是正向的（送個讚、語音貼圖、「我看到了」）；警示類通知只走 Medfriend 模式（多次未回應才通知家人，家人回敬的是關心不是查勤）。全域尊重 `prefers-reduced-motion`。
