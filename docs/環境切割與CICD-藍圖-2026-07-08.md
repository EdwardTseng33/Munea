## 1. 三層環境（比喻：廚房 → 試吃檯 → 餐廳）

| 環境 | 白話 | 引擎在哪 | 資料在哪 | 誰在用 |
|---|---|---|---|---|
| **開發（廚房）** | 兩台電腦本機 | 本機跑（現況） | 本機檔案＋測試資料 | 蘇菲/Mac 日常開發 |
| **測試（試吃檯）** | 雲端上的預演版 | Cloud Run `munea-brain-staging`／`munea-voice-staging` | Supabase **測試專案**（假資料） | 我們自己＋TestFlight 試飛員 |
| **正式（餐廳）** | 用戶真的在用的 | Cloud Run `munea-brain`／`munea-voice` | Supabase **正式專案**（真用戶資料） | App Store 用戶 |

**鐵律**：
- 測試與正式**各自一把鑰匙、各自一個資料櫃**——試吃檯打翻了，餐廳完全無感
- 正式資料絕不拿來測試；測試假資料絕不混進正式
- App 打包時就決定連哪個環境（TestFlight 版連測試、上架版連正式），靠 `munea.brainUrl`／`munea.liveVoiceUrl` 兩個設定值

## 2. 程式進版路線（誰改了什麼、怎麼到用戶手上）

```
兩台電腦改程式
   │  存檔上傳（現況照舊：main 線）
   ▼
【自動】測試環境更新：一上傳 → 自動打包 → 自動部署到「試吃檯」→ 自動跑冒煙檢查
   │  （GitHub Actions；壞了立刻知道、正式環境完全沒動）
   ▼
【手動】晉升正式：確認試吃檯沒問題 → 一個按鈕把「同一包」推上正式
   │  （永遠是推「驗過的那一包」、不重新打包，杜絕「測的跟上的不一樣」）
   ▼
出事回滾：Cloud Run 留著每個舊版本，點兩下秒回上一版
```

**自動冒煙檢查（每次測試環境更新必跑）**：
1. `npm run release:check`（現有：靜態冒煙＋登入閘門＋資料庫醫生）
2. `python engine/test_guardian_crisis.py`（危機 13 紅線）
3. 探針一問（真地名驗證、聊聊主文件 §11）

## 3. App（手機端）的節奏

手機 App 跟引擎不同步更新——引擎天天可更、App 過蘋果審核才更：
- **引擎改動**（人格、查證、簡報、修話術）：上雲後隨時更新，用戶無感升級 ← 這是把引擎搬上雲最大的紅利
- **App 改動**（畫面、功能）：版號紀律照舊（1.1.x）→ TestFlight 試飛 → 送審 → 上架
- 引擎改動**必須跟舊版 App 相容**（升級不同步是常態）

## 4. 鑰匙與安全

- 鑰匙只放 Google 保險箱（Secret Manager），程式庫永遠沒有鑰匙
- 測試/正式各一把（測試鑰匙可設額度上限、燒不到正式）
- 鑰匙第一次上雲、每次換鑰匙：過沙利曼信任關卡
- 費用警戒已設（NT$500/月三段通知）；上正式後依用戶量調高並分環境看報表

## 6. 分階段上路（跟上架節奏綁）

1. **現在**：配方已入庫、App 已會認新住址——條件到齊隨時可搬
2. **TestFlight 前**：先立「試吃檯」（測試環境）＝試飛員連的就是雲端、不再連家裡電腦
3. **送審前**：立正式環境＋正式鑰匙＋正式資料櫃，App 上架版指向正式
4. **上架後**：自動更新流程全開（上傳→試吃檯自動更新→人工晉升正式）
