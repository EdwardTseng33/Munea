> 2026-07-20 · Edward 拍板方向、蘇菲成文
> 分支：`feat/enterprise-seats`
> 一句話：**在每個帳號上加一個「這人是 A 公司付費的」記號**，用這把鑰匙同時解開「批次開通」「按公司出報告」「到期轉換」「月結請款」四件事。

## 0. 範圍與非範圍

**做**

## 1. 現況（施工前的事實）

以 2026-07-20 程式碼實查為準：

| 項目 | 現況 |
|---|---|
| 帳號結構 | `accounts` = 一個家庭，底下掛 `persons`、`account_members`。**沒有「公司」這一層** |

## 2. 資料模型（新增）

### 2.1 `enterprise_clients` — 企業客戶

| 欄位 | 說明 |
|---|---|
| `id` | 主鍵 |
| `name` | 公司名稱 |
| `tax_id` | 統一編號（請款單用） |
| `billing_address` | 帳單地址 |
| `contact_name` / `contact_email` / `contact_phone` | 窗口 |
| `plan_tier` | 授予等級：`plus` / `pro` |
| `unit_price_twd` | 每席月費（未稅） |
| `contract_start` / `contract_end` | 合約期間 |
| `seat_quota` | 合約席次上限 |
| `status` | `active` / `expiring` / `ended` |
| `report_recipients` | 月報＋請款單收件人（可多位） |
| `notes` | 備註 |

### 2.2 `enterprise_seats` — 席次（歸屬記號的本體）

| 欄位 | 說明 |
|---|---|
| `id` | 主鍵 |
| `enterprise_client_id` | 屬於哪家公司 |
| `invite_email` | 名單匯入時提供的識別（企業給的員工／家屬 email） |
| `account_id` | 綁定到哪個家庭帳號（尚未註冊時為 null） |
| `status` | `pending` / `active` / `grace` / `released` |
| `activated_at` | 綁定生效時間 |
| `grace_started_at` / `grace_until` | 緩衝期起訖 |
| `released_at` | 正式釋出時間 |
| `released_reason` | `contract_end` / `removed_by_client` / `converted_to_personal` |

**狀態流轉**
```
pending  ──該 email 註冊／比對成功──▶  active
active   ──合約到期 or 企業移除──────▶  grace（30 天，服務照舊）
grace    ──家屬轉自費──────────────▶  released（reason=converted_to_personal）
grace    ──30 天到期未轉────────────▶  released（降回免費版）
```

### 2.3 `enterprise_seat_events` — 異動紀錄

每一次狀態變更寫一筆：`seat_id`、`from_status`、`to_status`、`actor`（誰操作）、`reason`、`created_at`。

**用途**：收費爭議時的憑證。客戶問「這個月為什麼收 48 席不是 45 席」，這本帳答得出來。

### 2.4 `enterprise_invoices` — 請款單

| 欄位 | 說明 |
|---|---|
| `id` / `invoice_no` | 單號（格式 `MU-YYYYMM-<公司代碼>`） |
| `enterprise_client_id` | 哪家公司 |
| `period_start` / `period_end` | 帳單期間 |
| `billable_seats` | 計費席次數 |
| `unit_price_twd` / `subtotal_twd` / `tax_twd` / `total_twd` | 金額（稅率 5%） |
| `status` | `draft` / `issued` / `paid` / `void` |
| `due_date` | 付款期限 |
| `seat_snapshot` | 當期席次明細快照（去識別化：只存 seat_id，不存姓名） |
| `report_ref` | 對應的月報 |

### 2.5 會員資格授予的來源標記

在既有的訂閱／權益寫入路徑上，新增授予來源：

- `subscription_ledger.provider` 新增可用值 `enterprise`
- `active_plan` 照常寫 `plus` / `pro`
- 新增 `grant_ref` 欄位（指向 `enterprise_seats.id`）

**鐵律**：任何一筆非 Apple 來源的會員資格，`grant_ref` 必填。沒有來源的授予一律拒絕——這是防止「改了 Pro 但沒人記得為什麼」的唯一手段。

## 3. 後台功能（我們自己操作）

### 3.1 企業客戶列表
公司名、合約期間、席次上限、已啟用席次、狀態燈號（正常／30 天內到期／已到期）、本月預估金額。

### 3.2 名單匯入
- **範本匯出**：CSV，欄位 `email`（必填）、`備註`（選填）
- **上傳匯入**：逐列建立 `enterprise_seats`（status=`pending`）
- **匯入前預檢並顯示**：
  - 幾筆是新的
  - 幾筆該 email 已註冊 → 匯入後可直接變 `active`
  - 幾筆重複（同公司已存在）→ 跳過
  - 幾筆已屬於**其他公司** → 🔴 擋下、不自動搶，列出來讓我們人工處理
  - 是否超過 `seat_quota` → 超過要二次確認
- **匯入結果可下載**：成功／失敗清單

### 3.3 批次授予
勾選席次 → 一鍵授予公司設定的 `plan_tier`。每一筆都寫 `grant_ref` 與異動紀錄。

### 3.4 席次明細
一家公司底下所有席次的狀態、綁定時間、本月用量摘要。

**權限鐵律**：這一頁是**我們內部看的**。企業客戶永遠拿不到這一頁，只會拿到月報上的彙總數字。

### 3.5 後台畫面（`web/admin.html`）

功能不只做在底層，**要真的做進現有的營運後台畫面**。新增「企業客戶」分頁，含四個畫面：

| 畫面 | 內容 |
|---|---|
| **企業客戶列表** | 公司名、合約期間、席次上限／已啟用、狀態燈號（正常／30 天內到期／逾期未付）、本月預估金額、累計欠款 |
| **單一公司** | 基本資料編輯、席次明細表、收款紀錄表、月報與請款單下載 |
| **名單匯入** | 範本下載、上傳、**預檢結果表**（新增／已註冊／重複／屬他家公司／超額，五種分別列出）、確認匯入 |
| **收款登記** | 請款單列表、按「已寄出」、填入帳日與實收金額、**按「已入帳」＝開通開關** |

視覺與互動一律沿用 `web/admin.html` 既有的版面、配色、元件寫法，不自創風格。「已入帳」按鈕要有二次確認（按下去就會開通一批人）。

## 4. 月結（每月 1 日，結算上個月）

**一次跑完，同時產出兩份東西**，寄給同一批收件人。

### 4.1 計費規則（拍板）

- **計費席次數 = 帳單月最後一天仍為 `active` 的席次數**
- 當月新加入的席次 → **當月免費**，次月起計費（給客戶儘早加人的誘因）
- 當月中途被移除的席次 → **當月照收**（已提供服務）
- `grace` 狀態的席次 → **不計費**（合約已結束，我們自願吸收這 30 天）

理由：好算、好講、客戶不會覺得被佔便宜、我們不會虧太多。

### 4.2 請款單（自動產出）

含：公司抬頭／統編／地址、單號、帳單期間、席次數 × 單價 = 小計、營業稅 5%、總計、付款期限（**次月 15 日前**）、匯款帳戶資訊。

產出後狀態為 `draft`，**由我們人工確認後才轉 `issued`**——自動算、人工放行，避免算錯直接寄出去。

### 4.3 ESG 成效月報（自動產出）

**五個區塊**：

| 區塊 | 內容 | 資料來源 |
|---|---|---|
| 一、涵蓋與參與 | 席次數、已啟用數、實際使用數、啟用率 | `enterprise_seats` |
| 二、陪伴成效 | 平均每人每週對話次數、有意義陪伴日比例、平均通話長度、持續使用比例 | `daily_user_metrics`、`voice_session_metrics` |
| 三、照護成效 | 用藥／回診提醒完成率、關懷通知發出次數、及時接住的異常次數 | `reminder_events`、`safety_events` |
| 四、員工端價值 | 有多少比例的家庭收到過近況／安全通知 | `family_interaction_events` |
| 五、GRI 對應 | 可直接貼進永續報告書的段落與圖表 | 上述數字組成 |

### 4.4 月報的隱私鐵律（不可協商）

1. **不出現任何一位長輩的姓名、帳號、聯絡方式**
2. **不出現任何一句對話內容、任何摘要片段**
3. **任何分組人數少於 5 人時，不單獨呈現該組數字**（防反推）
4. 只有彙總數字與比例

違反任一條 = 這門生意的信任基礎破掉。實作時要有檢查，不是靠人記得。

### 4.5 產出方式
- 月報：HTML（可轉 PDF）
- 請款單：HTML（可轉 PDF）
- 兩份一起，後台一個按鈕下載或寄出

## 5. 收款與開通（2026-07-20 Edward 拍板 · 憲法級）

**核心：錢沒到，帳號不開。沒有任何自動開通的路。**

### 5.1 現金安全鐵律

1. **首月預收**。談定 → 開請款單 → 收款 → 才開通。新客戶永遠不會發生「先服務後收錢」。
2. **「已入帳」是唯一開通開關**，人工按。我們沒接任何金流，企業走 ATM／銀行匯款，靠人工刷本子對帳。
3. 名單匯入後只是**待開通名單**，未按「已入帳」前一個席次都不生效。
4. 每月同理：月結出請款單 → 收款 → 按「已入帳」→ 才續發下個月的配額。

**例外**：大客戶要求「先看到服務跑起來才付款」時，只開 **3–5 席、14 天**試辦，不整批開。每次由 Edward 個案決定，系統不自動放行。

### 5.2 收款紀錄欄位（`enterprise_invoices` 補充）

| 欄位 | 來源 |
|---|---|
| 請款單號、期間、金額 | 系統自動算 |
| `sent_at` 寄出日 | 按「已寄出」時記 |
| `due_date` 約定付款期限 | 系統算（次月 15 日） |
| `paid_at` **實際入帳日** | **人工填**（刷本子看到才填） |
| `paid_amount_twd` 實收金額 | 人工填（可能與請款金額不同） |
| `payment_note` 匯款帳號末五碼／備註 | 人工填，對帳用 |
| `invoice_number` / `invoice_issued_at` 發票號碼與開立日 | 人工填 |
| `status` | `draft` → `issued` → **`paid`** → `invoiced` → (`void`) |

逾期天數、累計欠款由系統依 `due_date` 與 `paid_at` 自動算，後台顯示。

### 5.3 欠款階梯（拍板 · 鍘刀砍公司、不砍長輩）

| 時間 | 對公司 | 對長輩 | 我們的成本 |
|---|---|---|---|
| 逾期 7 天 | 後台紅燈、**暫停新增席次**、**不出月報**、業務催收 | 照常（當月配額已發） | 已收過的量 |
| 進入下個月仍未付 | **停發新配額** | **自動回到免費版的用量** | ≈ 0 |
| 之後 | 正式催告／中止合約 | 帳號保留，隨時可自費開通 | 0 |

**關鍵**：止血靠「不發配額」，不靠「砍掉長輩」。免費版本身幾乎不耗成本（`premiumAvatarMinutesMonthly: 0`），所以長輩回到免費版時我們的支出立刻歸零。

理由：企業請款流程慢是常態（財務月結、簽核卡住），7 天就砍長輩會誤傷好客戶；而且長輩沒做錯任何事，他只會知道「寧寧不理我了」。壓力要落在能解決問題的人身上。

合約須載明：付款期限、逾期處理階梯、滯納金（建議月 0.5%）。

### 5.4 到期與緩衝期（修正版）

- 合約到期 or 企業移除某人 → 該席次進 `grace`，**30 天**
- 緩衝期內：**帳號留著、記憶都在、不出現任何付費牆**，但**用量回到免費版**
  - ⚠️ 這一條 2026-07-20 修正過。原設計是「服務等級不變」＝公司不付錢我們還燒 30 天通話費，會把成本壓在自己身上，已作廢。
  - 長輩的感受是「今天聊得比較短」，不是「她消失了」
- 緩衝期內聯繫家屬：公司方案結束了，要不要自己接著用（附轉換優惠）
- 30 天後未轉自費 → `released`
- 提醒：合約到期前 **30 天／7 天** → 後台跳提醒（給我們自己）
- 席次使用率低於 50% → 月報產出時標注（續約談判籌碼）

### 5.5 降級鐵律
降級**不是切斷**，是回到免費版的正常用量。**任何情況下不得出現「請訂閱」擋住長輩的畫面。**

理由：一位每天跟寧寧講話的 80 歲長輩，某天早上打開看到付費牆——這對長輩是傷害，對品牌是災難。

## 5A. 個人已付費者的處理（拍板）

員工的爸媽可能**已經自己在 App Store 買了 Plus／Pro**。企業席次不得與之衝突。

**總原則：取高的那個，絕不重複收錢。**

### 情況 A｜企業等級 ≥ 個人已買的等級
- 立刻改吃企業席次
- App 內跳溫和通知：「您的公司已為您支付會員，**您可以取消自己的訂閱了**」＋ 附取消步驟連結
- 🔴 **我們不得、也不會代為取消 Apple 訂閱**（技術上無權、法律上不該）。通知必須寫清楚是用戶自己去取消，否則用戶誤以為我們會處理 → 被重複扣款 → 客訴

### 情況 B｜企業等級 < 個人已買的等級
- **不動他**，個人訂閱繼續有效
- 企業席次掛新狀態 `waiting`（等待中）
- 個人訂閱到期日當天自動接手，用戶無需任何操作、無空窗

### 計費
`waiting` 狀態的席次**不計費**（沿用 4.1「只有 `active` 才計費」的規則）。真正接手後才起算。公司也認同——那個人本來就有了。

### 三條硬規則
1. 絕不同時扣兩份錢
2. 絕不幫用戶取消 Apple 訂閱（只提醒、附教學）
3. `waiting` 不計費

> `enterprise_seats.status` 因此擴充為：`pending` / `waiting` / `active` / `grace` / `released`
