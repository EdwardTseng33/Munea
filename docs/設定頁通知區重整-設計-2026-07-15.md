# 設定頁「健康資料安全」通知區重整 — UX + 前後端 API 設計

> 2026-07-15 Edward 拍板：通知與提醒／通知中心／測試通知／鎖定畫面內容 四列體驗混亂、部分沒作用。
> 原則：一般 App 的做法——設定裡一個推播開關由用戶自行決定；拿掉測試通知與鎖定畫面內容。

## 一、UX 定案

### 設定頁只留一列
| 原本 5 列 | 改為 |
|---|---|
| 通知與提醒／通知中心／測試通知／鎖定畫面內容／安全通知 | **「通知中心」一列**（右側顯示 已開啟/已關閉 ›） |

### 「通知中心」內頁結構（2026-07-15 Edward 定稿）
1. **推播通知 總開關**（預設關；開啟時觸發 iOS 系統授權；系統拒絕時引導到 iPhone 設定）
2. **分類開關**（總開關開啟才可用）：用藥提醒／看診提醒／家人消息
3. **安全通知**（原獨立列整合進來）：開關＋說明「偵測到異常（危機字句、長時間無回應等）會通知你授權的家人」。
   ⚠️ 安全邊界：此開關只停「自己手機」的安全推播；**通知授權家人的守護機制不受此開關影響**（由家人頁的授權管理），避免一鍵把守護腦告警全關、出事沒人知道。

### 拿掉的東西
- **測試通知**：一般用戶不該看到。保留能力但移進開發包隱藏區（development profile 才顯示）。
- **鎖定畫面內容 開關**：不給開關、直接內建為固定隱私行為——**所有推播文案一律不含藥名、健康數值與家人訊息內文**（例：「該吃藥了，打開沐寧看看」）。細節只在 App 內看。
  - 附帶根治 #48 殘留問題：後端用原始藥名組推播繞過顯示守門 → 改為通用文案後不再組原文，外文髒字也無從外漏。

## 二、API 設計（前後端契約）

### 設定
- `GET /notifications/settings` → `{ pushEnabled, categories: { medication, clinic, family, safety }, deviceRegistered }`
- `PUT /notifications/settings`（同構、冪等；登入者身分綁定，寫 Supabase）
- 總開關關閉＝**伺服器端停發**（不只前端不顯示）；分類關閉＝該類事件仍寫後端紀錄、但不推本人 APNs。
- `safety` 開關語意：只停本人手機推播；**家人告警路徑不受影響**（伺服器端規則）。

### 裝置與發送（沿用 #52 通知平台，不重造）
- `POST /push/devices`（既有）：註冊/更新 APNs token；登出或總開關關閉時登記停用。
- APNs outbox 發送前查 settings：`pushEnabled && categories[type]` 才出手機；payload 一律通用文案（隱私內建）。

### 紀錄
- `GET /notifications?limit=`（既有 durable events）：後端保留供家人安全通知與稽核；本次設定頁不使用。

## 三、實作分工
- **後端**（engine/server.py＋supabase migration：`notification_settings` 表或併入 app_profile）：settings GET/PUT、outbox 發送前檢查、通用文案組字。
- **App**（web/src/app.js＋設定頁 HTML；影響包版、需升版重包）：四列併一列、內頁、開發包才顯示測試通知。與 Codex 通知平台（#52）銜接，動工前照規矩看板認領。

## 四、驗收
- 設定頁只剩一列「通知」；總開關/分類開關真的能停推播（伺服器端驗證）。
- 任一推播的鎖屏文案不含藥名/數值/訊息內文。
- 正式包看不到「測試通知」；開發包看得到。
