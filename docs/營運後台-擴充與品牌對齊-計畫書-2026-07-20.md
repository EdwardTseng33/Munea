## 0. 現況同步（前後端 / API / 資料排點）

**架構**
- 前端：`web/admin.html`（外殼）＋`web/src/admin.js`（選單、`EP_LIST`、各頁畫面）＋`web/src/admin.css`（樣式）。
- 後端：`engine/server.py` 的 `/admin/*` handler（例：`north_star_summary`、`admin_credits_summary`…），資料走 `data_backend()`／Supabase。
- 資料庫：Supabase，49 張表，schema 在 `supabase/sql/001~019`。
- 兩道門：① `X-Munea-Key`（全站薄門，擋外部）② `X-Munea-Admin-Token`（只有 `/admin/*` 走）。**兩道都是共用密碼、不是個人帳號**——這點會擋到「守護反應力」的「誰處理的」。

**現有 9 個接口**：north-star／usage／accounts／subscription-metrics／feedback／safety-events／privacy-requests／conversation-summaries／audit-events。
**已寫好、沒開燈的 3 支**：`/admin/credits`（點數經濟，完整可用）、`/admin/voice-diagnostics`（通話診斷）、`/admin/notifications/drain`（動作型）。

**三個要 Edward 知道的底層真相**
1. **兩張分析表從沒被寫過**：`cost_ledger`（成本）、`daily_user_metrics`（每日彙總），還有 `voice_session_metrics`／`ai_brain_runs`——只建表、零寫入。影響「單位經濟／毛利」「留存曲線」「算力健康」，這幾項要先補資料寫入才算得出來。
2. **兩條排程沒接**：推播外寄匣、過期通話席位回收，都沒有定時觸發（已知 P0 缺口）。影響「通知送達率」「算力健康」的數字穩定性。
3. **用戶意見存在本機檔案（`feedback_store.json`）不是資料庫**——雲端重新部署可能歸零，有掉資料風險，建議順手搬到 Supabase。
