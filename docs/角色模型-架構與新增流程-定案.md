## 一、架構總覽（一個角色 = 三層 + 四種素材 + 兩個設定）

**顯示三層**（見顯示定案手冊）：
1. **全身立繪（背景層）** = 9:16 完整照片鋪滿手機（`web/flashhead/bg-aXX.png`，1080×1920）
2. **會動的臉（模型層）** = 模型只動「頭~肩」框、輸出貼回立繪對應位置
3. **撥號前（等待層）** = 打招呼→待機影片

**資料流（一句話）**：一張 9:16 原始立繪 →（裁「頭肩框」）→ 條件圖餵臉模型 → 模型生動臉 →（貼回同一個框）→ 疊在全身立繪上。

## 三、關鍵原則（為什麼要「一角色一調」——今天男生踩的坑）

**原則一 · 頭肩框要一角色一調，讓臉填滿框**
- 每個角色在原始立繪裡「頭的位置與大小」不同。用同一組裁切框，有的臉大（清）、有的臉小（軟）。
- **今天實例**：女生頭低/大 → [top8%,height60%] 臉就填滿=清；男生頭高/小 → 同框臉只佔一小塊=同 640 卻軟。男生改 [top14%,height50%]（裁緊、放大臉）才追平。
- **選框準則**：從全身立繪裁「頭~肩」，讓**臉在 640² 裡夠大**（眼睛約在畫面上 1/3、下巴約中線、肩膀入鏡收尾）。頭高的角色 top 往下、height 縮小；頭低的角色相反。

**原則二 · 640 是基礎服務解析度**（512 對 2026 螢幕太糊、不用）。輸出 640² 正方形工作格式（壓縮、非給人看）。

**原則三（鐵律）· 條件圖裁切框 % = App 顯示框 %**
- 伺服器裁條件圖用的 [top,height]，和 App `_box` 的 [top,height] **必須同一組數字**，臉貼回立繪才不變形、不錯位。
- 幾何原因：條件圖是「原立繪的 [t,h] 區域 resize 640²」；App 把 640² 用 `object-fit:fill` 拉回「立繪的 [t,h] 顯示框」。兩邊 % 一致→拉伸比例一致→對齊。**改框＝兩邊一起改，且同一次包版上線**（只改一邊會錯位）。

**原則四 · 換框要「伺服器＋App 同一次包版」**（不然現役 App 錯位）。

## 四、新增角色 SOP（照這 8 步、不試錯）

1. **原始立繪** 放進 `avatar-candidates-9x16-final/<名>.png`（9:16、臉清楚、頭肩入鏡）。
2. **調頭肩框**：用 PIL 從立繪裁 [top t%, height h%]→640²，肉眼比對「臉夠不夠大、跟現有角色同級」，定出這角色的 (t,h)（工具：`deploy/demo-videos/` 那段裁切 python，改 t/h 出幾版比對）。
3. **條件圖上伺服器**：把該裁切圖 scp 成 `/root/char-aXXB.png`，重啟服務。
4. **全身立繪**：原始立繪 →（滿版處理）→ `web/flashhead/bg-aXX.png`。
5. **App 設定**：`FLASHHEAD_CHAR_MAP` 加「中文名→aXX」；`_fhComposite` 的 `_box` 加這角色的 (t,h)（跟步驟 2 同一組）。
6. **大頭照**：從立繪裁頭肩 → thumbAsset/homeAsset/#avatarPick。
7. **聲音＋人格**：CHARS 加 `voice`（挑聲線）＋ `persona`。
8. **撥號前影片**：生 hello/idle → `web/avatars/motion/<名>-*`。
9. **驗收**：撥號前墊圖/影片、接通後活臉、大頭照 三處同一張臉；臉夠清（跟現有角色同級）；不變形。**伺服器＋App 同一次包版上線**。

## 六、踩過的坑（別重踩）
| 症狀 | 根因 | 解 |
|---|---|---|
| 某角色臉較軟（同 640） | 頭在原圖較高較小、裁切框沒為它調→臉分到畫素少 | 一角色一調頭肩框、讓臉填滿 |
| 臉貼回立繪變形/錯位 | 條件圖框 % ≠ App 顯示框 % | 兩邊同一組 %、同一次包版 |
| 換角色後撥號前新臉、接通舊臉 | 只換了部分素材 | 8 步全走、四種素材都換 |
