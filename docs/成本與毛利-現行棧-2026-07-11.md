## 一、Gemini Live 語音真實單價

**信心程度：中高（三次獨立查證官方定價頁 ai.google.dev/gemini-api/docs/pricing，數字一致；per-分鐘換算有小幅四捨五入誤差但已用官方直接公布的分鐘價，非我自行推算）**

Munea 用的模型類型應屬 Gemini Live 原生語音對話（native audio dialog），對照官方定價頁上兩個可比模型（`gemini-2.5-flash-native-audio-preview-12-2025` 與 `gemini-3.1-flash-live-preview`）音訊費率完全相同：

| 項目 | 每 100 萬 tokens | 官方公布的每分鐘等值價 |
|---|---|---|
| 語音輸入（audio input） | US$3.00 | US$0.005 / 分鐘 |
| 語音輸出（audio output） | US$12.00 | US$0.018 / 分鐘 |

- 語音輸出的分鐘價可交叉驗證：輸出音訊 tokenize 速率為 25 tokens/秒 → 每分鐘 1,500 tokens → 1,500 × $12/1,000,000 = **$0.018/分鐘，與官方數字完全吻合**（高信心）。
- 語音輸入的分鐘價官方公布 $0.005/分鐘，用 tokens/秒 反推約落在 25-32 tokens/秒區間（不同來源數字不完全一致，但對最終金額影響很小）。**本分析直接採用官方公布的 $0.005/分鐘，不自行用有疑義的 tokens/秒 反推**，以官方數字為準。
- 換算台幣：輸入 US$0.005/分 × 32 = **NT$0.16/分鐘**；輸出 US$0.018/分 × 32 = **NT$0.576/分鐘**。

來源：[Gemini Developer API pricing（官方）](https://ai.google.dev/gemini-api/docs/pricing)

### ⚠️ 重要成本風險：逐輪累積計費（未反映在上面單價、但會實際推高帳單）

查證發現一個**必須讓 Edward 知道的機制性風險**，且有 Google 官方論壇回覆確認為真（信心：中高，非單一來源猜測）：

> Gemini Live 的 session 是**逐輪（per-turn）累積計費**——每一輪對話，系統會把「這一輪新內容」加上「這個 session 之前所有輪次累積的音訊 tokens」一起重新計費，不是單純照通話時長線性算。實測案例顯示：兩通長度接近的通話（5.6分 vs 5.8分、7.1分 vs 7.1分），因為「輪次」（一來一往的次數）不同，帳單相差 **31%-37%**。也就是說**對話越「碎」（一來一往次數越多），成本比純線性估算更貴**；反之少數幾輪、每輪講比較長，成本比較接近線性估算。

**這代表什麼**：
- 下面第二、三節的「正常情境」是用**官方公布的線性分鐘價**直接算出來的**理論下限**，實際帳單很可能更高。
- 我另外算了一個「悲觀情境」，在正常情境的 Gemini Live 成本上加 **35%**（取實測 31-37% 區間中點）作為風險緩衝，這才是比較保守可信的規劃數字。
- **強烈建議**：技術端務必開啟 Google 建議的 `contextWindowCompression`（壓縮觸發閾值如 25,000 tokens、滑動視窗如 8,000 tokens），否則長對話（尤其是長輩很愛聊、一來一往很多輪的場景）成本會比預期高不少。這不只是成本問題，官方原話是不開啟壓縮的話 session 甚至無法無限延長。
- 另外有個技術限制要注意：**純語音 Live session 上限 15 分鐘**、需要用「session resumption」機制無縫接續，重強度使用者（一天 60 分鐘）若是一次性長通話，技術上需要重連至少 3-4 次。這不直接增加成本，但影響工程設計，順帶提醒。

來源：[Google AI Developer Forum 討論串（含官方回覆確認）](https://discuss.ai.google.dev/t/does-gemini-live-native-audio-bill-cumulative-prompt-tokens-on-every-turn-cost-seems-to-scale-with-turn-count-not-call-duration/173248)｜[Live API best practices（官方，contextWindowCompression 建議）](https://ai.google.dev/gemini-api/docs/live-api/best-practices)

## 四、紅線結論

1. **Plus（NT$499）安全上限約 18-28 分鐘/天**（依情境假設不同），保守建議用 **~20 分鐘/天** 當內部警戒線。
2. **Pro（NT$999）安全上限約 37-57 分鐘/天**，保守建議用 **~40 分鐘/天** 當內部警戒線。
3. **中強度使用（30分/天）已經是 Plus 方案的危險區**——幾乎所有情境下毛利轉負或逼近零。若長輩用戶黏著度高（這正是產品想要的結果），Plus 方案在成本結構上撐不住。
4. **重強度使用（60分/天）不管哪個方案都虧錢**——這是產品成功、但商業模式會失血的情境，需要及早設計因應（例如：用量到一定門檻後溫和限流/提示升級、或重新設計定價分層、或找 Gemini Live 更便宜的替代方案）。
5. **逐輪累積計費是目前最大的不確定性來源**——實際成本可能比「正常情境」高 30-40%，技術端務必落地 `contextWindowCompression`，且應在正式上線後用真實用量數據回填校準這份估算（現在的數字是查證+假設推算，不是實測值）。
