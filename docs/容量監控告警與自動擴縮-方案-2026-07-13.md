## 給 Edward 的人話摘要（先看這段就夠）

**現在的狀態**：軟體骨架昨晚已經建好一半——閘道會排隊、會判斷「顯卡跟語音都有空位才放行」，引擎也能一台卡塞多人。但這些東西目前**沒有眼睛**：沒人知道現在幾個人在通話、卡快滿了沒人會講、真的滿了也不會自動開新卡。這份方案就是要把「眼睛」跟「反射動作」補上。

**分兩階段，因為開卡的鑰匙目前是壞的**（客服正在補發正確的）：

- **階段 1（現在就能建，不用等鑰匙）＝ 監控＋告警＋人工開卡**。系統會知道「現在幾人在通話、卡用到幾成、有沒有人在排隊、語音有沒有變慢」，快滿了會在 Slack 敲你或蘇菲，我們手動開一張卡頂著。這階段**不會盲目**——出事會有人知道、也有人能立刻反應。
- **階段 2（等到正確鑰匙後）＝ 系統自己開卡/退卡**，不用等人看 Slack 再手動按。

**我的建議（原生 vs 自建）**：GLOWS 後台雖然有一個「Auto Scaling」的現成功能，但我不建議靠它當主力——因為我們的開卡不是「流量大就多開一台預設機器」這麼單純，而是要「從快照開（帶著整組模型環境）→ 起來後自動跑我們的臉引擎服務 → 立刻告訴閘道新機器的門牌 → 還要跟語音池一起判斷有沒有空位」，這一串客製邏輯，通用的 Auto Scaling 工具大概率不懂。所以主力用我們自己寫的 `glowsctl.py` + 閘道；GLOWS 原生 Auto Scaling 值得花 15-30 分鐘點進去看清楚能不能當「備援雙保險」，但不會是唯一依靠。

**階段 1 要多少工**：軟體部分（監控腳本＋告警規則＋排隊等待時間統計）樂觀 2.5 天、保守 5 天（雙軌工時，下面 §D 有拆解）；當中「寫程式＋單元測試」用移動城堡並行執行只要 1-2 小時就能生出來，真正花時間的是「拿現役那張真卡實測監控腳本真的抓得到數字、Slack 真的會響」跟 Edward 審核回合——**能趕上首波**，因為完全不需要等 token，現在就能開工。

### A.2 彙整方式：新增 `deploy/gateway/monitor.py`（階段 1 核心工作）

一支輕量常駐輪詢腳本（CPU-only，不需要 GPU）：

1. 每 15-30 秒對每台**已登記進閘道**的 GPU worker 打一次 `/health`，把回傳的 `capacity` / `gen_compute_ms_rolling` / `audio_underrun` / `video_underrun` 更新進閘道（沿用現成的 `POST /v1/admin/worker/{id}/health` 端點，不必新開介面）。
2. 對語音橋打新增的 `/healthz`，抓 `_key_active` 總和跟每把鑰匙的占用。
3. 把彙整後的快照（沿用 `Gateway.snapshot()` 的形狀，加上語音第一句延遲跟卡失敗計數）存成一份會被覆寫的 JSON（例如 `deploy/gateway/last_snapshot.json`），同時可以掛一個 `GET /v1/admin/metrics`（給人看的儀表，跟既有 `/health` 分開，避免混淆客戶端用的健康檢查跟維運看的指標頁）。
4. 每輪彙整後跑一次 §B 的門檻判斷，命中就發 Slack。

不重造輪子：`gateway_core.WorkerRegistry.burst_threshold_hit(0.8)`、`Gateway.joint_free()`（聯合准入 min 邏輯）、`Gateway.queue.depth()` 這些判斷邏輯**都已經寫好在 `gateway_core.py`**，monitor.py 只是「定時去問一次真實數字、灌進去、照既有邏輯讀結果」，不必重寫判斷邏輯本身。

### B.2 管道：沿用既有 Slack 雙線，不另建

`engine/notify.py` 已經有一套現成機制：`MUNEA_SLACK_OPS_WEBHOOK`（#沐寧-營運，好消息線，訂閱/購買/意見回饋在用）、`MUNEA_SLACK_ALERT_WEBHOOK`（#沐寧-告警，壞消息線，帳號刪除/引擎 500 已在用），且**已內建 10 分鐘同類防洪**（`_last_alert` dict）——容量告警語意上屬於「壞消息/要人反應」，走 `notify.alert(kind="capacity", where=worker_id或queue或voice, detail=...)` 這條線最貼切，不需要新建 Slack App、不需要新的 webhook 網址。

**唯一要處理的接線問題**：閘道（`deploy/gateway/`）是刻意設計成**零外部依賴、CPU-only 獨立服務**（跟 `engine/server.py` 分開部署），不應該反過來 import engine 內部模組造成兩個服務耦合。做法：在 `deploy/gateway/` 底下新增一支約 20 行的 `slack_notify.py`，複製 `notify.py` 的 `_post`/`alert`/`ops` 邏輯（同樣讀 `MUNEA_SLACK_ALERT_WEBHOOK` / `MUNEA_SLACK_OPS_WEBHOOK` 環境變數，同一組 webhook 網址、不用重新申請），這樣閘道跟語音/引擎共用同一個 Slack App、但程式碼互不依賴。

## C. 自動擴縮流程——完整決策樹

### C.1 全流程（撥通 -> 監控 -> 觸發開卡 -> 補位 -> 縮容）

```
用戶按下撥通
 |
 |- Gateway.request_call()：查「顯卡池空位 AND 語音池空位」（joint_free，已實作）
 |
 |- 有空位 -> fullest-first 配對到 worker（已實作）-> 直連 worker /offer
 |      |- monitor.py 每輪把最新 active/使用率灌回閘道
 |      |- 使用率 > 80%（burst_threshold_hit）-> B.1 門檻命中 -> 發 Slack「建議開卡」
 |             階段1：人工看到 Slack、手動 glowsctl.py create + 手動 register worker
 |             階段2：monitor.py 自動呼叫 glowsctl.py create（或 GLOWS 原生 Auto Scaling）
 |
 |- 沒空位、佇列 < 上限 -> 進排隊（已實作，FIFO＋離線遞補）
 |      |- 排隊深度 > 0 -> Slack「排隊中，建議立刻開卡」（比 80% 門檻更急）
 |      |- 新卡就緒（不管人工或自動）-> 呼叫 POST /v1/admin/worker/register 登記進閘道
 |             -> Gateway.try_advance_queue()（已實作）自動把排隊最前面還在線的人接上
 |
 |- 佇列已滿 -> 真拒絕（已實作，Slack 告警：佇列逼近上限那條，提前預警避免真的走到這步）

通話結束
 |
 |- worker 呼叫 POST /v1/call/release（**這段目前還沒接，見 §D.1 待辦**）
        -> 閘道釋放槽位 + 記錄通話時長 + try_advance_queue()
        -> monitor.py 偵測到某 worker 連續 active=0 超過閒置門檻（建議 5-10 分鐘）
             -> 階段2：呼叫 glowsctl.py snapshot（保留環境）-> release（DELETE、停止計費）
             -> warm anchor（保底 1-2 台）例外，永遠不縮
```

### C.2 擴容觸發規則

- **提前墊卡**：池使用率 > 80% 就該預開下一台（不是等 100% 滿才開，冷啟藏在滿載之前）——邏輯已在 `WorkerRegistry.burst_threshold_hit()`，只差「偵測到之後誰去按開卡」這個動作。
- **急件**：排隊深度 > 0（代表已經有人真的在等）-> 優先權高於 80% 門檻，應該立刻觸發（不用等下一輪 80% 判斷週期）。
- **fullest-first 塞卡**：已經在 `WorkerRegistry.pick_fullest_first()` 實作，優先塞進「已經有人在用、還有空槽」的卡，讓其餘卡盡量閒置好被回收——擴縮邏輯不用重做。

### C.3 縮容觸發規則

- **worker 閒置門檻**：`active == 0` 超過 5-10 分鐘才退卡（**不能設太短**——GLOWS 是「刪除整台」不是「暫停」，重開要 snapshot+create、秒數未實測，太頻繁開關反而更貴更慢；也不能設太長，避免空燒 NT$15.7/hr）。
- **warm anchor 例外**：保底 1-2 台（現有 RunPod 美/Glows 台）首波期間不跟著自動睡，作為零冷啟基本盤。
- **縮容動作**是先 `glowsctl.py snapshot <ins-id> <名字>` 保留環境，再 `glowsctl.py release <ins-id>`（DELETE，真的停止計費）——兩步都要做，只做 release 不做 snapshot 的話，下次要重建環境得走 5 分鐘的手動重建 SOP（`deploy/glows/README.md` 已有）。

### C.4 Snapshot 快速重開

- 目前 `glowsctl.py cmd_create()` 是從固定映像 `DEFAULT_IMAGE`（CUDA12.8 Torch2.7.1 Base）開機，**不是從 snapshot 開**——階段 2 要補一個 `from_snapshot_id` 參數（需要先查 GLOWS SDK `POST /instance` 是否支援這個欄位，見 §E）。
- 新機器起來後 SSH 埠號、HTTP 門牌一定會變（README 已知眉角）——自動化流程必須「create 完先查 instance 詳情拿新門牌，再呼叫 `POST /v1/admin/worker/register` 動態登記」，不能寫死網址。
- 重開要多久（snapshot 到 create 到服務就緒）**目前完全沒有實測數字**，這是本方案排在最前面的未知數（直接決定「排隊該提前多久觸發擴容」），見 §E。

### C.5 庫存上限保護

- `glowsctl.py` 已有雙區 fallback（`DEFAULT_REGIONS = ["TW-03", "TW-04"]`，`cmd_create` 第一區開不出來自動換第二區）。
- 若兩區都吃緊，下一步是美國 RunPod 線（已備但目前是關的，見 7/12 架構圖 §7.1）。
- 台灣區實際庫存尖峰可用性今晚沒有系統性查（可跑 `glowsctl.py specs` 確認即時規格/庫存），列入 §E。

### C.6 與排隊、語音聯合准入的接法

`Gateway.joint_free() = min(顯卡池空位, 語音池空位)` 已經是現成邏輯——擴縮容只影響顯卡池那一半（`WorkerRegistry`），語音池那一半是獨立的多鑰匙擴充（`GEMINI_API_KEYS`），兩邊各自擴、閘道永遠用兩者的交集做最終放行判斷，不需要為這份方案重新設計聯合准入，只需要讓 monitor.py 把兩邊的即時數字都灌進閘道即可。

## D. 分兩階段施工

### D.1 階段 1（首波、不用 token）＝監控＋告警＋人工開卡

**具體要寫的程式**：

| # | 檔案 | 內容 | 樂觀工時 | 保守工時（x2）|
|---|---|---|---|---|
| 1 | `engine/live_voice_server.py`（改） | `process_request` 加一個 `/healthz` 分支，回傳 `_key_active` 總和＋每把鑰匙占用＋（若已埋）第一句延遲滾動平均 | 0.25 天 | 0.5 天 |
| 2 | `engine/live_voice_server.py`（改，可選但建議） | 加「session 建立到第一個 caption 送出」的時間差埋點，回報進 `/healthz` | 0.25 天 | 0.5 天 |
| 3 | `deploy/gateway/gateway_core.py`（改） | `QueueEntry` 加 `enqueued_ts`；`Gateway.try_advance_queue()`/`_connect_response()` 命中連線時計算等待秒數，累積另一個滾動平均（沿用 `_recent_call_s` 的 deque 手法）；worker `mark_health(False)` 次數計數 | 0.5 天 | 1 天 |
| 4 | `deploy/gateway/slack_notify.py`（新增） | 複製 `notify.py` 的 `_post`/`alert`/`ops`，CPU-only 零依賴，跟閘道其餘程式碼風格一致 | 0.25 天 | 0.5 天 |
| 5 | `deploy/gateway/monitor.py`（新增，核心） | 輪詢已登記 worker 的 `/health`＋語音 `/healthz`，灌回閘道（用既有 admin 端點），跑 §B 門檻判斷，命中發 Slack，輸出彙整快照（`GET /v1/admin/metrics`） | 1 天 | 2 天 |
| 6 | `deploy/glows/manual-scale-sop.md`（新增文件，非程式） | 「Slack 告警到人工按 glowsctl.py create 到手動 POST /v1/admin/worker/register 登記到通話結束後手動 release 端點/退卡」的操作 SOP，Edward 或蘇菲照著做即可，不用等工程師在場 | 0.25 天 | 0.5 天 |

**階段 1 總工時：樂觀 2.5 天 / 保守 5 天**（雙軌工時標準）。

**移動城堡估**：上面 6 項都是「有現成單元測試框架可套」的小模組（`gateway_core.py` 已有完整假引擎測試手法，`scripts/test_gateway.py` 可以直接照抄擴充），並行執行＋寫測試，agent 端 1-2 小時可以生出全部程式碼＋單元測試全綠（不需要真卡）。真正花時間的是兩塊：一是 Edward/蘇菲審核回合（Chrome MCP 走一次排隊流程、看 Slack 訊息長相對不對）；二是**拿現役那張真卡做一次唯讀 `/health` 輪詢實測**——這步驟需要現役卡的 SSH/門牌（今晚 Edward 已示範過給法，不需要 SDK token，走人工給門號的方式也行），只是唯讀查詢、零風險、不影響服務中的通話。這兩塊合計抓半天到一天的人工審核時間，**整體能趕上首波**，因為完全不卡在 token 上。

**能不能趕上首波**：能。這階段的所有程式碼都不需要真的開新卡、不需要 SDK token——只需要「讀」現役卡跟語音服務的既有 `/health`／新加的 `/healthz`，然後決定要不要發 Slack。唯一的人力依賴是階段 1 最後一步「人工開卡」——這本來就是現況（Edward 昨晚就是這樣手動 SSH 進卡做實測），只是現在多了「系統會主動提醒你該按了」，比現在的「完全沒人知道」進步一大截。

### D.2 階段 2（拿到 token 後）＝全自動開卡/退卡

**具體接法**：

1. `glowsctl.py cmd_create()` 補 `from_snapshot_id` 參數（先查 GLOWS SDK doc 的 `POST /instance` 是否支援，若支援直接加一個可選欄位；若不支援，退回「開新機＋重跑 install-flashhead.sh」的較慢路徑，估時要重算）。
2. `monitor.py` 的門檻判斷從「發 Slack 通知人工按」升級成「直接呼叫 glowsctl.py create（或封裝成 Python 函式呼叫，不必真的拉 subprocess）到起來後自動查 instance 詳情拿新門牌到自動 POST /v1/admin/worker/register」，全程不用人碰。
3. 縮容同理：`monitor.py` 偵測到 worker 閒置超過門檻，自動 `glowsctl.py snapshot`，再自動 `glowsctl.py release`。
4. Slack 訊息從「要你決定」降級成「純知會」（系統已經自己處理了，只是讓 Edward 知道發生過），除非自動擴容失敗（例如兩區都開不出卡）才升回「要人介入」。

**GLOWS 原生 Auto Scaling vs 自建（最終建議）**：

我沒有把握斷言 GLOWS 原生 Auto Scaling 的完整行為（今晚只看到後台選單存在，沒有深入測試它的參數跟觸發方式），但基於已知的硬事實可以給一個有把握的方向判斷：

- 我們的「開卡」不是通用型「流量大就多開一台預設鏡像」——而是要：第一，從 snapshot 開（帶著整組模型權重跟環境，不是空鏡像）；第二，起來後自動跑我們的 `flashhead_server.py` 服務進程；第三，立刻讓閘道知道新機器的 SSH/HTTP 門牌（每次都變）；第四，要跟語音池一起做聯合准入判斷（GLOWS 原生工具大概率不知道語音池的狀態，那是我們自己的邏輯）。這四件事是產品特有的客製流程，通用型 Auto Scaling 面板大概率只做「閾值到了開一台預設鏡像」這種動作，不太可能懂「從我們的 snapshot 開、接進我們的 gateway」。
- **建議**：主力用自建（glowsctl.py + monitor.py + gateway_core.py），因為一，已經寫了一半（fullest-first、聯合准入、排隊都在 gateway_core.py 裡）；二，這是產品最關鍵的差異化邏輯（GPU 池要跟語音池一起看空位），原生工具大概率做不到。
- GLOWS 原生 Auto Scaling **值得花 15-30 分鐘點進去看清楚**它支不支援「事件觸發時打一個 webhook 通知外部系統」這種半自動模式——如果支援，可以當自建方案的**備援雙保險**（原生系統偵測到異常時多一層通知），但不應該取代自建的聯合准入邏輯，因為那部分是原生工具結構性做不到的事。
