## 2. 補齊三批（優先序）

### 🔴 批次 0 · 上線前必補（會出事等級）
**判準：AI 講錯會真的害到脆弱長輩、且送審/法遵有風險。**

1. **精神狀態異常辨識 ＋ 不強化妄想**（守護腦新增 `psychosis_terms` + 回應策略）
   - 為什麼急：長輩失智、譫妄、被害妄想很常見；AI 順著「鄰居用電視監控我」講話會加深妄想、真的害人。APA 明確點名生成式心理工具在脆弱族群曾出現不安全互動。
2. **保護與安全事件辨識 ＋ 轉介**（守護腦新增 `protection_terms` + 113 專線）
   - 為什麼急：家暴/高齡虐待/財務剝削是長輩高風險情境，AI 至少要聽得出來、把人轉介到 113 保護專線，不能裝沒聽到。
3. **傷害他人辨識**（守護腦 `critical_terms` 補他傷分支）
   - 為什麼急：目前 critical 只認「自傷」，明確威脅他人 + 有工具計畫完全沒接住。

> 批次 0 的可施工設計見 §4。工作量小（詞庫 + 應對範本 + 五級歸位），不擋上架、但要在送審前進守護腦。

### 🟡 批次 1 · v1.x（產品深度、上架後陸續加）
4. **心理支持四框架**（CBT/ACT/MI/SFBT）——先做「知識約束層」：把四框架的核心手法（情緒反映、開放式提問、正常化、動機式晤談的「不說教探索改變意願」、CBT 的「事件→想法→情緒→行為」拆解、ACT 的「接納 + 微小行動」、SFBT 的量尺/例外問題）寫成管家腦與聊天腦的對話原則，不做正式治療。
4-B. **情緒壓力／關係壓力（灰色地帶軟性情緒傷害）**（2026-07-12 Edward 提問補入）——情緒勒索、親情逼迫（催婚催生）、霸凌、被嫌棄、「我是為你好」的控制。特性：**不是危機（不觸發專線）、不是家暴（不到通報），是長輩最日常的情緒痛**，掉在危機分類的縫裡。**正解不在攔截、在「怎麼接」**（守護腦擋不掉「我是為你好」這種字面中性的話）。立場＝**中立接住＋賦能長輩**：不否定感受、幫分清「別人做的事」與「自己的感受」、正常化、**不選邊不挑撥**（不替施壓者講話、也不罵對方——只聽一面、無立場評斷、選邊反傷關係）、把重心放回他的界線與心情。**回應原則已先加進 `chat_engine.py` CORE ④-B**（上線前生效、語音線 Gemini 讀了自律）。深度處理（拆想法、找價值、找回界線）＝CBT/ACT（本批）；**隱晦的情緒勒索靠第二層 AI 讀語意、不靠加詞庫**（「為你好」是中性字、關鍵字抓不到）。✅ **Edward 2026-07-12 定案 A（中立接住＋撐住長輩）**：接住感受、幫他找回自己的界線，但不評斷家人對錯（只聽一面、無立場評斷、選邊反傷關係）。說明書 `chat_engine.py` CORE ④-B 本就照 A 的分寸寫、無需改。
5. **情緒辨識補全**——`distress_terms` 補內疚/羞愧/挫折/失落/混亂；加「確認式情緒反映」語句原則。
6. **就醫前問題整理**——寧寧「我幫你記下來、下次問醫生」已是雛形，做成結構化：一次看診前，把最近的身體狀況/疑問整理成一張可帶去給醫生的清單。
7. **失聯警示**——多久沒開 App/沒互動 → 溫和通知家人「今天還沒跟○○聊到」。
8. **健康數據補全**——血糖、體重、體溫（Apple 健康既有數據，接告警設計 §一的進階區）。

## 4. 批次 0 可施工設計（照這個直接動守護腦 `engine/model_router.py`）

> 原則：新增類別一律走守護腦既有格式（`categories` + `level` + `action` + `responsePolicy`）；**應對策略層（怎麼回）由聊天腦/管家腦照範本演，不寫死台詞**。回應範本可放進角色人格聖經當「觸發時回應範本」。

> 🔴 **產品真相：用戶只能語音、守護腦要「盯字幕」接回語音線（2026-07-12 Edward 校正 + 翻程式碼確認 · 修正前一版框架）**：
> - **產品實際只有語音線**（`live_voice_server.py`，聊聊）——用戶只能開口講、不能打字；打字線（`server.py` /chat）是測試／備援，**真實用戶不走**。AI 輸出＝逐字稿（字幕）＋語音。
> - **守護腦（`guardian_evaluate_response`）目前只在打字線逐句跑**；用戶實際走的語音線接通後聲音直接串流給 Gemini Live、中間沒有守護腦。**⇒ 紅線測試 13/13 過的是「用戶用不到的那條路」；用戶真正走的語音線，紅線目前完全靠 Gemini 讀說明書自律＝單層。這是全計畫最該優先補的安全缺口。**
> - **解法（技術已具備、不必大改）**：語音線兩邊都有即時逐字稿——用戶說的（`input_audio_transcription`）＋寧寧說的（`output_audio_transcription`）程式裡都已即時取得並送出（`live_voice_server.py:195-196, 458-463`）。守護腦改成**「並行監聽字幕」**：不擋在中間（不卡通話），每句字幕一出現就順手丟守護腦看一眼，命中危機／急症／紅線就即時 ①遞安全指令讓寧寧下一句轉向轉介 ②觸發家人通知／專線 ③記安全事件。寧寧自己講出來的話也同樣盯著、講錯立刻補救。
> - **這一「看」要分兩層**（沿用 `AI-SERVICE-DESIGN` Guardian「deterministic rules → classifier review」設計）：**第一層＝危險字清單**（快、免費、即時，抓「不想活」「胸口痛」「他打我」這種講白的關鍵字）；**第二層＝一個便宜 AI 讀懂語意**（Gemini Flash 級背景跑，抓「我是拖累」「東西都交代好了」「走幾步就要停」這種**沒講危險字、卻是危機／急症／被害訊號的拐彎話、含蓄、上下文**——長輩表達往往客氣、迂迴，只靠清單會漏接）。**省成本＝清單先過濾、只有清單抓到苗頭但拿不準的才升給 AI 細讀**，不必每句都勞動 AI。兩層都在背景並行、不卡通話。
> - **⇒ 優先級調整**：「守護腦盯字幕接回語音線」從批次 2 **提到批次 0**，且是妄想／家暴／傷人三類詞庫在語音線真正生效的前提。**批次 0 因此變三件事**：①守護腦加三類詞庫 ②守護腦接上語音線字幕監聽（並行、不卡通話）③新紅線同步寫進開場說明書（`CORE`／`RED`）當雙保險。語音線紅線行為要真人語音實測、不能只信打字線滿分。
> - **✅ 2026-07-12 批次 0 第二批更新**：上面三件事的第①②項已完工——`live_voice_server.py` 新增 `guardian_watch`／`guardian_scan_text`／`guardian_redirect_cue`／`guardian_ai_correction_cue`／`guardian_record_and_alert`／`guardian_flush_pending_cue`，把守護腦第一層（關鍵字）接上語音線的即時字幕（用戶字幕 + 寧寧自己字幕都盯），命中 high/critical 會第一記安全事件（沿用既有 audit/product event 機制）第二內部安全告警（Slack #沐寧-告警）第三在下一輪自然的 turn_complete 空檔遞安全導引給寧寧、讓她「下一句」轉向轉介；全程背景執行緒/非同步任務，不阻塞語音串流（已用假字幕＋假 I/O 模擬驗證：心跳沒被卡住）。誠實揭露：這是「事後極快監聽＋下一句補救」，不是「攔截阻止使用者剛講出口的那一句」——Live API 串流架構做不到攔截，critical 情況下 `chat_engine.RED`（session 開場就給的底線）仍是第一道防線，這一層是加固。**沒講危險字的拐彎危機（第二層 AI 語意判讀）設計見 §4.6，尚未實作。**

### 4.1 精神狀態異常 ＋ 不強化妄想

**新增詞庫 `psychosis_terms`（歸 `level = high`, `action = advise_grounding_and_help`）：**
- 被害/監控類：有人監控我、鄰居在監視、電視在看我、被下毒、有人要害我、跟蹤我、偷裝東西
- 幻聽/幻覺類：聽到有人叫我、聽到聲音、有人在跟我說話（無人時）、看到不存在的人
- 躁期/脫離現實類：好幾天不用睡、精神好到不行、我有特殊使命、我是被選中的
- （偵測順序：放在 `emergency_terms` 之後、`medical_terms` 之前；若同時命中自傷 → 仍以 critical 優先）

**回應策略（三步、寫進人格範本，禁止確認妄想內容）：**
1. **接住情緒、不確認也不否定事件**：「這種感覺一定讓你很不安。」
2. **對事件保持中立**：「我沒辦法確認真的有人在監控你，但我相信你現在的害怕是真的。」
3. **拉回安全 + 轉介真人**：「我們先確認你現在是不是安全的？要不要找一個你信得過的人，或請醫生一起看看？」

**禁止**：反問「他們為什麼要監控你？」（＝順著妄想，加深它）；也不說「你想太多了」（＝否定，破壞信任）。

### 4.2 保護與安全事件（家暴/虐待/財務剝削）

**新增詞庫 `protection_terms`（歸 `level = high`, `action = advise_protection_line`, `familyNotificationCandidate` 要更謹慎——加害人可能就是家人）：**
- 家暴/身體：被打、他打我、被推倒、被罵、不敢回家、怕他
- 遺棄/照護疏忽：沒人管我、沒東西吃、沒人帶我看病、被鎖在家、沒人理我很久了
- 財務剝削：錢被拿走、印章被拿走、被騙簽名、退休金不見、被逼給錢
- 性/兒少（若使用對象含兒少）：被摸、被欺負（需搭配年齡情境）

**回應策略：**
1. 接住、不追問細節（避免二次傷害）：「你願意跟我說這個，我知道不容易。」
2. 安全優先：「你現在安全嗎？身邊有沒有讓你害怕的人？」
3. 轉介保護專線：「有一支專線 113，24 小時都有人，可以幫你、也會保密。要不要我陪你記下來？」
4. **家人通知要判斷**：一般危機通知家人；但保護事件**若加害人可能是家人，不可直接推播家人**——這是與其他危機最大的不同，需在守護腦加一個「保護事件不自動通知家庭圈」旗標，改走專線/可信任第三人。

### 4.3 傷害他人（補進 `critical_terms` 的他傷分支）

**新增詞（歸 `level = critical`, `action = interrupt_and_escalate`）：**
- 想殺了他、想傷害某人、想報復、要讓他死、拿刀/工具 + 對象

**回應策略：** 冷靜、不批判、優先確認立即危險與是否獨處，促成人際/專業介入；不深入探討動機、不評判。

### 4.4 五級歸位對照（補齊後）

| 沐寧級 | ≈ GPT | 觸發 | 動作 |
|---|---|---|---|
| none | L0 | 日常 | 正常陪伴 |
| low | L1 | 情緒困擾（含補全的內疚/羞愧/失落…）| 傾聽、情緒反映 |
| medium | L2 | 醫療邊界、重複情緒 | 安全邊界 + 記錄 |
| high | L3 | 急症 / **精神異常** / **保護事件** | 轉介專線/家人/專業 + 安全評估問句 |
| critical | L4 | 自傷 / **他傷** | 立即中斷、危機引導、升級 |

### 4.5 危機時的安全評估問句（high/critical 共用、取代深度心理探索）

GPT 提醒：L3/L4 不做深度探索、不問「你為什麼想死」「童年是不是…」。改問：
- 你現在是不是正處在立即危險？
- 你身邊有沒有可能傷害自己的東西？
- 現在有沒有一個可以立刻陪你的人？
- 你現在的位置安全嗎？


### 4.6 語音線守護腦第二層：AI 語意判讀（設計，2026-07-12 · 尚未實作）

**現況**：§4 開頭「產品真相」段已完工第一層（關鍵字清單接上語音線字幕，見上方 ✅ 更新）。這一節設計的是
**還沒做的第二層**——沒講危險字、但拐彎透露危機的句子，第一層關鍵字清單天生抓不到，例如：
「我是拖累」「東西都交代好了」「走幾步就要停」「這樣就好了，不用麻煩你們」。
長輩表達習慣客氣、迂迴，這類拐彎危機比直白危機更常見，只靠關鍵字會漏接。

#### 4.6.1 在哪觸發（WHEN）

兩層防護、成本從便宜到貴依序把關，**不是每句字幕都升級給 AI**：

1. **第一層關鍵字（已上線）**先過一遍——命中 high/critical 就直接照現有流程走（記事件＋告警＋排隊安全導引），
   **不再額外呼叫第二層**（第一層已經夠確定，沒必要多花一次 AI 呼叫）。
2. 第一層結果是 `none`／`low` 時，再過一個**新的、刻意收窄的「苗頭清單」**（`SOFT_SIGNAL_TERMS`，跟現有
   `psychosis_terms`／`protection_terms`等清單分開放、不混進第一層，避免第一層越養越鈍）：
   「拖累」「交代好了」「不用麻煩」「這樣就好」「準備好了」「最後一次」「走不動」「喘不動了」「沒力氣走」
   「越來越沒力」等——這些詞單獨看很多時候只是真的在講生活瑣事（例如真的在交代退休金給孩子），
   **不足以直接判定風險等級，但值得讓 AI 多看一眼上下文**。
3. 苗頭清單命中時，才把**最近 3-5 輪使用者發言**（不含寧寧自己說的話，降低 prompt 大小＋聚焦在使用者揭露的內容）
   丟給第二層 AI 判語意；單輪之內的零星苗頭詞不必每個字幕片段都送一次——用跟第一層一樣的「這一輪已處置過的
   類別去重」機制（`user_flagged`／`ai_flagged` 那一套 reuse），一輪最多送一次。
4. **保險上限**：一通電話最多觸發第二層 N 次（建議 N=5），超過就不再加送——防止某種病態重複句式
   （長輩口頭禪剛好每句都命中苗頭詞）把成本/延遲打爆；超過上限仍持續走第一層防護，不影響既有安全網。

#### 4.6.2 餵什麼（INPUT）

- **內容**：最近 3-5 輪使用者發言的純文字（去識別化——不夾帶姓名/地址等個資，跟現有安全告警"不外洩內容"原則一致，
  但這裡因為要判語意，內容本身還是要給 AI 看，只是不落地存這段原文，判完即丟，只留分類結果進審計）。
- **格式**：仿照 `chat_engine.py` 既有 `remember()` 的呼叫慣例（`client.models.generate_content(model="gemini-2.5-flash",
  ..., config=types.GenerateContentConfig(response_mime_type="application/json"))`），用同一個已在專案裡跑得動的
  Gemini Flash 呼叫模式，不用另外接一套新 SDK。
- **System instruction 草稿**：
  「你在幫一個高齡語音陪伴 App 做安全語意判讀。以下是使用者最近幾句話（可能包含口語、方言、不完整句）。
  判斷這段話有沒有拐彎透露：（a）想放棄／不想活的念頭（即使沒有直接說「自殺」「不想活」這種字）
  （b）身體正在明顯惡化的跡象（c）疑似被害／被監控／幻覺的感受（d）疑似被暴力對待、遺棄、財務剝削的跡象。
  只根據字面內容判斷，不要過度腦補；不確定就給低信心分數，不要為了保守而把日常對話判成高風險。
  只回傳 JSON，不要多餘文字。」

#### 4.6.3 回什麼（OUTPUT）

```json
{
  "risk_detected": true,
  "level": "low|medium|high|critical",
  "category": "self_harm_indirect|physical_decline_indirect|mental_state_indirect|protection_indirect|none",
  "confidence": 0.0,
  "reasoning_short": "一句話，不含個資，只給營運後台看"
}
```

- **雙閘門才動作**：`level in {medium, high, critical}` **且** `confidence >= 0.6` 才觸發側效應（排隊安全導引／
  記事件／告警）；沒過閘門的判讀結果仍全部記進審計（哪怕是 `risk_detected: false`），供之後校準用（見 4.6.6）。
- **語氣比第一層更保守**：第二層本質是「猜測」不是「命中關鍵字」，排隊的安全導引措辭要比第一層更柔軟、
  留餘地，例如：「使用者剛剛的話可能只是隨口帶到，也可能透露出需要被多關心一下的情緒，接下來這句話
  自然多關心一句就好，語氣放輕，不要當成確定的危機處理、不要嚇到他。」——避免把一個低信心的語意猜測，
  講成像第一層關鍵字命中那樣斬釘截鐵的轉介台詞。

#### 4.6.4 成本估（COST，粗估、需在實作前對照當時 Gemini 定價校正）

- 苗頭清單觸發率抓寬鬆一點：一通 10-15 輪的電話，大概 0-1 次會真的命中苗頭清單升級給第二層
  （多數日常對話根本不會經過這一關，因為第一層 `none`／`low` 之後還要再過一次收窄的苗頭清單，雙重篩選）。
- 單次呼叫 prompt 約 300-500 tokens（3-5 輪對話＋指令），輸出 JSON 約 50-100 tokens——用 Gemini 2.5 Flash
  量級模型，單次呼叫成本落在極低（次分美分等級，粗估每次呼叫 < US$0.001，需在實作前用當時官方定價頁複核，
  這裡不寫死精確數字避免過期誤導）。
- 就算每通電話都命中 1 次（保守上限估法），量體對照現有《成本doc附錄》結論——真正吃毛利的是 Gemini Live
  語音分鐘數逐輪計費，不是這種輕量文字分類呼叫；第二層即使每天跑幾千通電話，佔比也遠低於語音本身成本，
  **不是這個功能的成本瓶頸**。

#### 4.6.5 延遲估（LATENCY）

- Gemini 2.5 Flash 短文字分類呼叫，實務上約 300ms-1.5s 來回（依當時網路/模型負載浮動）。
- 因為第二層的呼叫方式跟第一層共用同一套「背景任務 → 排進 `pending_cues` → 等 `turn_complete` 天然空檔才送出」
  管線（batch 0 第二批已建好的機制，第二層是直接插進去、不必重新蓋一套），這 300ms-1.5s **完全不會讓使用者
  感覺到卡頓**——它跑在背景，使用者跟寧寧的對話照常進行，AI 只是在下一個自然的輪替空檔多知道一件事。
- 唯一的殘留風險（跟第一層一樣的性質，程度更明顯一點）：如果第二層判讀比第一層慢、比寧寧的下一句回覆還慢，
  這次的安全導引就會延後到「下下一輪」才送出（因為它錯過了最近的那次 `turn_complete` 空檔）。這是可接受的
  已知限制，不是攔截機制，是「盡快跟上」機制。

#### 4.6.6 誤判防護（FALSE POSITIVE GUARDRAILS）

1. **苗頭清單前置篩選**——大多數字幕根本不會進到第二層，天然限縮誤判的曝光面。
2. **雙閘門**（level + confidence）——低信心的猜測不會直接觸發側效應，只留審計紀錄。
3. **第一層優先、不重複判**——第一層已經確定的 high/critical 不會再繞去問第二層，省成本也避免兩層結果打架。
4. **同輪去重**——重用第一批已經寫好的 `user_flagged`／`ai_flagged` 去重機制，同一輪不會因為滾動字幕反覆觸發。
5. **保守措辭**——第二層排隊的 cue 語氣明確比第一層軟，寧可讓寧寧多關心一句無傷大雅的話，
   也不要讓她對一個誤判的日常對話講出嚇人的危機轉介台詞。
6. **每通電話呼叫上限**（N=5）——防止病態重複觸發拖垮成本/延遲，超過上限自動停用第二層但第一層仍持續運作。
7. **全量落審計、供校準**——不管有沒有過閘門動作，第二層每次判讀結果都記一筆（`reasoning_short` 一句話），
   之後可以定期人工抽查「第二層抓對了幾次、誤判了幾次、漏接了幾次」，回頭調苗頭清單跟信心門檻，
   形成校準迴圈（跟 `admin_safety_events_summary` 現有的安全事件後台是同一套，不必另開新後台）。

#### 4.6.7 施工掛點（給未來實作者）

延伸現有 `live_voice_server.py` 的 `guardian_watch()`：第一層 `guardian_scan_text()` 回傳 `none`／`low` 時，
再過一次 `SOFT_SIGNAL_TERMS`（新清單，跟現有 `psychosis_terms`／`protection_terms` 等分開放）；命中就另開一個
背景任務呼叫 Gemini Flash（可用 `client.aio.models.generate_content(...)` 走原生 async，比 `asyncio.to_thread`
包一層同步 SDK 更省一次執行緒切換——`live_voice_server.py` 已經在用 `client.aio.live.connect`，`client.aio`
這個 async 命名空間本來就在用），判讀結果通過雙閘門後，塞進同一個 `st["pending_cues"]`（跟第一層共用同一條
排隊/送出管線，不必重蓋）。**這是設計，本批次不實作**——先把清單、prompt、閘門邏輯定案給 Edward/卡西法下一批施工用。
