## 一、整體評級

**資安成熟度：起步偏基本（Starting → Basic 之間）。**

一句話依據：**基礎衛生做得比多數個人專案好**（機密不進 git、body 有大小上限、防目錄穿越、有門衛/帳單哨兵、gen-auth-config 主動拒絕把萬能鑰匙送進瀏覽器、smoke 測試內建 secret 掃描）——**但「誰能讀誰的資料」這條授權主幹還沒建起來**：資料端點只驗「你是不是合法用戶」、不驗「這筆資料是不是你的」（BOLA），Supabase 目前 RLS「account 級全開」（他們自己的設計文件白紙黑字承認），加上刪帳號沒真刪雲端。對一個裝滿長輩健康資料的 App，這幾條是上架前真的會出事的必修。

把「授權邊界＋真刪帳號＋邀請碼防爆破」三件補上，就能從「起步」跳到「基本穩健、可上架」。

## 二、前 3 大風險（給 Edward 的 30 秒版）

1. **🔴 跨人/跨家庭讀得到長輩敏感資料（BOLA，OWASP API #1）** — 資料端點（`/family/state`、`/wellbeing/*`、`/memory/retrieve` 等）只信 request body 帶的 `personId`/`familyGroupId`，不驗呼叫者是否擁有它；後端又用 Supabase service-role key（繞過 RLS）。等於「知道別人家的編號就讀得到血壓、心情、對話記憶」。**P0**

2. **🔴 「刪除我的資料」沒真刪雲端** — `/account-deletion` 只記一張單，沒刪 Supabase。隱私頁卻明寫「刪除帳號會清除你在雲端的個人資料」＝說的跟做的不一致。這條同時踩**蘋果 5.1.1(v) 會被退件** ＋ **個資法/GDPR 刪除權** ＋ **對用戶陳述不實**。**P0**

3. **🔴 家庭邀請碼可爆破 → 進圈 → 讀全家資料** — 6 位數邀請碼（100 萬組）兌換端點無速率限制、無嘗試計數，且兌換不驗「你是不是被邀請的本人」。配上第 1 條 BOLA，這是最現實的一條攻擊鏈。**P0（防爆破）**

## 三、逐項體檢（7 大類 · 附檔案:行號證據）

### 1. 身分與授權

**✅ 做對的：**
- `/admin/*` 全部端點都逐一呼叫 `admin_authorized`（`engine/server.py:4364-4468`），且與 `MUNEA_REQUIRE_AUTH` 無關、一律要 token。沒有漏擋的 admin 接口。
- 金流寫入類（`/subscription-event`、`/credits/grant`、`/credits/consume`、`/entitlements` save）走 `privileged_billing_write_authorized`（`server.py:4497、4540、4566…`），要 admin 或 provider token。
- staging 部署有設 `MUNEA_REQUIRE_AUTH=1` ＋ `MUNEA_ENABLE_DEV_AUTH_BYPASS=false`（`scripts/cloud-run-deploy-staging.ps1:153-154`）——資料端點在 staging 至少要一張有效 Supabase token。
- dev bypass 需環境變數明確開啟才生效（`server.py:153-155`），預設關。

**🔴 P0 — BOLA / 物件級授權缺失（OWASP API1:2023）**
- 證據：do_POST 分派時，資料端點只傳 `data`、**不傳 headers/auth context**：`family_state_response(data)`（`server.py:4375`）、`wellbeing_trend_response(data)`（`4388`）、`memory_retrieve_response(data)`（`4357`）、`family_members_response(data)`（`4379`）。
- `family_state_response` 直接用 body 的 `familyGroupId` 當索引（`server.py:579`），`save` 分支還能**寫**別家的 vitals/circle（`582-626`）。`memory_retrieve_response` 的本地路徑 `load_memory_items()` 完全沒有 personId scoping（`1662`）。
- 會怎麼出事：任何一個登入用戶（甚至只要有薄門 app-key）帶著別人的 `familyGroupId` 打 `/family/state`，就讀得到那家長輩的血壓、心率、血氧、心情、家人動態；帶 `save` 還能竄改或灌假數據（假的高血壓告警 → 嚇家人）。
- 一人團隊修法：把 auth context 傳進每個資料 handler，用「登入者的 authUserId → 他所屬的 person/family」對照表，驗證 body 的 `personId`/`familyGroupId` 確實屬於他，不符就 403。這需要 `auth_user_id ↔ person ↔ family` 的對應（正好是 `docs/RLS-子女授權同意分層…md` §3 卡點 B 要補的那張表）——一次補齊，同時解 BOLA 和 RLS 兩題。

**🟠 P1 — admin token 比對非 timing-safe**
- 證據：`admin_authorized` 用 `supplied != token`（`server.py:3175`），`provider_webhook_authorized` 同（`3185`）。Python 的 `!=` 遇到第一個不同 byte 就短路，理論上可做時間側信道。
- 會怎麼出事：實務上跨網路有抖動、24 byte 隨機 token 幾乎不可能真被爆出來，風險低；但這是 OWASP 會標、且修起來一行的事。
- 修法：`import hmac` → `hmac.compare_digest(supplied, token)`。

**🟢 P2 — dev bypass token 格式寬鬆**
- `verify_dev_auth_token` 只要 `dev-local-token-<任意字串>` 就過（`server.py:156-167`）。目前靠環境變數關閉（staging 已關）。維持「prod/staging 永遠關」即可，不必額外做事，但值得寫進部署 checklist 防手滑。

### 2. 會員資料存取控制（Supabase / RLS / 家庭隔離）

**🔴 P0 — RLS 目前「account 級全開」，分層同意只在文件**
- 證據（他們自己的設計文件 `docs/RLS-子女授權同意分層-設計與待決-2026-07-01.md` §1）：
  > 「所有敏感表的 RLS 都是 account 級全開：任何 active account_members（含 viewer）都能讀該帳號底下所有 persons 的 conversation_summaries / safety_events / memory_items / perception_snapshots … 文件設計的『長輩控制、預設保護』在資料庫層等於不存在。」
- 後端這條路更關鍵：`supabase_adapter` 一律用 `SUPABASE_SERVICE_ROLE_KEY` 當 `apikey`（`engine/supabase_adapter.py:60、2311`）＝**完全繞過 RLS**。所以就算 RLS 收緊，只要走 brain server 就無效；brain server 的 app 層又有上面第 1 類的 BOLA。→ 兩條路都通往同一個洞。
- 商業決策要分清楚：Edward 2026-07-01 已拍板「家人連結長輩後完整看健康數據」（§3 卡點 A），這部分**是可接受的產品決定、不是漏洞**。但同文件保留兩條硬底線目前**還沒落到 DB/程式**：
  1. **連結當下長輩要一次性同意 ＋ 留同意紀錄**（個資法，長輩是資料主人）。
  2. **L4「原始對話逐字 ＋ 情緒崩潰當下 details」永不自動攤開**——這條現在在 DB 層是攤開的（`conversation_summaries` 原文、`memory_items` taboo tag、`safety_events.details`）。
- 會怎麼出事：家人（或任何拿到 account 成員身分的人）讀得到長輩最私密的原始對話與情緒崩潰紀錄——這是「上新聞等級」的長輩隱私，且與隱私頁 line 86「避免預設攤開原始逐字對話」的承諾不符。
- 一人團隊修法（縮小版，接受 Edward 健康數據全開的決定後其實不多）：
  1. 寫 `007_consent_tier_rls.sql`：敏感表的 `select`，非 owner/admin 檢視者只給 L1/L2；**L4 欄位/列永不對非 service_role 開**（文件 §4 已幫你把 SQL 方向寫好）。
  2. 敏感表的 `insert/update/delete` 收回後端 service_role，client 只能 select（同時補稽核完整性）。
  3. 連結流程加一次性同意寫進 `consent_records`（端點 `consent-records` 已存在，`server.py:4381`）。

**🟠 P1 — 試營運預設共用一本帳（cross-tenant 混資料）**
- 證據：`family_state_response` 沒帶 `familyGroupId` 時預設 `group = "shared"`（`server.py:579`），且 `group == "shared"` 會走雲端（`581`）。試營運期多台裝置若沒帶到 familyGroupId，family state 會全部落進同一個 "shared" 桶。
- 會怎麼出事：試營運/demo 期不同用戶的家庭狀態互相看得到、互相覆蓋。正式接真帳號（UUID）後會改善，但上架前若還有裝置走 fallback，就有真人資料混桶風險。
- 修法：把 "shared" 預設拿掉，沒有合法 familyGroupId 就回錯誤（或發一個裝置專屬隨機 group），別讓多人落同一桶。

**🟢 P2 — 家庭資料存取沒有讀取稽核**
- 只有金流/admin 變更會 `append_audit_event`；家庭/健康資料的**讀取**沒有留痕。BOLA 若被利用，事後查不到。上線後補一條輕量 access log（誰、何時、讀哪個 family）即可。

### 5. 刪帳號 / 資料可攜（個資法 / GDPR / 蘋果）

**🔴 P0 — `/account-deletion` 沒真刪雲端**
- 證據：`account_deletion_response`（`server.py:3901-3923`）只 `append_privacy_request("account_deletion", data)` 記一張單，回傳的 `productionSteps`（`3915-3922`）是「production **應該**要做」的描述（"soft-delete user-scoped data"…），**實際沒刪 Supabase 任何一列**。STATUS ㉑「給 Mac」也自標了這條待補。
- 對照隱私頁 line 92：「刪除帳號會清除你在**本機與雲端**的個人資料」——說的跟做的不一致。
- 會怎麼出事：① 蘋果審查實測「刪除帳號」發現資料還在 → **5.1.1(v) 退件**（此要求 2022-06-30 起強制執行）。② 個資法/GDPR 刪除權未落實 → 用戶申訴/主管機關風險。③ 隱私政策陳述不實 → 法律曝險。
- 一人團隊修法：`account_deletion_response` 的 `request` 分支，實際呼叫 Supabase（service_role）刪或 soft-delete 該 auth 使用者名下所有 person-scoped 資料（conversation_summaries / memory_items / family_state / wellbeing / consent / perception…），並刪 Supabase Auth user；保留法律必要的 billing/audit（隱私頁已這樣寫、對）。做成同步刪即可，量小不必非同步。刪完回 `status: completed`。

**🟠 P1 — 資料匯出回傳的是全域資料、沒 scope 到本人**
- 證據：`privacy_export_response`（`server.py:3876-3898`）把 `load_app_profile_store()` 整包、`load_conversation_summaries(limit=500)` 全部、`load_billing_store()`、`load_privacy_requests_store()` 全塞進 exportPackage，**完全沒有 per-user scoping**（它自己的 `productionNote` line 3897 也承認 "should require authentication, and redact…"）。
- 會怎麼出事：任一登入用戶打 `/privacy-export {action:request}` → 拿到的是本機 store 裡**所有人**的對話摘要、帳務、隱私單。這是資料外洩。
- 修法：export 也要收 auth context，只回該 authUserId 名下 person 的資料；redact 掉 provider secret/內部 log。

**✅ 做對的：** App 內有「匯出資料 / 刪除帳號」入口（隱私頁 line 92、設定頁），路徑存在、對蘋果「易於尋找」的要求是好的起點——只差把後端做成真的。

### 6. 後台本身的資安

**🟠 P1 — 營運後台公開在 Cloud Run，只靠一把 admin token**
- 證據：`admin.html` 由同一個 Cloud Run 公開服務（do_GET 靜態檔，`server.py:4312-4320`；STATUS 記載 `/admin.html` 回 200）。頁面本身不含 token（好），但整個後台只靠 `X-Munea-Admin-Token` 一道擋（`server.py:4364-4468`）。
- token 洩漏的後果**極大**：admin 端點能撈全體用戶名冊、對話摘要、安全事件、隱私單、訂閱（`/admin/accounts`、`/admin/conversation-summaries`、`/admin/safety-events`…）＝**全體長輩 PII 一把撈**。
- 會怎麼出事：token 若因 XSS、螢幕分享、log 誤印、或下面那條 localStorage 被偷，攻擊者無需其他條件就看光所有用戶。
- 一人團隊修法（挑 1-2 個做，成本都低）：
  1. **加第二道**：Cloud Run 前面放 IP allowlist（只允許 Edward 家/常用 IP），或把 `/admin*` 拆到獨立服務加 Cloud Run IAM（真正的 Google 登入才進得去）。
  2. admin token 定期輪換 ＋ 至少 32 byte。
  3. 後台加操作稽核（誰在何時撈了什麼）。

**🟠 P1 — admin token 存進瀏覽器 localStorage**
- 證據：`web/src/admin.js:55` `ADMIN_TOKEN_KEY = "munea.admin.token"`；勾「記住通行碼」就 `localStorage.setItem(ADMIN_TOKEN_KEY, token)`（`admin.js:635、746-747、767`）。
- 會怎麼出事：localStorage 可被任何在該網域執行的 JS 讀走（XSS 一發就外流），且永久留存。這把 token = 全體用戶 PII。
- 修法：① 至少改成 sessionStorage（關頁即失效）或不記憶、每次貼；② 若要「記住」，配合上面的 IP allowlist/獨立服務把爆炸半徑縮小；③ 後台頁面加基本 CSP，降低 XSS 面。

**🟢 P2 — 缺安全標頭（HSTS / X-Frame-Options / CSP）**
- `_send`（`server.py:4276-4281`）只設 Content-Type/Length，沒有 HSTS、X-Frame-Options、CSP。Cloud Run 有 HTTPS 但沒 HSTS；admin.html 可被 iframe 包（clickjacking，低）。上線後補幾個 header 即可，一次幾行。

### 7. 相依與監控

**✅ 做對的：**
- 相依極少：`engine/requirements.txt` 只有 `google-genai>=2.10`、`websockets>=16`；引擎用 Python 標準庫 `http.server`，沒有肥大 web 框架 → 已知 CVE 攻擊面小。
- **DoS 基本防護有**：JSON body 上限 `MAX_JSON_BODY_BYTES = 1_000_000`（`server.py:49、4293-4297`）、音訊 `MAX_AUDIO_NOTE_BYTES = 12_000_000`（`50、4225`）；do_GET 有防目錄穿越（`4316` `full.startswith(WEB_DIR)`）。
- **有監控雛形**：`tools/door-sentinel.py`（門一鎖就發 Slack #munea-營運 ＋ 附開門指令）＋ `tools/billing-sentinel.py`（帳單哨兵，STATUS 記 NT$500 警戒）＋定時任務 `munea-door-sentinel`。這對一人團隊很務實。

**🟠 P1 — 語音服務只有薄門、且 app-key 半公開 → 成本濫用**
- 證據：`munea-voice-staging` 部署只給 `GEMINI_API_KEY`、**沒設 `MUNEA_REQUIRE_AUTH`**（`cloud-run-deploy-staging.ps1:190-191`），`--allow-unauthenticated`（`198`）；`live_voice_server.py:232-241` 只驗 `?key=` 薄門。app-key 又硬編在前端（見第 3 類）。
- 會怎麼出事：拿到 app-key 的人可無限開 Gemini Live session → 燒 Gemini 額度/帳單（Unrestricted Resource Consumption，OWASP API4），也可冒用「長輩陪伴」角色。帳單哨兵能事後告警，但擋不住當下濫用。
- 修法：① 語音服務也要求有效 Supabase token（把 REQUIRE_AUTH 概念帶進語音層）；② 每 session 綁真用戶、加簡單頻率限制（每用戶每分鐘 N 通）；③ 短期先靠帳單哨兵 ＋ 收緊 Cloud Run 最大實例數擋成本尖峰。

**🟠 P1 — 相依用 `>=` 未釘版（供應鏈）**
- `requirements.txt` 用 `>=`，建置不可重現；未來 `google-genai` 若被投毒或引入破壞性改動，部署會被動吃到。
- 修法：釘死版本（`==`）＋定期手動 `pip list --outdated` 檢查一次。一人團隊每月一次夠。

**🟢 P2 — 缺集中錯誤告警**
- 目前靠 `log_fallback_exception` 寫 log ＋門衛/帳單哨兵。資料層異常（例如 BOLA 被打、大量 403、Supabase 錯誤率飆）沒有主動告警。上線後可把 Cloud Run log-based alert 接一條到同一個 Slack。

## 六、正面評價（別只看紅字 — 這些做對了）

一人團隊能做到這幾點，資安意識其實在水準之上：
- 真機密**不進 git、歷史也乾淨**，gitignore 完整，gen-auth-config 主動拒絕萬能鑰匙進瀏覽器。
- Cloud Run 走 gcloud secrets、body 有大小上限、防目錄穿越、smoke 內建 secret 掃描。
- 隱私頁**誠實**：點名跨境（Gemini + Supabase 澳洲）、精確 GPS 已拿掉、用藥照片只存本機、不做廣告追蹤——方向全對。
- 有門衛/帳單哨兵、有 RLS 分層同意的**設計文件**（代表知道問題在哪，只是還沒落地）。

主要缺口集中在「授權主幹」這一塊——不是散落各處的小洞，是一條沒建完的主樑。把 P0 那 5 條（其實核心是那張對應表）補起來，成熟度就能實質跳一級。

## 七、我沒能驗證的（誠實標註）

- **iOS App 是否直接讀 Supabase 資料表**（用 publishable key + supabase-js SDK，`gen-auth-config.py` 有注入）：我確認 SDK/登入走 Supabase，但「資料讀取是否也直連 Supabase 而非只走 brain server」未能從程式完全確認。**若有直連**，則 RLS「account 級全開」直接生效在 client 這條路（P0-4 更緊迫）；**若只走 brain server**，則 service_role 繞過 RLS、由 app 層 BOLA 決定（P0-2）。兩種情形 P0 結論不變、只是主戰場不同。建議 Edward/Mac 確認一下 App 的資料讀取路徑。
- **RLS 是否真的 enabled**：依他們自己的設計文件推斷是「ON 但 policy 全開」，我沒有直接連 Supabase 驗 policy。上線前值得實際查一次 `pg_policies`。
- **Prod 與 staging 的差異**：目前 staging 即試營運環境，沒有獨立 prod。我只讀到 staging 部署腳本。正式上架時若另開 prod，這份 checklist 要對 prod 重跑一次。
