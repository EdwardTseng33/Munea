## 三、時間軸設計（今天 / 7 天 / 30 天切換）

**原則：操作看週、策略看月、趨勢圖疊 7 日平均。** 建議切換模式：

| 頁 | 建議時間切換 |
|---|---|
| 總覽 | 頂部一個全域切換「今日 / 近 7 天 / 近 30 天」，北極星＋KPI＋圖都跟著換 |
| 產品成長 | DAU 圖「近 8 週 / 近 6 月」；留存本來就是月 |
| 用戶管理 | 「今日活躍」給今日；名冊不用時間切換 |
| 安全守護 | 「近 7 天 / 近 30 天」（警訊要快看） |
| 訂閱 | 月為主（訂閱是月週期）；趨勢近 6 月 |
| AI 用量 | 「近 7 天 / 近 30 天 / 本季」 |

**現在的問題**：只有數據看板有 30/90 天切換，其他頁時間窗口寫死。**上線版要把總覽做成全域時間切換**（今日/7天/30天），其他頁按上表補。

## 五、上線與串接步驟（分三階段）

**第 0 階段（現在就能做，不用埋點）：**
- 把 9 個接口全接上，能接真的頁面（總覽/成長/用戶/安全/意見/系統紀錄）換真資料＋標「真資料」。
- 追不到的（心情球、完成率%、SLA、金流帳本、獲客花費）先拿掉或標「待埋點」，不留假數字。
- 補「全體訂閱聚合」接口（MRR/方案分布用）＋「世代留存」算法（各半天工）。
- 總覽加全域時間切換（今日/7天/30天）。

**第 1 階段（補埋點，之後有真數字）：**
1. ~~`subscription_cancelled`~~ → **改走 Apple API（第 2 階段），不埋點**
2. `routine_reminder_due`（算提醒完成率）— ⚠ 提醒響在原生層（App 關著也響），要原生回報，不是純網頁
3. `mood_logged`（心情球、情緒關注）— ⚠ App 目前心情只顯示、沒有「打一顆」的記錄動作，要先做記錄 UI
4. `health_log_submitted`（健康規律回報）
5. `safety_event_acknowledged` / `resolved`（安全回應時效 SLA）
6. `family_member_joined`（家庭圈啟用率）
＋事件補欄位：角色（char）、方案（plan）、來源（主動/被動）、單筆估算成本。

> ⚠ **埋點難度校正（2026-07-10 查證）**：原以為「提醒到期／心情球」網頁層就能埋——**錯**。提醒通知響那刻在原生層（App 關著、網頁沒在跑），要原生回報；心情球目前只顯示、要先做記錄動作。所以這幾個都要動原生／後端，不是加一行網頁程式。

**第 2 階段（外部串接）：**
- **訂閱／營收改走 Apple 開發者 API，不埋點**（Edward 2026-07-10 拍板 · 更準更乾淨）：
  - **App Store Connect API**（拉報表）：目前有效訂閱數、各方案人數、營收（proceeds）、退款——算真 MRR、方案分布、ARPU 的來源。用開發者帳號建一把 API 金鑰（Keys 頁）。
  - **App Store Server Notifications v2**（Apple 主動推）：續訂／取消／降階／退款／申訴 即時推到我們伺服器 → 真流失率（churn）＋真金流帳本＋爭議紀錄。設一個接收接口、填進 App Store Connect。
  - 這樣就**不用在 App 裡埋 `subscription_cancelled`**——Apple 自己的帳最權威、不會漏。第 1 階段的訂閱取消埋點可以取消、改走這條。
- Cloud Run 監控指標 → 真服務可用率/延遲。
- 「標記已處理」升級成後端共用（跨裝置）。

> 訂閱資料兩條路對照：**埋點**＝在 App 記「有人買了」，只看得到「進」、要另外埋才看得到「出」，且是我們自己的帳；**Apple API**＝直接拿 Apple 官方的訂閱與營收帳，進出退款申訴全都有、最權威。**結論：訂閱/營收走 Apple API，App 內互動行為（通話、提醒、心情）才走埋點。**
