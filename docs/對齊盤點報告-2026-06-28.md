## 4. 缺口（北極星/簡報要求、但沒文件覆蓋好）

### 🔴 G-1 · 對話畫面的互動規格（最大缺口 · 正是做歪的根）
- **北極星/簡報要求**：核心畫面=全屏管家臉、即時字幕、會說話會動。
- **現況**：**沒有一份文件規範「對話進行中畫面」的互動細節**——簡報只有一行藍圖（§四-3 三個元素），美術規格只管「臉長怎樣」，技術藍圖只管「臉怎麼算出來」。**沒有人定義「這個畫面上有什麼、字幕怎麼排、結束鈕多大、聊聊 vs 視訊兩個畫面差在哪、待機 idle 時長什麼樣」。**
- **後果**：執行者沒有畫面級 spec 可依 → 憑印象做 → 做成小圓頭像。
- **這是整個盤點最重要的發現**：對話畫面做歪不是執行者粗心，是**根本沒有對話畫面的規格文件**。
→ SPEC 必須新增一章「核心畫面互動規格」（見 §5 第 6 章），把全屏臉畫面的每個元素、兩個對話模式（聊聊/視訊）的區別、idle/說話/結束三態釘死。

### 🔴 G-2 · 即時語音延遲的端到端預算（散、未整合）
- **北極星 C**：「對話一來一往順、不卡」=命脈。建置藍圖標「開口 <1.5 秒」但標「目標·待實測」。
- **缺**：換 Gemini 3.1 後，延遲預算（Gemini 出音 + Ditto 生臉串流 + WebRTC）沒有用 Gemini 的數字重算；且「被打斷的自然處理」「上下文連貫」（北極星 B-1/B-2）這些對話節奏設計沒有任何技術文件落實怎麼做。
→ SPEC 要有「延遲預算 + 對話節奏」章節，引擎換 Gemini 後重估。

### 🟡 G-3 · 「會動的臉」怎麼接 Gemini（引擎換了、接法沒更新）
- avatar-雙引擎藍圖 §E 寫的是「gpt-realtime-2 出音 chunk → AOL → Ditto」。
- 換 Gemini 3.1 Flash Live 後，**音訊 chunk 的格式/串流方式可能不同**（Gemini Live 的 audio out 接口 ≠ OpenAI Realtime）→ AOL 餵 Ditto 的那一段接法要不要改？沒文件確認。

### 🟡 G-4 · 擴大客群後的 TAM/財務沒重算
