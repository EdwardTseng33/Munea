## Phase 1 · 地基：真帳號 ＋ 雲端後端

> 為什麼先做：其他所有東西（收錢、限制、同步）都要疊在「每個人有自己的帳號、資料放在手機連得到的雲端」上面。地基不穩，上面都白做。

### 規劃
- [ ] 決定用哪家雲端／帳號服務（現有程式已預留 Supabase，可沿用）— **需 Edward 拍板花錢**
- [ ] 決定登入方式（Apple 登入必備，蘋果強制；要不要加手機號/Email）
- [ ] 資料搬遷計畫：把現在的單機示範資料，換成每個帳號各自的雲端資料

### 設計
- [ ] 登入／註冊流程畫面收尾（現有 authSheet 已設計、需接真流程）
- [ ] 「訪客先逛、要用功能再登入」的引導設計

### 開發
- [ ] 雲端後端正式上線（不再只在本機跑）— **需 Edward 決定伺服器、要花錢**
- [ ] 打開帳號驗證（現在是關著的示範模式）
- [ ] 接 Apple 登入
- [ ] 資料改成「跟著帳號走、換手機能同步」

### 卡 Edward
- 雲端伺服器選型＋月費（要花錢）
- 是否沿用 Supabase

## Phase 2 · 收錢：蘋果內購 ＋ 三道免費／付費限制

### 規劃
- [ ] **方案命名對齊**：現在畫面寫「基礎/高級/尊榮」，程式裡卻是「plus/premium/concierge」，你先前又講 Free/Plus/Premium/Concierge — 三套名字要統一（**需 Edward 定一套**）
- [ ] 每個方案「能用什麼、不能用什麼」定案表（含價格）
- [ ] 點數（加值）要不要跟訂閱一起上，還是第二版再說

### 設計
- [ ] 付費牆畫面：免費用戶撞到限制時，看到的「升級提示」設計
- [ ] 三道限制各自的「被擋住」畫面（家人邀請／聊聊時間到／狀態頁鎖住）

### 開發
- [ ] **接蘋果內購**（用戶點訂閱→跳蘋果付款→真扣款→我們收得到）— 卡西法主做
- [ ] **三道免費／付費限制**（可先做，之後接真方案）：
  - [ ] 家人邀請：免費不能邀，Plus 以上才行（現有半套，補完）
  - [ ] 聊聊：免費每天 5 分鐘，超過提示升級並關閉（**已定案：每天重置**）
  - [ ] 狀態頁：免費只看今天，過去 7／30 天要 Plus 以上
- [ ] 一個「這個帳號是什麼方案、能用哪些功能」的單一判斷中心（讓限制跟付款狀態綁在一起）

### 卡 Edward
- 蘋果開發者帳號（US$99/年）＋公司銀行帳戶＋報稅資料（不設定蘋果不撥款）
- 方案名稱與價格定案
- 內購抽 15%，定價要含進去（NT$499 實拿約 424）

## Phase 3 · 打磨：推播 ＋ 健康數據 ＋ 設計收尾 ＋ 品質

- [ ] Apple 健康串接（若第一版要）
