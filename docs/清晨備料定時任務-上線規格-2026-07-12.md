## 二、重大發現：23514 失敗的真正根因，不只是併發風暴

背景任務描述提到「清晨沒定時抓、通話才臨時補、生出 23514 失敗執行緒；去抖已止血（commit 7e95e77）」。
這次實測（跑 run_daily_briefing.py 真打一次）復現了同一個錯誤，往下查到 schema 層級的根因：

RuntimeError: Supabase POST perception_snapshots failed: 400
code=23514, details 顯示 Failing row contains (..., daily_briefing, ...)

23514 = Postgres check_violation。查 supabase/sql/004_ai_memory_service_foundation.sql（唯讀查、未動 prod），
perception_snapshots 表的 snapshot_type 欄位有一條 CHECK constraint，允許清單是：
time, weather, calendar, location, current_topic, book_context, travel_context,
local_activity_context, exercise_context, finance_context, media_context, food_context,
news_context, wisdom_context, family_context, interest_graph

daily_briefing 從建表那天起就不在允許清單裡。這代表：

- server.py 的 refresh_daily_briefing() 每次要存 snapshotType 為 daily_briefing，Postgres 都會用 CHECK constraint 擋下、寫入從未真的成功過——不管是清晨定時跑、還是通話中臨時補，結果都一樣是 23514。
- 去抖（7e95e77）只讓失敗的頻率降低（同時只一條、5 分鐘內不重試），沒有讓寫入成功——這是止血、不是根治。
- 佐證：本機 engine/perception_snapshots.json（Supabase 接上前的本地備援）有 12 筆 daily_briefing 快照，全部落在 2026-07-08 之前（Supabase 於 7/9 接上後）；7/9 之後零筆——因為 Supabase 一啟用，寫入就一路被 23514 擋、也不會落到本地備份。
- 更深一層：append_perception_snapshots() 目前只在錯誤訊息含缺表(PGRST205) 或 22P02 時才會退回本地 JSON 備援；23514 不屬於這兩類，會被直接 raise——原本這裡沒有 try/except，整支 cron 會帶著未捕捉例外直接崩潰。這次已修：refresh_daily_briefing() 存檔那步包了 try/except，失敗時回乾淨的 ok:False、error:snapshot_save_failed（briefing 欄位仍附上算好的內容方便除錯），不會再讓 run_daily_briefing.py 噴原始 traceback。

但這只是失敗得優雅，不是真的能存進去——真正解法是 schema 補值，也就是 supabase/sql/009_perception_snapshot_daily_briefing.sql。這支 migration 沒跑之前，清晨定時任務即使掛上 Cloud Scheduler，每天還是會存檔失敗（AI 對話當下仍讀得到同一次 request 產出的內容，但下一輪、或下一次通話重新讀 snapshot 時就抓不到，因為根本沒真的存進資料庫）。

### migration 內容（supabase/sql/009_perception_snapshot_daily_briefing.sql）

純新增一個允許值，對既有 16 種 snapshot_type 的資料與行為零影響（動態找出既有 constraint 名稱再 drop 重建，避免硬猜名稱出錯；已寫好、未執行）：

alter table public.perception_snapshots add constraint perception_snapshots_snapshot_type_check
check (snapshot_type in (原本 16 種, 'daily_briefing'));

執行方式：Supabase Dashboard 的 SQL Editor 貼上執行（跟 001-008 同一套流程，SQL Editor-ready）。
這是本次任務的唯一硬阻擋——沒跑這支，Cloud Scheduler 部署再漂亮，清晨簡報也存不進去。

### 已知限制（誠實揭露、不是漏做）

Cloud Scheduler 的 --headers 是明文寫進 job 設定（gcloud scheduler jobs describe 看得到），
不像 Cloud Run 的 --set-secrets 能接 Secret Manager，這是 Cloud Scheduler HTTP target 本身的限制，不是這次沒做好。
風險等級跟現有 MUNEA_APP_KEY 直接寫進 --set-env-vars（兩支部署腳本都這樣做）同一個量級，算是專案既有的可接受風險，
沒有降級。若要更嚴謹，之後可以加一層小的 Cloud Run Job 或 Cloud Function 當 proxy，從 Secret Manager 讀 token
再代打，這次先不做（超出清晨備料這個任務範圍，列為可選硬化項）。

--schedule="30 6 * * *" 用的是標準 cron 語法，--time-zone=Asia/Taipei 確保無論 Scheduler 執行機器在哪個時區，
觸發時刻都對齊台灣時間 06:30。
