## 0. 一句話總結

現況是「一張卡一個人」的單機服務、沒有池、沒有調度、滿載就退化語音——這三件事都要改。顯卡池本身用 Modal 原生自動擴縮（或備援自建閘道）撐得住 100-1000 DAU 的尖峰；但真正的天花板不在顯卡、在語音層——Gemini Live 現在只實測穩 5 人、10 人就出聲不穩，這個數字比顯卡池的規劃容量小 10-40 倍，是上線前最急迫要查清楚的一件事。**另外，剛剛對正式服務做的一次唯讀健康檢查發現：現在線上餘裕只有 31.9%（比團隊兩份文件引用的 40%/67.8% 都更緊），且確認目前是「未開 compile」的較慢模式在跑——這是比 640/768 解析度之爭更優先的隱憂，應該先修。**

## 2. 顯卡分流池架構

### 2.1 核心設計：控制面（control plane）跟媒體面（data plane）分開

```
App(client)
   │ 按撥通
   ▼
分流閘道 Gateway（輕量、CPU-only、常駐低成本）
  ① 查顯卡池空位  ② 查語音池空位  ③ 兩者都有才放行
   │
   │有空位：配一個 worker、把 worker 門牌直接回給 App
   │        （App 之後 WebRTC 直連該 worker，media 不繞 gateway，
   │         避免多一跳延遲、避免 gateway 變頻寬瓶頸）
   │沒空位：進佇列（見第 5 節）、有空位時主動觸發連線
   ▼
顯卡池（GPU workers，每個 worker=1張卡，內部1或3路）        語音池（Gemini Live bridge）
┌───────────┬───────────┬───────────┐               ┌──────────────────────┐
│warm anchor │ 彈性 worker │ 彈性 worker │               │ Cloud Run munea-voice │
│(RunPod美/  │(Modal，scale│(Modal，scale│               │ + Gemini Live 上游     │
│Glows台，   │-to-zero)   │-to-zero)   │               │（真天花板待查，見6.3）  │
│常駐待命)   │           │           │               └──────────────────────┘
└───────────┴───────────┴───────────┘
```

為什麼 media 面不繞 gateway：WebRTC 的影像/聲音是即時串流，多一跳代理會疊加延遲、且 gateway 要背頻寬——gateway 只做「登記＋配對＋釋放＋開關卡」的控制決策，媒體流永遠是 App 與 worker 直連（這也是現有 flashhead_server.py/nening_modal.py 的 /offer WebRTC 交握模式已經在用的模式，不用重寫）。

### 2.2 兩個候選方案（技術選型待驗證，見第 9 節）

**方案 A（推薦、先驗證）：Modal 原生多容器池**

把現在的 bug 順手修掉：@modal.concurrent(max_inputs=20) 改成 max_inputs=1（一容器一路，防互看臉）或 max_inputs=3（一容器 3 路，配第 3 節多槽改造）。max_containers=1 改成 max_containers=N（N＝第 6 節卡數估算的池上限）。用 Modal 平台原生能力做池管理：scale-to-zero（沒人用自動歸零，scaledown_window 控制多久沒人用才睡）、burst（併發請求超過現有容器數，平台自動開新容器）、buffer_containers（提前多開 1-3 顆待命容器，蓋掉多數冷啟等待）。

優點：零自建排程器，池管理全交給平台，維運負擔最小。

未驗證風險（上線前必須 PoC）：/offer（開連線）跟音訊 WS（餵聲音）是兩個獨立 HTTP/WS 請求，兩者必須落在同一個容器（session affinity）才能對到同一個引擎實例——這件事 Modal 平台層面會不會保證，目前沒驗證過。若驗證失敗（黏不住、兩人可能分別落到不同容器導致連線失敗或串線），退到方案 B。

**方案 B（備援）：自建輕量閘道 + RunPod/Glows 手動池**

不依賴平台原生 session affinity，閘道自己維護 worker registry（worker 可以是 RunPod pod、Glows instance、甚至 Modal container 都能登記），/offer 進來時閘道先選一台有空位的 worker，把該 worker 完整門牌直接回給 client，client 直連該 worker——affinity 天生成立（同一個 client 全程只認一個 worker 網址，不會被平台換池）。worker 端沿用現有 flashhead_server.py 邏輯（單例或多槽皆可），只加一個「通話結束回報 gateway 釋放」的 webhook。開卡/關卡：閘道呼叫 podctl.py/glowsctl.py 同款 API（RunPod、Glows 都已有腳本化開關）；Modal 若也登記進池，開卡就是呼叫該 function（Modal 本身仍會 scale-to-zero，閘道只是多一層帳本）。

優點：架構掌控度高、不賭平台未驗證行為。缺點：多一個服務要建置維運，GPU 開關卡邏輯（閒置偵測→呼叫 stop API）要自己寫，比方案 A 費工（估 3-5 個施工日，馬魯克排）。

**建議路徑**：先花半天做方案 A 的 session affinity PoC（2 個真人同時打同一個 Modal 池、看有沒有串線/失聯）——過關就上方案 A（省事、上線前來得及）；沒過關就轉方案 B。**這是本文件唯一一項「先驗證才能定案」的池架構選型，其餘項目都可以直接動工。**

### 2.3 調度邏輯（怎麼分配、怎麼開新卡、怎麼關卡）

- 配對演算法：Fullest-first bin packing——優先把新通話塞進「已經有人在用、還有空槽」的 GPU（而非空的新 GPU），讓部分卡盡量填滿、其餘卡盡量閒置好被 scale-to-zero 回收。純 1 路/卡時退化成單純的「找一台空的」。
- Burst 開卡門檻：不要等到 100% 滿才開新卡（那樣下一個人一定卡進佇列）。建議池使用率 > 80% 時就主動預開下一台（Modal 用 buffer_containers；方案 B 由 gateway 主動呼叫開卡 API）——這是「提前墊卡」而非「滿了才開」，把冷啟時間藏在使用者按下撥通之前。
- Scale-to-zero 門檻：GPU 上 0 人使用超過 scaledown_window（建議 2-5 分鐘，太短會頻繁冷啟、太長會空燒）才釋放。warm anchor（RunPod 美／Glows 台）例外——首波上線期間建議常駐（見第 7 節），不跟著自動睡。
- 語音層聯合准入：閘道放行一通新通話的條件＝「顯卡池有空位 AND 語音池有空位」，兩者取交集，不是只看顯卡（見 6.3 節，語音天花板現在比顯卡池小很多，必須一起判斷，否則顯卡有空位也沒用、聲音接不上）。

### 2.4 冷啟動 8-10 秒怎麼藏

現有機制（Avatar.wake()，進聊聊頁就開始探健康）已經是對的方向，池化後延續＋加強：

1. warm anchor：常駐待命的 1-2 台卡（RunPod/Glows）永遠 0 冷啟，首波上線的基本盤（見第 7 節）。
2. buffer_containers 預熱：彈性池提前多開 1-3 顆待命（2.3 節），把冷啟藏在「使用率上升」而非「按下撥通那一刻」。
3. 待機動畫頂著：現有 FaceIdle 待機畫面 + 文案（如「幫你接通中，稍等一下下」）在等待期間持續播放，不留白屏/不留死畫面——這段 UX 文案是女巫/霍爾的活，這裡只定「一定要有東西頂著、不能空等」的技術要求。
4. 分時段 keep-warm（上線後擴，非首波必備）：長輩活躍時段（早 7-9、晚 18-21）拉高 warm anchor 數量，離峰縮回最低。

## 3. 單卡並發驗證計畫（FlashHead 3 路實測）

### 3.1 工程前提（描述改造方向，不寫程式碼）

現行 flashhead_server.py 是全域單例（一個 outer／一個 pipeline／一個 feeder／一個 sink／一把 char_lock）。3 路化需要把單例改成「N 槽（slot）陣列」：每槽各自擁有一份 pipeline 實例、feeder、FrameSink、AudioOutBuffer、角色狀態；/offer 進來時透過 admission lock 找一個空槽鎖定，該 session 的音訊 WS 與影像軌都綁定同一槽（槽內 session affinity，比方案 A/B 的容器級 affinity 更小顆粒、更好掌控）；/health 回報 capacity.limit=N, active=x（取代現在的 limit=1）。

### 3.2 測試矩陣（6 項，上線前必跑）

| # | 測項 | 驗什麼 | 方法 |
|---|---|---|---|
| 1 | VRAM 上限 | 3 個 pipeline 實例同時常駐是否 < 24GB 留安全邊界 | 3×6.3GB=18.9GB 理論值，實機同時載入量測峰值（含 CUDA context 開銷）|
| 2 | 運算爭用 | 3 路同時生成，每路 p95 是否仍在 chunk budget 內 | 3 條真實錄音同時餵、量各槽 p50/p95/underrun，沿用現有 bench_fh.py/驗收腳本方法論。**注意用「真實服務條件」量、不能只信 bench_fh.py 的孤立數字**——0.1 節剛量到的現場落差（孤立 bench 305ms vs 現場 654ms，接近 2 倍）證明兩者可能差很大，3 路驗證務必用真的 /offer + WebRTC + 音訊 WS 全鏈路測，不能只測裸生成迴圈 |
| 3 | 串線隔離 | 確認任何一路絕不會收到另一路的音/畫（正是 Modal max_inputs=20 那個已知 bug 的正面回歸測試）| 3 種不同輸入餵 3 槽，逐槽比對輸出是否只反映自己的輸入 |
| 4 | 長時穩定 | 30-60 分鐘連續通話是否有 VRAM 洩漏/佇列漂移/執行緒崩潰 | 3 槽各跑 30-60 分鐘、盯記憶體曲線與 underrun 計數 |
| 5 | 故障隔離 | 1 槽故障（強制拋錯/OOM）是否波及其他 2 槽 | 人工注入錯誤，確認其餘槽不斷線——這點比現況更重要：現在 1 個 process 出事只影響 1 人，3 路化後 1 個 process 出事影響 3 人，故障半徑放大 3 倍，必須驗證隔離有效 |
| 6 | 多槽冷啟 | 開機時 3 槽一起載入 vs 懶載入（用到才載）對冷啟時間的影響 | 比較「開機即載 3 槽」（冷啟變慢但常駐後零延遲）vs「懶載入」（開機快但第 2/3 人要多等一次槽內冷啟）|

### 3.3 決策閘門

全綠 → 第 6 節卡數估算採「3 路/卡」欄；只穩 2 路 → 採 2 路欄；只穩 1 路 → 沿用現況「1 路/卡」欄。這份驗證結果直接決定尖峰卡數估算的除數，是本文件最關鍵的一組待補實測數字——即使首波上線先用 1 路，也建議 DAU 還小、風險低的階段就把這輪測完，別等真的要擴大時才發現 3 路不穩。

## 4. 畫質升級三合一驗證計畫（現行真相＋768 升級路徑＋並發×解析度矩陣）

> 這節是 Edward 追問畫質後擴充的範圍，取代原本單純的「640 驗證」。三個子題：現行到底是幾、768 要幾週且加不加錢、單卡並發怎麼變。

### 4.1 子題一：現行真相釐清（文件打架，這裡先定案能定的部分）

**打架的兩份文件**：霍爾 `畫質優化路線調研-2026-07-11.md` 說「現行解析度 640×640、p95 576ms、餘裕 40%、單卡約 1.6 路」，標「實測」；但 `flashhead_server.py:170` 寫死 `cv2.resize(p, (512, 512))`，且更早的深度調研文件說官方模型「固定輸出 512×512」。

**已經能定案的部分（程式碼路徑讀出來的事實）**：line 170 那個 512 resize，是 `_load_poster()` 函式在做的，只用來產生 `self.poster`——那是**待機時顯示的靜態照片**，跟真正驅動嘴型/表情生成的 `get_base_data()` 呼叫（line 76-77）**是兩件不同的事**。`get_base_data()` 直接吃 `CHAR_SRC[self.char]`（也就是 `/root/char-a05B.png` 這個檔案本身）的原始尺寸，並沒有經過那個 512 resize。**所以「模型實際生成解析度＝512」這個結論，程式碼本身並不能百分之百證明——512 resize 只影響待機照片，不影響推論解析度；推論解析度取決於 `/root/char-a05B.png` 這個檔案在遠端機器上實際存了多大。**

**已經即時查證但還不夠完整的部分**（0.1 節）：現場 /health 顯示 p95=653.7ms、餘裕 31.9%，跟霍爾 640 版的「576ms/40%」量級接近（都是「不到 1 秒但吃掉大半預算」），比 Glows README 早期 eager bench 的 305-309ms（餘裕 67.8%）慢超過 2 倍——這比較接近「現在在跑 640 附近的東西」而不是「跑 512」，因為 512 理論上運算量該更小、餘裕該更接近 305ms 那組數字，不該是 654ms。**這是間接證據，指向「現行很可能真的接近 640，霍爾的數字方向是對的」，但不是直接證據。**

**還缺的最後一塊拼圖（30 秒鐘就能查完，但需要有權限的人）**：SSH 進 `tw-06.access.glows.ai`（埠號要從 Glows 主控台或 `glowsctl.py status` 查，本機缺 SDK token 查不到），跑一行 `python3 -c "from PIL import Image; print(Image.open('/root/char-a05B.png').size)"`，直接得到答案。**這是本文件唯一沒辦法自己查完、需要移交的一步**，建議 Edward 或有 Glows 主控台權限的人補一下（或把 `GLOWS_SDK_TOKEN` 放進 `deploy/glows/.env`，下次卡西法 session 就能自己查）。

**現階段結論（給 Edward 的暫定判斷）**：**先別糾結「512 還是 640」這個標籤，把「現在只有 31.9% 餘裕、而且確認沒開 compile」當成現行真相的核心事實去處理**——不管底圖實際是 512 還是 640，「開 compile」都是霍爾排名第一、零成本、風險低的動作，且現場數據顯示它可能還沒被套用到正式服務——這件事的優先權高於解開解析度標籤之爭。

## 5. 「滿載排隊／預約」取代「退純語音」

### 5.1 要拔掉的東西

web/src/app.js:3216-3222：Avatar.start() 失敗且 _lastError==='capacity_full' 時自動切純語音（voice_only 模式、埋點 avatar_capacity_fallback）——這段整段拔除，換成 5.2 節的排隊流程。**這是產品定義層級的改動，不是優化項，上線前必備。**

### 5.2 即時排隊設計（上線前必備）

1. 使用者按撥通 → 閘道查「顯卡池空位 AND 語音池空位」（2.3 節）。
2. 有空位 → 現況良好路徑不變（直接接通）。
3. 沒空位、佇列深度 < 上限（建議上限例如 20，實際數字隨首波監控校準）→ 進入排隊：
   - App 顯示「現在很多人在跟寧寧聊天，你排第 N 位，預估等待約 X 秒」，畫面維持待機動畫（不接通、不計費、不消耗點數、完全不啟動任何通話——排隊本身不是退化服務）。
   - client 對閘道保持一條輕量心跳（WS 或短輪詢），回報「還在排隊/還在前景」；閘道有空位釋出時，依序 FIFO pop，若該 client 仍在線 → 直接觸發連線（沿用現有 Avatar.start()／LiveVoice.start() 邏輯，只是延後觸發時機，不必重寫連線本體）；若 client 已離開/斷線 → 跳過、遞補下一位。
   - 等待時間預估：用「近期滾動平均通話時長」×排隊位置做粗估，不用精確保證——目的是不讓人乾等沒訊息，這是 UX 底線，技術上不難（維護一個最近 N 通通話時長的滑動窗口平均值即可）。
   - 若 App 被切到背景太久，有空位輪到時用 push notification 叫回來（「寧寧準備好要跟你聊了」）。
4. 佇列已達上限（真的太忙）→ 這是唯一保留的「真拒絕」出口，但必須說明原因＋給選項（不是靜默失敗、不是偷偷退語音）：告知「現在真的很忙」+ 導向 5.3 節預約（若已上線）或建議稍後再試。
5. 排隊中使用者可隨時取消，不留殘留佔位。

排隊文案/UX 細節（畫面怎麼呈現、位置數字要不要顯示、等待動畫長什麼樣）是女巫/霍爾/蘇菲的活，本節只定資料流與技術實作方式。排隊優先權要不要依訂閱等級分（例如 Pro 用戶優先）是商業判斷，技術上支援（佇列可以是優先權佇列而非純 FIFO），但決定權不在卡西法。

### 5.3 預約時段設計（上線後擴，非首波必備）

長輩使用情境分散全天、尖峰可預期（早晚固定時段），可加開「預約撥通」：使用者選一個未來時間點（如 15 分鐘後），系統保證屆時開一張卡候著（提前 15-20 秒觸發 burst 開卡，蓋掉冷啟），到點用 push notification 提醒。這是即時排隊之外的體驗加分項，不擋首波上線，建議排在排隊機制穩定運作、且首波用戶量驗證過排隊確實會被觸發之後再做。

### 5.4 UX 決策樹（技術面摘要，給實作對照）

```
按下撥通
 │
 ├─ 顯卡+語音都有空位 → 直接接通（現況路徑，不變）
 │
 ├─ 沒空位、佇列 < 上限 → 進排隊（顯示位置+預估等待，可取消）
 │      └─ 輪到 → 自動接通（沿用現有連線邏輯）
 │      └─ 使用者離線/取消 → 遞補下一位
 │
 └─ 佇列已滿 → 明確告知「現在真的很忙」→ 建議預約（若有）或稍後再試
      （唯一保留的真拒絕出口，不靜默、不退語音）
```

## 6. 卡數估算（100 / 1000 DAU × 尖峰同時率）

### 6.1 假設

- 尖峰同時率取 Edward 給的區間，切三檔：保守 10% ／ 標準 15% ／ 尖峰 20%（長輩通話集中時段）。

### 6.2 卡數表

**100 DAU**

|---|---|---|---|
| 保守（10%） | 10 | 10 | 4 |
| 標準（15%） | 15 | 15 | 5 |

**1000 DAU**

| 檔位 | 同時通話數 | 1 路/卡所需 GPU | 3 路/卡所需 GPU |
|---|---|---|---|

對照題目原給的「尖峰 10-100 人同時撥通」：100 DAU 三檔（10-20）完全落在範圍內；1000 DAU 保守檔（100）剛好卡在上限，標準/尖峰檔（150-200）已超出原示範範圍——這是 1000 DAU 情境下的真實需要，先攤開讓 Edward 知道規模，不是估算錯誤。

**3 路/卡欄位的前提是第 3 節驗證全過；驗證前只能用 1 路/卡欄位規劃。若第 4 節的 768 升級也要一起上（C 組），3 路/卡欄位可能要打折（若 C 組不過關、只能 2 路），屆時卡數會落在 1 路欄與 3 路欄之間。**

### 6.3 語音層對照（真正的瓶頸警示）

Gemini Live 現在實測 5 人穩、10 人出聲但一致性不穩——這個數字已經比 100 DAU 標準/尖峰檔（15-20）小，更遠比 1000 DAU 三檔（100-200，小 10-40 倍）小非常多。**顯卡池不管怎麼設計都撐得住上表數字（Modal/RunPod/Glows 三家都有現成的擴容手段），但語音層現在的已知穩定上限撐不住——這是規劃裡最急迫要查清楚的一件事，優先權高於顯卡池架構本身。**

上線前必查兩件事（技術面，非成本面）：

1. **這個上限是 Google API 配額硬牆，還是 Cloud Run 容器內資源競爭？**——現行 munea-voice prod 設定 --max-instances 5 --concurrency 40（容器層理論 200 併發），但實測 5-10 人就不穩，落差很大，代表瓶頸極可能不在 Cloud Run 容器數，而在單一容器內部（1Gi 記憶體處理多條並行音訊串流的資源競爭）或 Gemini Live API 本身的 per-project 併發配額。兩者診斷方式不同：容器內競爭→查 CPU/記憶體曲線；API 配額→查 Google Cloud Console 的 Generative Language API 配額頁。
2. 若是容器內競爭：解法可能是「降低單容器 concurrency（例如 40→10）、拉高 max-instances」，用更多小容器分散壓力，而非少數大容器塞好塞滿。若是 API 配額硬牆：解法可能是向 Google 申請提高配額，或用多組 API 金鑰/多 GCP 專案做 sharding（不同使用者分流到不同金鑰，各自享有獨立配額，效果是把天花板乘以金鑰數）。
3. 建議做一次純語音壓測（不接臉、只測聲音）、從 1 人逐步加到 20-30 人，記錄退化發生的確切人數、同時記錄 Cloud Run 的 CPU/記憶體/容器數曲線——這組數據能直接回答「是配額還是資源」，是本文件排在最前面的待驗證項。

閘道的聯合准入邏輯（2.3 節）已經把這個現實納入設計——不管顯卡池開多大，實際能放行的通話數永遠是 min(顯卡池空位, 語音池空位)。若語音天花板沒解，顯卡池開再多也只是晾在那裡。

## 7. 首波上線容量建議

現階段 DAU 遠未到 100/1000（早期/軟性上線），不建議一開始就為千人規模預先鋪好整套池——建議走「小而穩」起步，隨監控訊號逐步擴容。

### 7.1 首波配置建議

- warm anchor（常駐待命）：沿用現有 2 台已備卡（RunPod 美國 4090＋Glows 台灣 4090），首波期間不跟著自動睡，作為零冷啟的基本盤——**現況美國那台是關的（0.1 節查證），首波上線前建議至少確認要不要把它重新開回常駐，否則等於只有台灣一台在扛全部流量、沒有真正的第二地備援**。1 路架構下＝最多 2 個常駐名額；若第 3 節驗證通過 3 路，同 2 台卡就有 6 個常駐名額，暫不用急著加卡。
- 彈性池（Modal 或方案 B）：設一個保守上限（例如 max_containers=5-10），scale-to-zero，作為 warm anchor 滿載後的溢出承接——不預先為 1000 DAU 鋪好，池上限隨 7.3 節監控訊號往上調。
- 語音層：先完成 6.3 節的純語音壓測，找到真實穩定上限，並以此數字反過來設閘道的「語音池空位」上限——這一步優先於加開顯卡數量，因為顯卡池目前的規模已經超前於語音層。

### 7.2 監控指標（上線前必備基本款）

| 指標 | 用途 |
|---|---|
| 即時同時通話數（顯卡池 + 語音池分開看） | 隨時知道離池上限多近 |
| 各 worker GPU 使用率 | 判斷是否需要調 fullest-first 打包參數 |

| 冷啟命中率（本次連線是否踩到冷啟 vs 命中 warm） | 判斷 warm anchor / buffer_containers 數量夠不夠 |
| 語音池併發數 vs 已知穩定上限 | 逼近上限時提前預警，避免真的撞牆才發現 |
| /health 的 gen_compute p95 / headroom_pct（沿用現有欄位） | 持續盯著 0.1 節查到的「31.9%」有沒有隨優化改善、有沒有隨用量上升再惡化 |

### 7.3 擴容觸發規則（上線後擴）

建議先用「人工審視＋手動調上限」（每週看一次監控，佇列觸發率 > 5% 或冷啟命中率明顯上升就調高池上限），不必一開始就做全自動擴容決策——池「內」的開卡/關卡本來就自動（scale-to-zero/burst），只有池上限這個數字先由人決定，等規則穩定、數據夠多，再考慮把調上限這件事也自動化。
