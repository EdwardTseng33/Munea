> 為何不直接寫 SQL：這條寫錯 = 洩漏長輩私密資料或鎖死家人存取，是「上新聞等級」的防線，不能盲寫。

## 1. 現況（問題）
所有敏感表的 RLS 都是 **account 級全開**：任何 active `account_members`（含 `viewer`）都能讀該帳號底下**所有** persons 的 `conversation_summaries` / `safety_events` / `memory_items` / `perception_snapshots`。
`family_memberships.permissions` jsonb 存了，但**沒有任何 policy 去讀它**（證據：`001_initial_munea_schema.sql:435-475`、`004:103-135`）。
→ 子女裝了 App 就能看長輩全部對話摘要與健康細節。文件設計的「長輩控制、預設保護」在資料庫層等於不存在。

## 2. 目標模型（文件 §2.3 · 長輩控制、預設保護）
| 層 | 子女看得到 | 預設 | 對應資料表/欄位 |
|---|---|---|---|
| L1 安心燈號 | 好/要注意、今天事情有無完成 | ✅ 開 | `safety_events`(status/severity 摘要)、`routine_reminders` 完成狀態 |
| L2 健康摘要 | 用藥順從、活動趨勢（白話非精確）| 🟡 開可關 | `conversation_summaries`（摘要層）、彙總指標 |
| L3 詳細數字 | 精確血壓/睡眠/時間軸 | 🔴 關（長輩主動開）| 詳細 metric 表/欄位 |
| L4 永不給 | 原始對話逐字、禁忌話題、情緒崩潰 | 🔴 永關無開關 | `conversation_summaries` 原文、`memory_items`(taboo tag)、`safety_events.details` 明文 |

## 3. 兩個必須先解的卡點
**🟢 卡點 A — Edward 已拍板（2026-07-01）**
> **決定：家人一旦「連結」上長輩，就完整顯示健康狀態數據。**
> 理由（Edward）：看得到健康狀態正是家人下載這個 App 的理由之一；把數據藏起來反而製造家庭糾紛。
> 影響：健康數據（L1 安心燈號～L3 詳細數字）連結後全開、**不預設隱藏**——這其實接近現在的「帳號內全看」，所以健康數據的存取大致維持開放。
> 仍要守的兩件（不牴觸此決定）：
> 1. **連結那一刻，長輩仍要同意一次**（台灣個資法，長輩是資料的主人）。這不是藏資料——是設定時長輩點頭一次、之後家人完整看，並留一筆同意紀錄。
> 2. **只有「原始對話逐字＋情緒崩潰當下」不自動攤開**（L4）——這不是健康數據；用 Edward 自己的邏輯（避免家庭糾紛），這種私密談心給看反而會引爆衝突。⏳ 這一條待 Edward 確認要不要也開。

**🏗️ 卡點 B — schema 缺一塊（要跟 Codex 協調，別各改各的 schema）**
現在 `account_members` 是 `user_id↔account+role`；`family_memberships` 是 `person_id↔group+role+permissions`——**沒有「哪個 auth 使用者 ↔ 哪個長輩 person」的連結**，也沒有「per 檢視者的分層授權」。
要落 RLS 需補：① auth 使用者對應到自己的 person（分辨「我是長輩本人 vs 我是來看的子女」）② 授權以 (檢視者→長輩, 層級) 可查。
→ 這是資料模型變更，Codex 正在 schema/adapter 深水區，**需一起定，不該我單方加表**。

## 4. 建議的實作方向（決定後就能寫成 `007_consent_tier_rls.sql`）
1. 加一個 SQL helper：`can_view_person_tier(viewer_uid, person_id, tier) returns boolean`——讀長輩該 person 的 `family_memberships.permissions`，判斷該檢視者是否被授予該層。
2. `owner`/`admin` 與後端 `service_role` 維持全存取（backend-authoritative 不變）。
3. 收緊 `conversation_summaries`/`safety_events`/`memory_items`/`perception_snapshots` 的 `select`：非 owner/admin 的檢視者，只給 `can_view_person_tier(..., 'L1'|'L2')` 通過的列；**L4 欄位/列永不對非後端開**（原文摘要、taboo 記憶、情緒 details）。
4. 寫入（`insert/update/delete`）敏感表收回後端 service_role，client 端只 `select`（順帶補沙利曼 #6 稽核完整性）。
5. 配 migration 一起補：`002_demo_bootstrap` 的 demo permissions 用明確假資料示範 L1 開、L3 關。

## 5. 現在的狀態（Edward 決定後）
決定＝「連結後完整顯示健康數據」，健康數據維持開放（大致等於現況）。要補的縮小為：
1. 連結時長輩「一次性同意」＋同意紀錄（法律必要）。
2. 原始對話逐字／情緒崩潰不自動攤開的小防線（待 Edward 確認要不要也開）。
3. 敏感表的「寫入」收回後端，家人端只能看不能改（補稽核完整性）。
這幾項的落點與 schema 我跟 Codex 對齊後寫成 migration（因 Codex 正在動後端/認證，避免各改各的）。
